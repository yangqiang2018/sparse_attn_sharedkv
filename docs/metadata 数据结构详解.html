metadata 详解 · 一张分给几十个核的「排班表」

sparse attention shared kv 算子的 metadata 参数不是要算的数据，而是一张工单：把一大堆重复计算公平地分给芯片上几十个计算核。SWA / CFA / SCFA 三种场景用的是同一张表，只是填进去的数字不同。

这篇讲什么：这个算子有一个叫 metadata 的输入参数。本文说清楚两件事 ——（1）它的数据结构到底长什么样（就是一串 1024 个整数、分成两张表）；（2）在 SWA / CFA / SCFA 三种场景下，这张表有什么不同。

目标读者：完全没接触过神经网络的高中生也能读懂。你只需要会数数、加减，以及一个生活常识：一堆活儿分给几个人干，怎么分才最快。「注意力」是什么、算子在算什么，第 0 节用大白话从零讲起。全文用同一个小例子贯穿到底。

0先说人话：metadata 是一张「排班表」

先不管神经网络。你只要知道：这个算子要做的事，是同一种计算，重复很多很多遍。

这个「同一种计算」叫注意力：处理当前这个词时，回头看前面的一大批词，给每个词打个「该关注多少」的分，再按分数把它们的信息混合起来。细节这里不重要 —— 重要的是：要处理的词很多，每个词都要回头扫一大批，活儿又多又重复。

好在，跑这个算子的芯片（一块 NPU）上有几十个一模一样的计算核（把它们想成几十个工人），可以同时开工。于是问题来了：

这么一大堆活儿，怎么分给几十个工人，才能让大家几乎同时收工？

答案是：开工前，先让一个「工头」（一个专门的调度器）把活儿排好班，写成一张表。每个工人一看这张表，就知道「我负责哪一段」，然后各干各的、互不打扰。

这张排班表，就是 metadata。

⚠️ 记住这一点，全文都不会绕晕：metadata 里没有任何「计算结果」，只有「谁干哪一段」的安排。它是给几十个核看的一张分工表。

1为什么排班这么讲究：忙闲不均就慢

为什么不能随便分？因为有一条铁律：

所有工人一起开工，但要等最慢的那个干完，整件事才算完。

所以如果分得不匀 —— 比如把一大半活儿堆给一个工人，其他工人各分一点点 —— 那大家就得干等着那个被压垮的工人，白白浪费。

还有一个细节：活儿不是每份一样重。有的序列长、要看的词多；有的序列短、几下就算完。所以工头不是傻乎乎地「按份数」平均，而是先估一估每份活儿有多重（成本），再按总重量去均分。第 5 节会看到「每份活儿的重量」正是三种场景的差别所在。

2数据结构：1024 个整数 = 两张表

现在看这张排班表长什么样。说穿了，metadata 就是一串 1024 个整数（都是普通整数），一个挨一个排在内存里。这 1024 个数分成两大块、也就是两张表：

这张表占多少个整数形状管什么
🟦 FA 排班表前 288 个36 行 × 8 列把主计算分给主力核：一行 = 一个主力核的工单
🟪 FD 合并表中间 576 个72 行 × 8 列收尾：一个大活儿被拆给多个核时，怎么把半成品拼回去
⬜ 预留末尾若干—凑齐 1024、备用，填 0

「一行 8 个数」是什么意思？就是每个核的工单，用 8 个整数描述完。下面两节分别拆开这两张表的「一行」。先记住这句话：

一份 metadata = FA 排班表（谁算哪段）+ FD 合并表（拆开的怎么拼回）+ 预留。大小固定，永远 1024 个整数。

「36 行」是固定预留的槽位数，不等于「实际用几个核」。这张表永远留 36 行（288 个数）—— 这是写死的结构尺寸，传参也改不了。但实际会点亮几行（开关=1），由一个「可用核数」设定单独决定：这个设定默认是 24（可以传别的值，但必须 ≤ 36 才不会出错）。所以默认情况下，第 24～35 行永远是 0 —— 它们在结构上就被封掉了，跟这次活儿多少无关。

换句话说，一行的开关为 0 有两种原因：① 被「可用核数」封掉（默认第 24～35 行，恒为 0）；② 这次活儿不够多，还没轮到这个核（临时歇着）。（FD 合并表的 72 行同理：72 是固定槽位数，实际用几个由另一个默认 48 的设定决定。）

3第一张表（FA 排班表）：每个核「从哪干到哪」

FA 排班表有 36 行，一行对应一个主力核槽位（真正做注意力大计算的核；36 是固定预留的槽位数，实际会点亮几行见上一节说明）。一行是 8 个整数，含义是这样的：

第几个数叫什么大白话
① 开关1 = 这行点亮、有活儿；0 = 没点亮 —— 要么被「可用核数」封掉（默认第 24～35 行恒为 0），要么这次活儿不够多、用不上它
②③④起点（3 个数）这个核从哪个工位开始干（工位地址要 3 个数，见下）
⑤⑥⑦终点（3 个数）干到哪个工位为止
⑧ 合并记号给第二张表（FD 合并表）用的一个编号，第 4 节讲；用不上时是 0

所以一个核的活儿，就是「从起点工位一路干到终点工位」的一整段连续工作 —— 像值班表上写「你从 3 楼 5 号扫到 7 楼 2 号」。

3.1 一个「工位地址」为什么要 3 个数？

因为所有活儿是三层嵌套编号的。你可以想成「哪本书、第几页、第几行」那样一层套一层：

这里的「组」和「块」到底是什么？它俩是同一件事的两个方向 —— 都是把一个大维度切成固定大小的小片。为什么要切？因为芯片的快内存一次装不下一整条，只能一小片一小片地处理。注意力说到底是个大矩阵乘（一边是 query 行、一边是 key 列），于是两条维度各切各的：

  • 第 2 层「组」= 切 query 行：把 query 行切成一个个小片，一组 = 固定行数的一批 query 行；
  • 第 3 层「块」= 切 key：把要看的 key 切成一个个小片，一块 = 固定数目的 key（参考实现里一块 = 512 个）。

所以一个「工位」，就是某个〔序列 × 头〕下的「第几组 query 行 × 第几块 key」 —— 相当于把那张大矩阵铺成网格后的一个小格子。（第 7 节把活儿数成「一个个块、每块 1 分」，指的就是这些格子。）

3.2 一个省数字的小心机：相邻两个核首尾相接

你可能注意到：表里既有「起点」又有「终点」，是不是有点重复？其实工头切活儿时，是把一整条活儿切成不重不漏的几段：

上一个核的终点，正好就是下一个核的起点。就像分蛋糕：切三刀分四块，每一刀既是左边一块的右边界、又是右边一块的左边界。

所以整段活儿被这些「切点」分给各个核，不重不漏，每个核领走相邻两切点之间的那一段。

4第二张表（FD 合并表）：大活儿拆开后怎么拼回去

大多数活儿一个核就能包圆。但偶尔会遇到一个特别重的工位 —— 同一组 query 行，却要配上特别特别多的 key 块（回想第 3 节：一个工位 = 一组 query 行 × 一块 key；这里是这一组 query 行要从头扫过好多好多块 key），一个核一块一块扫下来太慢。这，就是所谓的「大活儿」。

这时工头会把这一个工位再拆开，让好几个核各扫其中一部分 key 块。可这样一来，每个核只扫了一部分 key 块，算出的只是一个半成品 —— 准确说，是那一段自己的三个小账本：见过的最大值 m、累加的分母 d、还没除以分母的输出 O（正是 online softmax 里那套账本）。于是需要有人把这些半成品正确地合并成最终答案。

合并不是把各段的 O 直接相加 —— 因为每段是各自「减过自己那段的最大值」算出来的，尺子不一样，硬加会错。如果你读过姊妹篇 在线 Softmax 原理详解，这里就是那篇讲的「跨段合并」，四步就拼好：

  1. 在各段的最大值里取全局最大的那个 m，当作新尺子；
  2. 每段的 O 和 d 都乘上自己的修正因子 exp(段的 m − 全局 m)（换到新尺子上，≤ 1，打个折）；
  3. 各段对齐后相加，得到总的 O 和总的 d；
  4. 整个过程只除一次：最终输出 = O / d。

每段其实只要交出一个 LSE = m + ln(d)（把 m 和 d 打包成一个数）再加上自己的 O，合并核就能照这四步拼回去 —— 这正是那篇说 LSE「几乎白送、专为跨段/跨核合并而生」的原因。

⚠️ 实际情况：在这个算子的当前实现里，「把一个工位拆给多个核」这一步并没有启用，所以 FD 合并表始终是空的（整块 576 个数全是 0）。但它的位置一直预留在那 —— 这样不管用不用得上，metadata 的大小和结构都固定不变，读表的一方永远知道去哪找什么。

5活儿是从哪来的：窗口一趟 + 压缩一趟

现在回到最关键的问题：每个 query 的活儿到底有多少？这决定了工头怎么切分，也正是 SWA / CFA / SCFA 三种场景的分水岭。

每个 query（当前这个词）要看的 key，最多分两趟：

压缩趟里还藏着一个区别，它把 CFA 和 SCFA 分开：摘要块可能也很多，是全看，还是只看最相关的几个？

  • 全看：把（到目前为止的）所有摘要块都过一遍 —— 这是 CFA。摘要块数随序列变长而越来越多。
  • 只看最相关的 top-K 个：只挑最相关的 K 个摘要块看，其余不看 —— 这是 SCFA（S = sparse 稀疏）。不管序列多长，最多看 K 个，活儿有上限。

6三种场景 SWA / CFA / SCFA：同一张表，不同的数字

先把最重要的结论摆上来：

三种场景的 metadata，数据结构完全一样 —— 都是 1024 个整数、FA 排班表 36×8 + FD 合并表 72×8。变的只是工头填进表里的那些数字，因为每个 query 的工作量不一样，工头据此把每个核的边界画在不同地方。

场景窗口趟压缩趟每 query 的活儿序列变长时
SWA
滑窗		✓✗ 没有压缩 KV最轻（只有窗口）基本不变（窗口固定）
CFA
压缩·全看		✓✓ 全部摘要块中～重越来越重（摘要随长度增长）
SCFA
压缩·稀疏		✓✓ 只看 top-K 个中等，有上限封顶（窗口 + K，不再涨）

换成工头的视角（大白话）：

  • SWA：每个工位只有「窗口块」，压缩块数 = 0 → 活儿最轻、最规整，最好分。
  • CFA：每个工位 = 窗口块 + 压缩块，压缩块数 ≈ 序列长 ÷ 压缩比 → 长序列的工位特别重，还容易分得不匀（要靠拆工位 + FD 合并）。
  • SCFA：每个工位 = 窗口块 + 压缩块，但压缩块封顶在 K → 比 CFA 轻、比 SWA 重，且长序列也不会失控。

7完整小例子：同 3 条序列，三种场景各排一次班

把上面的话变成数字。设定（为了手算，全部简化）：

  • 3 条序列 A、B、C（A、C 短，B 长），只有 1 个「头」，所以第 1 层地址就是序列号。
  • 有 3 个主力核可用。
  • 把活儿简化成一个个「块」，每块算 1 分。窗口趟：每条序列固定 4 块。
  • 为看得清，工位地址用一个总编号（0、1、2…）代替真实的三层地址。
场景压缩块（A / B / C）每条总分（窗口4 + 压缩）总分3 核各分
SWA0 / 0 / 04 / 4 / 412每核 4
CFA1 / 3 / 2 （随长度）5 / 7 / 618每核 6
SCFA (K=1)1 / 1 / 1 （封顶 K）5 / 5 / 515每核 5

工头按「每核平均分」切一刀，排出的 FA 表（只列有活儿的 3 行，每行 = 开关 / 起点编号 / 终点编号）：

SWA · 每核 4 分

核01 / 0 / 4= A
核11 / 4 / 8= B
核21 / 8 / 12= C

正好一核一条序列，干净。

CFA · 每核 6 分

核01 / 0 / 6A + B 一点
核11 / 6 / 12B 剩下的
核21 / 12 / 18= C

B 被从中间切开给了核0、核1 → 需要 FD 合并表登记一行。

SCFA · 每核 5 分

核01 / 0 / 5= A
核11 / 5 / 10= B
核21 / 10 / 15= C

封顶后又变规整，一核一条。

看出门道了吗？三张表长得一模一样（都是 3 行「开关 / 起点 / 终点」），但里面的数字全不同 —— 因为三种场景的总活儿（12 / 18 / 15）和每条的轻重都不一样。本例把「可用核数」设成 3，所以核3～核35 那些行的开关全是 0（用不上的行一律填 0；换成默认的 24，则第 24～35 行恒为 0，见第 2 节）。这就是「同一张表，不同的数字」。

三句话总结

  1. metadata 不是数据，是一张「排班表」。它把海量重复的注意力计算，按工作量尽量均分给芯片上几十个计算核 —— 因为「最慢的核决定总时间」，所以必须分匀（负载均衡）。
  2. 结构固定：1024 个整数 = FA 排班表（36 行×8）+ FD 合并表（72 行×8）+ 预留。FA 每行 = 开关 + 起点(3 个数) + 终点(3 个数) + 合并记号；工位地址是「序列×头 / query 行组 / key 块」三层；相邻核首尾相接、不重不漏。FD 表管「大活儿拆开后怎么拼回」，当前实现里始终是空的、但位置永远预留。
  3. SWA / CFA / SCFA 结构完全相同，只是数字不同。差别的根源是每个 query 的活儿量：SWA（只窗口）最轻、CFA（窗口 + 全压缩）最重且随序列膨胀、SCFA（窗口 + top-K 压缩）居中且封顶。活儿量不同 → 工头切分点不同 → FA 表里的起点/终点数字不同。
