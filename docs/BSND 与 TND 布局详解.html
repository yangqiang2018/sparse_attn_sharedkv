张量布局 · 内存排布

BSND 与 TND · 为什么能统一按 TND 处理

两种 Q 的摆放约定，一条内核路径 —— 「padding 到等长的矩形」其实是「变长拼接」的一个特例，统一起来只花 O(B) 的小账本，那个大张量一次都不用搬。

这篇讲什么：注意力的输入 Q（以及输出）在内存里有两种常见摆法 —— BSND（规整矩形）和 TND（紧凑变长）。本文说清楚它们各自长什么样、为什么能用同一套内核逻辑统一按 TND 处理，以及这么做在 host（CPU 主机端）的代价到底有多大。剧透：几乎为零。

需要的前置知识：知道注意力的 Q 是「很多条序列、每条若干 token、每个 token 一串数」就够了。会一点「连续内存」「数组下标」的概念更好，没有也能跟下来。全文用同一个小例子贯穿到底。

0先说人话：什么是「布局」，为什么 Q 有两种摆法

注意力一次要处理的 Q 是这样一坨数据：一批序列（比如同时来的几条请求），每条序列若干个 token，每个 token 又是一串数（更准确说是 N 个注意力头、每个头 D 维）。问题来了：这一大坨数，要怎么摆进一块连续的内存？

麻烦在于每条序列长短不一：有的 100 个 token，有的 3 个。把长短不齐的东西塞进规整的内存，业界有两种约定 —— 这就是 BSND 和 TND。

打个比方 📚：你有一摞长短不一的文章要装订成册，怎么装？

本文的主角问题：这两种摆法看起来很不一样，内核（真正干活的那段设备代码）要不要为它们各写一套？答案是 不用 —— 统一按 TND 处理就行，而且几乎不花钱。下面先把两种布局各自讲清楚。

0.1 贯穿全文的小例子

一批 3 条序列（B = 3），真实长度分别是 2、3、2 个 token。我们给它们的 token 起名：

  • 序列 A（2 个）：a0, a1
  • 序列 B（3 个）：b0, b1, b2
  • 序列 C（2 个）：c0, c1

（每个 token 本身还有 N、D 两维，但那两维在两种布局里都一样、不受影响，所以下面画图时把它们省掉，只盯着「token 怎么排」。）

1BSND：规整的矩形（padding 到等长）

BSND 是四个维度的首字母：[B, S, N, D]。

维度含义
Bbatch —— 序列条数（例子里 = 3）
S统一的最大序列长度（例子里 = 3，因为最长那条 B 有 3 个 token）
N注意力头数
D每个头的向量维度

因为这是个规整的矩形，每条序列都得占满 S 个槽位。本例里 S = 3，但序列 A、C 只有 2 个 token，于是末尾补一个 padding 凑齐：

所以完整张量是 [B, S, N, D] = [3, 3, N, D]：

2TND：紧凑的变长（首尾拼接，无 padding）

TND 是三个维度：[T, N, D]。关键就在第一维 T：

T = total tokens = 所有序列 token 数之和（Σ 各序列长度）。所有序列首尾拼接成一长条，不补任何 padding。

例子里 T = 2 + 3 + 2 = 7。七个 token 排成一条：

cu_seqlens 怎么读：它是累积长度，cu = [0, 2, 5, 7] 意思是「第 0 条从 0 起、第 1 条从 2 起、第 2 条从 5 起、总共到 7」。第 b 条序列就占 [cu[b], cu[b+1])。

3关键观察：BSND 其实就是 TND 的一个特例

现在把两张图叠在一起看，会发现一件事 —— 它们的差别比看上去小得多。

把 BSND 的内存 [B, S, N, D] 按前两维合并着看，就是 [B×S, N, D]。而 [B×S, N, D] 不正是 一个 [T', N, D] 的 TND 张量吗（这里 T' = B×S = 9）？只不过这个「TND」比较特殊：

  • 它的每条序列长度都恰好 = S（因为都 padding 到等长了）；
  • 它的起点目录是规则等差的：cu = [0, S, 2S, …, B×S] = [0, 3, 6, 9]。

结论（本文的核心）：

BSND ＝「所有序列等长、且按固定步长 S 排布」的 TND。TND 是一般情形（变长、起点任意），BSND 是它的一个规则特例。

这正是「能统一」的根本原因：只要内核会处理一般的 TND（扁平 token 流 + 任意起点），BSND 自动落入其中 —— 不需要为它单写一套。

4统一处理：内核只认「扁平 token 流 + 每条序列的起点」

内核内部只在一个扁平张量 [total_tokens, N, D] 上干活。它定位第 b 条序列的第 s 个 token，用的是一个扁平 token 编号：

g = q_prefix[b] + s 扁平编号 = 这条序列的起点偏移 + 序列内位置

于是内核只需要两张 O(B) 的小账本（B 是序列条数，通常几十）：

  • q_prefix[b]：第 b 条序列在扁平流里的起点偏移；
  • act_q_lens[b]：第 b 条序列的实际长度（圈定有效范围，别越界算到 padding 或下一条）。

两种布局只是用不同方式把这两张账本填出来，填完之后走的是同一条内核路径：

用例子点一下：内核要取「第 b=1 条（序列 B）的第 s=0 个 token」，

  • TND：g = q_prefix[1] + 0 = 2 + 0 = 2 → 扁平流第 2 格，正是 b0 ✅
  • BSND：g = q_prefix[1] + 0 = 3 + 0 = 3 → 扁平流第 3 格，也正是 b0 ✅

两者算出的 g 不同（2 vs 3，因为 padding 让 BSND 的起点偏了），但取数那一步的逻辑一模一样。这就是「统一按 TND」的全部含义：内核只认一种寻址方式，布局差异被压进 q_prefix 这一张小表里消化掉了。

关于 BSND 的 padding：BSND 展平后的扁平流里夹着 padding 槽位（例子里第 2、8 格）。q_prefix[b] 配上 act_q_lens[b] 正好圈定每条的有效区间 [q_prefix[b], q_prefix[b] + act_q_lens[b])，padding 落在区间外、不参与计算。这些 padding 槽位占着内存是 BSND 这种布局本身的特性，不是「统一处理」引入的额外成本。

5不只是 Q：凡是「逐 token」的张量，都搭同一班 reshape

上一节讲的是 Q 怎么走。但 Q 不是唯一一个「逐 token」的张量。在稀疏场景（SCFA）里还有一个 cmp_sparse_indices —— 它记录每个 query token 选中了哪几个压缩 KV 块（稀疏注意力只看挑出来的少数块，而「挑哪些」是逐 token 各选各的）。

关键点：cmp_sparse_indices 既然是逐 query token 的，就必须和 Q 待在同一套 token 布局里 —— 内核取「第 g 个 token 选了哪些块」时，用的是和取 Q 完全相同的那个 g = q_prefix[b] + s。两者必须对得上，否则就张冠李戴了。

5.1 它的形状：前面跟着 token 走，后面是每 token 的载荷

cmp_sparse_indices 的形状是 [B, S, Nkv, K]（BSND）或 [T, Nkv, K]（TND），拆成两截看：

  • 最前面的 token 维（[B, S] 或 [T]）：和 Q 一模一样地跟着布局走；
  • 后面的 [Nkv, K]：每个 token 自己的载荷（Nkv 个 KV 头、每头 K 个选中索引），和布局无关 —— 就像 Q 的 [N, D] 一样，原样不动。

所以它的处理和 Q 同一招、同一个理由：TND 透传、BSND 一次免费 reshape 合并最前面的 B、S 两维。

5.2 一条统一规则，覆盖所有逐-token 张量

把这件事说穿：布局差异只存在于「最前面那一截 token 维」。任何「逐 token」的张量，处理方式都一样 —— token 维跟着 reshape，载荷维原样不动。这一条规则，一次性覆盖了 Q、输出、LSE、以及 cmp_sparse_indices：

逐-token 张量BSND 形状展平 / TND 形状跟 token 走的维每-token 载荷（不动）
Q（输入）[B, S, N, D][T, N, D]B, S → TN, D
cmp_sparse_indices[B, S, Nkv, K][T, Nkv, K]B, S → TNkv, K
输出 out[B, S, N, D][T, N, D]B, S → TN, D
LSE[B, S, N][T, N]B, S → TN

于是整套流程里只有一处需要按布局分情况：把这些逐-token 张量的 token 维 reshape 一下（TND 透传、BSND 免费 view），外加算一张 q_prefix。其余一切 —— 内核怎么寻址、怎么算 —— 完全不分布局。

用例子收一下：还是 3 条序列、长度 2/3/2。cmp_sparse_indices 在 BSND 下是 [3, 3, Nkv, K] → reshape 成 [9, Nkv, K]（含 A、C 各 1 个 padding token 的槽位，正好和 Q 展平后的 9 格对齐）；TND 下是 [7, Nkv, K] 直接透传。内核要查「序列 B 第 0 个 token（b=1, s=0）选了哪些块」，算的还是 g = q_prefix[1] + 0（TND 得 2、BSND 得 3）—— 和取 Q 那一格是同一个 g，分毫不差。

6host 侧代价：到底做了什么、有多便宜

把第 4 节 host 侧那几行摊开算账。先说结论：

全程没有任何 host 侧布局转换。那个唯一的「大」张量 —— Q（total_tokens × N × D，动辄几十上百 MB）—— 从头到尾一个字节都没被复制过。host 只造了几张 O(B) 的小账本。

5.1 「reshape 为什么是免费的」

这是整件事最关键、也最容易没想透的一点。BSND 的 [B, S, N, D] 在内存里是连续排列的（行优先，一个挨一个）。当你把相邻的 B 和 S 两维合并成 B×S 时，元素的实际排列顺序、步长统统没变 —— 变的只是「我们怎么称呼这块内存的形状」。

所以 reshape(B*S, N, D) 做的事是：PyTorch 改一下「形状元数据」（shape / stride 这么几个小数字），底层那块大数据连碰都不碰。新旧张量共享同一块内存，零拷贝、零延迟、零额外显存。

一句话：BSND 和「展平的 TND」本来就是同一块内存的两种叫法。reshape 只是换个叫法，不搬家。

⚠️ 前提是张量连续（contiguous）。如果调用方传进来的 Q 是某个非连续的视图（比如刚做过 transpose 没 contiguous），那次 reshape 才可能触发一次真实拷贝。但那是调用方布局没整理好的问题，不是「统一按 TND」这套逻辑引入的。常规情况下 Q 是连续的，reshape 就是纯 view。

5.2 量级对照

把两边的体量摆一起就一目了然：

  • host 侧造的小账本：q_prefix、act_q_lens 等，量级 O(B) —— B 是序列条数，通常几十。
  • 那个大张量 Q：量级 O(T×N×D) —— 百万到千万级元素。

两者差了好几个数量级。host 这点活，相对内核里随后要做的成片矩阵乘法，完全可以忽略不计。

7反面教材：真去做 BSND↔TND 物理转换会多贵

为了体会「统一按 TND」省下的是什么，设想一个笨办法：内核只支持其中一种布局，于是另一种必须在 host 上物理转换过去 —— 真的把数据搬一遍。

  • BSND → TND：要把每条序列的 padding 抠掉、把有效 token 紧凑地重新拷贝到一块新内存。这要读 + 写整个 Q，量级 O(T×N×D)。
  • TND → BSND：要按 S 给每条序列补 padding、散开拷贝到规整方阵。同样是整张大张量的搬运，还更大（含 padding）。

这种「为了喂给内核，先把大张量搬来搬去」的浪费，和「给每条序列预留一大段连续内存、再整段拷贝」是同一类病 —— 真正昂贵的从来不是那几个起点数字，而是动那块大内存。

笨办法：物理转换 BSND↔TND本方案：统一按 TND + q_prefix
拷贝大张量 Q？要（整份读 + 写）不要（reshape 只是 view）
host 复杂度O(T×N×D)O(B)
额外显存多占一整份 Q无
D2H 同步可能多次仅 [B+1] 个 int

「统一按 TND + q_prefix」把左边那一整列大开销整个省掉：内核直接吃调用方原始的内存，host 只额外补一张 O(B) 的起点表。

8收尾：输出按调用方的布局原样还原

内核算完，产出也是扁平的：out_flat [total_tokens, N, D]，以及（如果要）每个 token 的 lse_flat [total_tokens, N]。最后只需按调用方进来时的布局还回去 —— 而且又是零拷贝：

进来的布局输出 out 怎么还原LSE 形状
🟦 TNDout = out_flat —— 本来就是 [T,N,D]，原样返回[T, N]
🟪 BSNDout = out_flat.reshape(B,S,N,D) —— 又一次免费 view[B, S, N]

所以对调用方来说：进来什么布局、出去还是什么布局，中间统一走 TND，两头都是零拷贝的 view，全程无感。

三句话总结

  1. BSND 是「padding 到等长的规整矩形」，TND 是「首尾拼接的紧凑变长」；后者是一般情形，前者是它「等长 + 规则步长 S」的特例。
  2. 内核只认「扁平 token 流 [total_tokens, N, D] + 每条序列起点 q_prefix[b]」；TND 原生就是这形状（透传），BSND 只要一次 reshape（免费 view）+ 一张规则的 q_prefix。布局差异全被压进 q_prefix 消化掉。而且这条「token 维跟着 reshape、载荷维原样不动」的规则不只管 Q —— 输出、LSE、cmp_sparse_indices 等一切逐-token 张量都同此处理（见第 5 节）。
  3. 全程没有大张量的物理转换，host 只造 O(B) 的小账本 —— 统一按 TND 处理几乎零成本，进出两头还都是零拷贝 view。

B（3）：b0 b1 b22, 3, 43, 4, 5
C（2）：c0 c15, 66, 7　(8=pad)
q_prefix[0, 2, 5][0, 3, 6] = b×3

同一批序列、同一个内核 —— 差别只在 q_prefix 这三个数，和 BSND 多出来的两个 padding 槽位。
