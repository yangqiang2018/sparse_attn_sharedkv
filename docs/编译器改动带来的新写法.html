六个上游 PR、八处改动 —— 每一处让 TileLang 里原来写不出来（或者写出来编不过、算错、挂死）的东西，现在能写成什么样子

1整数标量 max/min 发成三元表达式 —— 让运行期夹取在昇腾后端编得过

Bugfix 已合入 2026-07-07 PR #1309

一句话：原来 kernel 里只要出现一个运行期（编译期折不掉）、且两个操作数整数宽度不一致的 T.max / T.min，生成的 AscendC 就会被 bisheng 顶回来 call to 'max' is ambiguous，整个 kernel 编不出来；现在整数标量 max/min 一律打成 (a > b ? a : b)，运行期夹取可以随便写。

1.1　这块机制是什么

非-PTO 的 CodeGenTileLangAscend 继承自 TVM 的 CodeGenC，干的事是把 TIR 打印成一份 AscendC C++ 源码，再交给 bisheng 编译。TIR 里 T.max(a, b) / T.min(a, b) 是 MaxNode / MinNode 两种表达式节点，CodeGenC 的默认打法是发一个裸的、不带命名空间的 max(a, b) / min(a, b) 函数调用，把类型决议甩给 C++ 重载。

生成源码的 prelude 只 include 了 tl_templates/ascend/common.h、acl/acl.h、runtime/rt_ffts.h（codegen_ascend.cc:102-104），并没有 <algorithm>，也没有 using namespace std。所以这个裸 max 解析到哪个重载，完全取决于 CANN 头文件链恰好拖进来了什么。

另一个关键前提：编译期能算出结果的 max/min 会在 Simplify 阶段被折成 IntImm 常量，根本走不到 codegen。只有依赖运行期值（张量取值、grid 变量、T.alloc_var 游标、T.symbolic 形状）的 max/min 才会真的被打印出来。

1.2　原来写不出来的东西

内核作者想写的是「运行期整数夹取」——滑动窗口的左边界要夹到序列开头 0、分页 KV 一次 DMA 能搬多少行要在「本物理页剩余行」和「窗口还差多少行」之间取小、tile 尾段的实际列宽要在 tile 宽和剩余 token 数之间取小。这些量依赖 seqused_kv / act_q_lens / block_table 或 grid 变量，编译期一个都定不下来。

改动前会这么写
BI, ORI_WIN_LEFT = 128, 127        # 编译期常量放 prim_func 外层

@T.prim_func
def main(K: T.Tensor((S, D), "float16"), O: T.Tensor((S, D), "float16")):
    with T.Kernel(core_num, is_npu=True) as (cid, vid):
        k_ub = T.alloc_ub((BI, D), "float16")
        # cid 是 grid block 变量,codegen 发成 auto cid = AscendC::GetBlockIdx(),
        # auto 取的是 GetBlockIdx() 的 int64 返回类型。它对 int 字面量 0 做整数
        # 夹取,就是一个操作数宽度不一致的 Max 节点。
        ori_left = T.max(cid * BI - ORI_WIN_LEFT, 0)   # <-- 这一行就是雷
        T.copy(K[ori_left, 0], k_ub)
        T.copy(k_ub, O[cid * BI, 0])
会发生什么不是算错，是根本编不出来。codegen 把夹取内联成裸调用 max((cid * 128 - 127), 0)，直接塞进 DMA 的行偏移表达式里；bisheng 编这份生成源码时报 error: call to 'max' is ambiguous —— max(int64_t, int) 的多个重载同样可行，决议不出来。报错点落在生成的 C++ 临时文件上，回溯不到 Python 源码的那一行，第一次撞上只看得到一坨陌生的行号。

触发条件比「任何运行期整数 max/min」窄。commit 正文把范围限定为 mixed integer widths：歧义来自宽度不一致的组合（典型就是 auto cid = AscendC::GetBlockIdx() 拿到的 int64 对上 int 字面量，见 codegen_ascend.cc:749）。同宽的运行期夹取（两个 int32_t）在改动前也能编过 —— 仓里 testing/python/language/test_tilelang_ascend_language_alloc_codegen.py 早在本 PR 之前就带着 T.min(prefix_len_b, block_N) 在 target="ascendc" 下通过。

为什么在内核侧绕不过去：

  1. 夹取结果紧接着就要当 T.copy / copy_pa 的切片上下界和搬运行数用（ori_kv[pv_ph, pv_rem : pv_rem + pv_run, ...]），必须是表达式。T.if_then_else 会 lower 成语句（int32_t condval; if {...}），塞不进 copy 的参数表 —— kernel.py 自己在 1961-1969 行把这条写进注释了。
  2. 唯一的表达式级替代是每处手写 tir.Select(a > b, a, b)。kernel.py 里 T.min( / T.max( 合计 37 处（T.min 28 + T.max 9），人肉替换啰嗦且漏一个就炸，而这正是编译器现在自动做的事。
  3. 更关键的是 Max 节点不只来自手写的 T.max：编译器自己的 pass 也会造。src/transform/ascend_storage_rewrite.cc:794 的 combo_size = max(combo_size, sz_nbits)（其中 sz 在超 int32 时被 make_const(DataType::Int(64), ...) 造成 int64，正是混宽场景）、src/transform/merge_shared_memory_allocations.cc:355 的 max_inner_offset = max(max_inner_offset, inner_offset)。内核侧一个 T.max 都不写，也不保证没有整数 Max 节点走到 codegen。

1.3　编译器改了什么

  • src/target/codegen_ascend.cc:436-449：新增 PrintIntMinMaxTernary(const PrimExpr &a, const PrimExpr &b, const char *cmp, std::ostream &os)，把节点打成 (a cmp b ? a : b)，用三元彻底绕开重载决议。注释里写明了不用 std::max<T> 的理由 —— 生成的 AscendC kernel prelude 没有 include <algorithm>。
  • src/target/codegen_ascend.cc:454-468：override VisitExpr_(const MaxNode *op, ...) 传 ">"、VisitExpr_(const MinNode *op, ...) 传 "<"。两者用同一个守卫（455 / 463 行）if ((op->dtype.is_int() || op->dtype.is_uint()) && op->dtype.lanes() == 1)，不满足就 CodeGenC::VisitExpr_(op, os) 原样落回。
  • src/target/codegen_ascend.h:54-57：声明这两个 VisitExpr_ 和 PrintIntMinMaxTernary；和邻居 FloorDivNode / FloorModNode 一样不加 final，与加了 final 的 MulNode / SelectNode 那几个区分开。
  • testing/python/language/test_tilelang_ascend_language_integer_minmax_index.py：新增回归单测（111 行）。三个刻意设计：用 grid 变量 cid（int64）对 int 字面量强制造出混宽的运行期 max/min；把夹取结果当成 DMA 的源行偏移用掉，让它逃过 DCE 真正走到 codegen；target 参数化跑 ["ascendc", "pto"]，op 参数化跑 ["max", "min"]。session 级 autouse fixture 先 tilelang.cache.clear_cache()，否则缓存 kernel 会把 codegen 回归整个掩盖掉。

本节所有编译器侧行号以 fork 分支 feat/ascend-kernel-driven-cube @ 3a546966（已含全部已合入 PR）为准。在合入点 cf1ae11d 上，codegen_ascend.cc 的这批位置整体前移一行（PrintIntMinMaxTernary 落在 435 行），codegen_ascend.h 的行号不变。

1.4　现在可以这么写

没有新 API，签名一个字都没变。写的还是 T.max(a, b) / T.min(a, b)（由 tilelang/language/__init__.py 的 from tvm.script.parser.tir import * 从 TIR 透传进来），变的只是它们在昇腾后端被打印成什么。生效条件严格限定为「整数或无符号整数」且「lanes() == 1 的标量」。

最小例子
import tilelang
import tilelang.language as T

M, N, BM = 1024, 256, 128
M_NUM = M // BM        # 8 个 block
ROWS = BM // 2         # 编译期常量必须留在 prim_func 外层

VEC_PASS_CONFIGS = {
    tilelang.PassConfigKey.TL_ASCEND_AUTO_CV_COMBINE: True,
    tilelang.PassConfigKey.TL_ASCEND_AUTO_CV_SYNC: True,
    tilelang.PassConfigKey.TL_ASCEND_AUTO_SYNC: True,
    tilelang.PassConfigKey.TL_ASCEND_MEMORY_PLANNING: True,
}

@T.prim_func
def main(A: T.Tensor((M, N), "float"), C: T.Tensor((M, N), "float")):
    with T.Kernel(M_NUM, is_npu=True) as (cid, vid):
        # cid 是 int64 的 grid block 变量,和 int 字面量 0 混宽做整数夹取。
        src = T.max(cid - 1, 0)
        a_ub = T.alloc_ub((ROWS, N), "float")
        voff = vid * ROWS
        T.copy(A[src * BM + voff, 0], a_ub)     # 读「夹取后的邻居 tile」
        T.copy(a_ub, C[cid * BM + voff, 0])     # 写自己这块 tile

func = tilelang.compile(main, out_idx=[-1],
                        pass_configs=VEC_PASS_CONFIGS, target="ascendc")

重点只有 src = T.max(cid - 1, 0) 这一行：cid 是 int64 的 grid 变量，0 是 int 字面量，混宽标量整数 max。改动前这行打成裸的 max(cid - 1, 0)，bisheng 报 call to 'max' is ambiguous，整个 kernel 编译失败；改动后打成 ((cid - 1) > 0 ? (cid - 1) : 0)，没有重载需要决议，结果就是「每块读它左边那块、第 0 块读自己」。另外三处不是凑数：把 src 用作 DMA 行偏移是为了让夹取活到 codegen（只算不用会被 DCE 删掉，问题就复现不出来）；M_NUM / ROWS 必须定义在 @T.prim_func 外层，否则会被 parser 绑成 Let-Var、把 a_ub 的 shape 变成非 IntImm；pass_configs 也不能省 —— _TARGET_PASS_DEFAULTS["ascendc"] 是空 dict，不显式开 AUTO_SYNC 的话 MTE2 和 MTE3 之间不插 flag，两条 DMA 无依赖就是数据竞争。

1.5　kernel.py 里的真实用法（一）：SWA 滑窗左边界

kernel.py:3506-3512 · SWA · QK 打分
act_q = act_q_lens[b]
if s < act_q:
    act_kv = seqused_kv[b]
    tok = q_prefix[b] + s
    s_global = act_kv - act_q + s
    ori_left = T.max(s_global - ori_win_left, 0)
    win = s_global + 1 - ori_left

这是 SWA 每个 query token 进 QK 之前必须先算出来的两件事：它能看到的 KV 区间从哪开始、有多长。s_global = act_kv - act_q + s 是该 token 在整条 KV 序列上的全局位置，由 seqused_kv[b]（该 batch 真实 KV 长度）和 act_q_lens[b]（该 batch 真实 query 长度）算出，两者都是运行期从张量里读的。滑窗左边界本该是 s_global - ori_win_left（滑窗左半径，本仓断言 ≤127），但对序列开头那几个 token 这个值是负的，必须夹到 0，否则 win 会算成大于 128、后面 T.copy 的切片直接越界读到别的 batch 的 KV。夹完之后 win = s_global + 1 - ori_left 就是该 token 真实的窗口长度：序列开头短、往后稳定在 128。这两个值一路喂给 copy_pa 的搬运行数、gemm 的 real_k / k_actual、以及在线 softmax 的有效列数，是整个 SWA 的骨架尺寸。

这里的操作数目前都在 int32 域内（3504-3505 行 b = T.cast(pid // max_seq, "int32") / s = T.cast(pid % max_seq, "int32") 已经把 grid 变量降到 int32），所以这条具体的 T.max 在改动前也能编过。改动的价值在于把「能不能编过」从「上游有没有记得 cast」这个约定里解耦出来 —— 少一个 cast、或者哪个 pass 自己造出一个 int64 操作数，就不再是编译失败。

1.6　kernel.py 里的真实用法（二）：分页 KV 一次能搬多少行

kernel.py:3833-3855 · SWA · PV 分页搬运
pv_done = pv_cur - pv_start
if pv_done < winm:
    pv_lg = pv_cur // ori_block_size
    pv_ph = ori_block_table[bm, pv_lg]
    pv_rem = pv_cur % ori_block_size
    pv_run = T.min(
        ori_block_size - pv_rem,
        winm - pv_done,
    )
    T.copy(
        ori_kv[
            pv_ph,
            pv_rem : pv_rem + pv_run,
            0,
            nl * PV_NW : nl * PV_NW + PV_NW,
        ],
        kv_ring[
            slot,
            pv_done : pv_done + pv_run,
            0:PV_NW,
        ],
    )
    pv_cur += pv_run

这是 PagedAttention 的分页走查循环：窗口内的 V 在显存里不连续，按 ori_block_size 行一页散落在 ori_kv 里，页的物理编号要经 ori_block_table[bm, pv_lg] 查表。每轮搬一段连续行，段长 pv_run 同时受两个上界约束 —— ori_block_size - pv_rem 是「当前物理页从 pv_rem 往后还剩多少行」（跨页得换 pv_ph，不能一口气搬过去），winm - pv_done 是「本 token 的窗口还差多少行没搬完」（搬多了会把窗口外的 KV 混进 PV 累加，结果静默错）。取小之后 pv_run 同时出现在源切片、目的切片和游标推进三个位置 —— 源和目的两侧都切，运行期长度才真的落到 DMA 的 validRow 上。三个操作数里只有 ori_block_size 是编译期常量，pv_rem / winm / pv_done 全是运行期值，这个 min 折不掉、必然走到 codegen。这也是「绕不过去」最直白的例子：pv_run 是切片边界，只能是表达式，T.if_then_else lower 成语句就塞不进 T.copy 的参数表。

1.7　kernel.py 里的真实用法（三）：SCFA 压缩域尾 tile 列宽

kernel.py:1957-1978 · SCFA · 压缩域 tile 划分
is_ori = tile == 0
ori_left = T.max(s_global - ori_win_left, 0)
win = s_global + 1 - ori_left
rel = tile - 1
# tile column width. Inner cmp-tail conditional
# is T.min (last tile: act_cmp-rel*512 < 512;
# else >=512 -> 512); the ori/cmp pick is a
# tir.Select. Both lower to an INLINE ternary
# (SelectNode); a runtime T.if_then_else instead
# lowers to a statement (int32_t condval; if{}),
# which the codegen cannot inline into the copy_pa
# arg list (bad C++). For ori (rel=-1) the cmp
# branch = min(512, act_cmp+512)=512, unused.
tw = tir.Select(
    is_ori,
    win,
    T.min(
        S2_BASE,
        T.min(act_cmp, topk_cmp)
        - rel * S2_BASE,
    ),
)

SCFA 的 KV 分两段：一段是原始滑窗（is_ori，tile 0），一段是被 topk 选中并合并过的压缩 KV（tile ≥1，每块 512 宽）。上面 1951-1954 行先算 act_cmp = (s_global + 1) // cmp_ratio（该 token 在压缩域上可见的 token 数）和 cmp_tiles = (T.min(act_cmp, topk_cmp) + S2_BASE - 1) // S2_BASE，topk_cmp 是 topk 只保留 K 个压缩块的上限，取小才是真正要算的压缩 token 数，直接决定这个 token 要跑几轮 tile 循环。这里的 tw 是当前 tile 的实际列宽：压缩分支里外层 T.min(S2_BASE, ...) 处理尾 tile 不足 512 的情况，tw 随后在 2010 行被切成 ncols = T.min(BI, tw - cb * BI) 的 128 宽列块喂给 gemm（CFA 的 505 行是同款代码，属于另一个 kernel）。

代码里那段原生注释顺带解释了为什么只能用内联三元类的东西（tir.Select / T.min）而不能用 T.if_then_else —— 后者 lower 成语句，codegen 没法把它内联进 copy_pa 的参数表，生成的是编不过的 C++。这正是本条改动「为什么必须在编译器侧做」的另一面证据。同时这也是全仓嵌套最深的一处夹取，直接对应 1.8 里的三元膨胀问题。

1.8　兼容性与坑

兼容性：浮点和向量路径的产物逐字节不变，靠的是守卫式落回 —— 两个 override 都是 if ((op->dtype.is_int() || op->dtype.is_uint()) && op->dtype.lanes() == 1) { 新路径 } else { CodeGenC::VisitExpr_(op, os); }，浮点标量以及任何 lanes() > 1 的向量 max/min 走的还是原封不动的 CodeGenC 打印，NaN 语义和向量语义一个字都没碰。整数标量这一侧则是语义等价、文本改变：仓里本来就存在运行期整数 max/min（test_tilelang_ascend_language_alloc_codegen.py 里的 T.min(prefix_len_b, block_N)、examples/tile_kernels/moe/moe_topk_gate.py:55 的 T.min(batch_idx_0 * VEC_NUM + vid, num_tokens - 1)），它们的生成源码会从裸调用变成三元。改动只动 codegen_ascend.cc 和 codegen_ascend.h（加上新增的单测，全 PR 共 3 个文件、156 行净增），codegen_ascend_pto.cc 完全没碰。

PR 描述里那句 Existing kernels only used compile-time-foldable T.max / T.min, so the gap was latent. 与仓内实际不符：alloc_codegen 那个测试在本 PR 之前就带着运行期标量整数 T.min 跑 target="ascendc"。所以「潜伏未被触发」只对混宽那一类成立，不能推出「老代码产物逐字节不变」。

上游回归单测的 docstring 写着 pto already emitted std::max / std::min，这句是错的，值得回上游订正。src/target/codegen_ascend_pto.h 的 overload 列表里没有 MaxNode / MinNode，codegen_ascend_pto.cc 里也没有对应的 VisitExpr_；pto 走的是和修复前 ascendc 完全同一条 CodeGenC 裸调用路径，prelude（codegen_ascend_pto.cc:474-477）同样没有 <algorithm>。文件顶部那个 #include <algorithm> 和 220 / 436 / 1146 行的 std::max / std::min 是 codegen 这份宿主 C++ 自己在用，跟生成的 kernel 无关。测试里 pto 用例能过，只说明 pto 工具链的头文件恰好能无歧义解析该组合，不代表 pto 结构上免疫；真要同样保险，pto 侧得照样加 override。

  • 只管标量整数。守卫是 (is_int() || is_uint()) && lanes() == 1。浮点标量的 T.max / T.min 仍走 CodeGenC 默认的裸 max(a, b)，同样的歧义风险原样还在，只是本仓还没撞到。真要做运行期浮点夹取，优先用向量原语（ascend_max / clamp_max 一类），或自己写 tir.Select(a > b, a, b)。
  • 三元把每个操作数各打印两遍，嵌套按 2 的幂膨胀。PrintIntMinMaxTernary 里 a 和 b 各 PrintExpr 两次，所以每嵌一层，内层节点的文本翻一倍。上面 1973-1977 行的 T.min(S2_BASE, T.min(act_cmp, topk_cmp) - rel * S2_BASE)：外层把操作数 b 打两遍，内层 min 的三元文本出现 2 次，它的叶操作数 act_cmp / topk_cmp 各出现 4 次；再往里嵌一层，最内层的叶操作数就是 8 次。写多层夹取时留意生成源码的体积和可读性，必要时先把内层结果落到 T.alloc_var 标量上。（SelectNode 走 PrintExpr 返回字符串、每个分支只印一次，不额外放大。）
  • dump codegen 看不到三元 ≠ 改动没生效。编译期能折的 max/min 在 Simplify 阶段就成了 IntImm，压根到不了 VisitExpr_(MaxNode)。要验证这条改动，操作数里必须至少有一个是真的运行期值（张量取值、grid 变量、alloc_var 游标）。
  • 只对 target="ascendc"（非-PTO）生效。pto 是另一份 codegen，没有这两个 override（见上面的 warn）。本仓 SWA / CFA / SCFA 全部走非-PTO codegen，所以这条是刚需；在 pto 上验证是验不出差别的。
  • 三元只解决重载决议，没解决整型提升。a 和 b 的符号扩展 / 整型提升还是 C 的老规矩：int64 的 grid 变量对 int 字面量取 max，结果类型是 int64。夹完要当 int32 索引用的话，仍然建议显式 T.cast(..., "int32") —— kernel.py:3504-3505 的两个 T.cast 就是先把 grid 变量降到 int32、让后面所有夹取都留在 int32 域的写法，值得照抄。
  • 生成的表达式外层已经带括号（os << "(" … os << ")"），可以直接嵌进更大的算式，不用自己再包一层。
  • 别指望它顺带修好整数除法。同一份 codegen 里的 FloorDiv / FloorMod 发成 C 的 / 和 %（截断语义），负被除数下和 TIR 的 floor 语义差 1，那是另一个独立的坑。夹取用 max/min、取页号用 // 的组合里，如果操作数可能为负，先确认除法那一侧。

2gemm_v0 在模板内部按列切 N，让 headDim=512 的 PV 矩阵乘能一句写完

Bugfix 已合入 2026-07-14 PR #1312

一句话：原来 gemm_v0 只切 K 不切 N，N 一大（PV 的 headDim=512）就要把 128KB 的 B 片塞进整块只有 64KB 的 L0B，运行期把 cube 打崩；现在模板内部自动按 128 列切 N，一句 T.gemm_v0 就能写完整个 PV。

2.1　这块机制是什么

昇腾 cube 单元做一次 mma，两个操作数必须先从 L1 搬到 L0A / L0B 这两块各 64KB 的片上缓冲，累加器落在 L0C。gemm_v0 是 TileLang Ascend 后端的块级矩阵乘模板（src/tl_templates/ascend/common.h），它把 <M, N, K> 的 gemm 拆成若干次 mma，并用 L0A/L0B 双缓冲（乒乓）把「L1→L0 搬运」和「上一片的 mma」重叠起来。

预算是分段的：整条 (N-tile, K-tile) 序列只有一步时，单槽可以占满 64KB；一旦步数大于 1、乒乓真的开始轮转，每槽就只剩 32KB。改动前 gemm_v0 只在 K 方向切（kL0Size=128），B 操作数在 L0B 里的槽大小恒等于 N * kL0Size——N 有多大就占多大，模板自己完全不管它装不装得下。

2.2　原来写不出来的东西

attention 的第二个矩阵乘 PV（O = P @ V）想按 Ascend C 参考的 ComputeMm2 那样一句写完：M=G=64、N=headDim=512、K=window≤128，P 和整块 V 各在一个 L1 buffer 里，结果累加进一块 [64, 512] 的 L0C。

改动前会这么写
# 想写成这样（sparse_attn_sharedkv 早期 SWA，一句 PV）：
kv_l1 = T.alloc_L1([BI, D], dtype)        # [128, 512] 整窗 V
p_l1  = T.alloc_L1([G, BI], dtype)        # [64, 128]  P
acc_o_l0c = T.alloc_L0C([G, D], accum_dtype)

# 实际当年被迫写成这样（K 劈两半的绕法）：
kv_l1_0 = T.alloc_L1([BI // 2, D], dtype)
kv_l1_1 = T.alloc_L1([BI // 2, D], dtype)
p_l1_0  = T.alloc_L1([G, BI // 2], dtype)
p_l1_1  = T.alloc_L1([G, BI // 2], dtype)
T.copy(workspace_p[cid, :, 0 : BI // 2], p_l1_0)
T.copy(workspace_p[cid, :, BI // 2 : BI], p_l1_1)
T.gemm_v0(p_l1_0, kv_l1_0, acc_o_l0c, init=True)
T.gemm_v0(p_l1_1, kv_l1_1, acc_o_l0c, init=False)
会发生什么第一种写法编译得过，运行期挂死：aicore 报 CCU instruction address check error。不是逻辑错，是地址越界——B 片 = K * N * sizeof(half) = 128 * 512 * 2 = 128KB，而整块 L0B 只有 64KB，超了 2 倍。（这个 shape 的 K=128 正好等于 kL0Size，kL0split == 1，旧循环只跑一轮、pp = kL0Idx & 1 恒为 0，乒乓根本不轮转，可用的就是整块 64KB；32KB 的每槽预算是修复后 kNumSteps > 1 才成立的约束。）写穿 L0B 之后 cube 的指令/地址被冲掉，表现为硬件异常而不是干净的报错。

同一个 kernel 的 QK 反而没事：它走 transpose_B 路径，N = block_I = 128、B 片 128 * 128 * 2 = 32KB，K=D2=256 分两个 K 片、乒乓轮转，32KB 正好卡在预算里。所以这个坑只有 N 大的那一侧才踩得到。

为什么在内核侧绕不过去：内核侧只能拿 K 换 N——把 K=BI 的收缩拆成两个 BI/2 半，让每个 B 片降到 512 * 64 * 2 = 64KB。切片操作数本身是能传的（_retrieve_shape / _retrieve_ptr 都有 BufferRegion 分支，tilelang/language/ascend.py:66-71 与 98-111，这两个分支在 #1312 的父提交上就已存在），但 _retrieve_ptr 是按紧凑 row-major 累加 indices[i] * strides[i] 算基址偏移的，而 L1 上的 B 走 zN 分形布局：沿 K 切出来的后半段在 zN 下根本不是一段连续内存，用单个指针偏移表达不出来，写出来也不报错、是静默指到错地址。所以「切一刀」必须真的多申请一组完整 L1 buffer。代价是三层的：

  1. L1 占用翻倍，且窗口被人为劈成两段；
  2. workspace 的读写从 1 次变 2 次，cube 侧和 vector 侧的 cross-core sync 点数要逐个配平（半 0 写↔半 0 读、半 1 写↔半 1 读），错一个就死锁；
  3. 为了配平，vector 侧的 s_ub / P 也得按列半段搬，而列半段的 UB 子块拷贝会把 s_ub 的 stride 搞错、逐行污染 LSE。

更要命的是：这么切之后 tiling 已经和 Ascend C 参考的 ComputeMm2（按 N_SPLIT_SIZE 切 N）对不上了，忠实复刻这个目标本身就废了。真正的切分点在 N 上，而 N 的切分只能发生在模板内部——L0B 的槽划分、乒乓事件、L0C 的列带偏移全都是模板自己管的，内核根本够不着。

2.3　编译器改了什么

  • src/tl_templates/ascend/common.h：新增编译期常量 nMaxByL0B = 32KB / (kL0Size * sizeof(T1))（half + kL0Size=128 时正好是 128，与 Ascend C 参考的 N_SPLIT_SIZE 对齐）、nTile = (transpose_B || N <= nMaxByL0B) ? N : nMaxByL0B、nL0split = N / nTile。
  • src/tl_templates/ascend/common.h：原来的单层 K 循环外面套了一层 N 循环。每个 N-tile 只把自己那 (kL0Size x nTile) 的 B 子块载进 L0B，mma 的模板参数从 <M, N> 改成 <M, nTile>，结果累加进 L0C 自己那条列带 C[cNOffset]。
  • src/tl_templates/ascend/common.h：子块偏移直接照抄 catlass 的 tla 分形布局——L0C 第 n0 列在 n0 * roundUp16(M)（MakeLayoutL0C 的 N1 stride），L1 里 zN 布局的 B 第 n0 列在 n0 * roundUp16(K)（MakeLayout<zN> 的 C1 stride），和循环里原本就在用的 K 偏移 B[kL0Idx * 16 * kL0Size] 同源（那个硬编码的 16 后来被 #1401 换成 ELE_NUM_PER_C0）。
  • src/tl_templates/ascend/common.h：(N-tile, K-tile) 二维序列被拍平成一个 tileIdx，乒乓槽选择改成 pp = tileIdx & 1，L0A/L0B 的 prime/drain（M_MTE1 的 SetFlag/WaitFlag）整段只做一次，让 N-tile 之间也能搬运/计算重叠。第一版「每个 N-tile 各自 drain」会把 N-tile 串行化，被这个拍平取代。
  • src/tl_templates/ascend/common.h：新增两条 static_assert 兜住 L0A/L0B 容量——B 片 nTile * kL0Size * sizeof(T1) 和 A 片 M * kL0Size * sizeof(T1) 都必须 ≤ kL0Budget（单步 64KB、乒乓真轮转时 32KB）；再加一条 N % nTile == 0。
  • testing/python/language/test_tilelang_ascend_language_gemm_v0_n_tiling.py：新增回归测试，覆盖 N=512 的大 N 路径与 N=128 的兼容对照。
前端签名一个参数都没加。切 N 是模板为了塞进 L0B 而做的内部行为，不该让内核作者操心。前端仍然是 T.gemm_v0(A, B, C, transpose_A=False, transpose_B=False, init=False, kL0Size=128, n_actual=None)（tilelang/language/ascend.py:343；其中 n_actual 是后来 #1352 加的，与本条无关），M/N/K 照旧从 A/B/C 三个 buffer 的 shape 推出来（M, N = C_shape[-2], C_shape[-1]）。变化只在语义上：以前「N 必须小到能整个塞进一个 L0B 槽」是一条没写出来的隐式前提，越界就是运行期硬件异常；现在超出单槽的非转置 B 路径会自动切成 nL0split 段，真正装不下的情况改由编译期 static_assert 明确拒绝。

CFA 的 QK 打分：把这个 token 的 64 个 query head（q_l1 的第 0 个 D 半段）和 KV 环里 slot s0 的 K 做 Q @ K 转置乘，结果累加进 cL0 的第 cs 个槽。DEBUG_SERIAL 是一条排障开关：打开时用整块 T.gemm_v0 加 T.barrier_all() 串行跑，关掉时（生产路径，DEBUG_SERIAL = False）走手工解构的 mma 流水。列在这里是为了展示 #1312 的兼容性保证，而不是它的切分能力：这一句是 transpose_B=True，而模板里 nTile = (transpose_B || N <= nMaxByL0B) ? N : nMaxByL0B，转置 B 的路径一律不切，nL0split == 1、tileIdx == kL0Idx，展开成和加 N-tiling 之前同一条单趟路径。M、N 由 cL0[cs] 的 shape [G=64, BI=128] 推出，B 片 128 * 128 * 2 = 32KB，本来就装得下——这也正是 QK 侧从来没踩过 L0B 溢出的原因。

SCFA（topk 稀疏压缩注意力）在 kernel.py:2129-2137 有一处与之逐字符相同的调用，两边的 ncols_a 也是同一个运行期表达式 (ncols + 15) // 16 * 16（506 / 2011）；差别在更上游的 tw——SCFA（1970-1978）多套一层 T.min(act_cmp, topk_cmp) 的 topk 钳位，CFA（477-481）只用 act_cmp。三个 kernel 里还活着的 T.gemm_v0 调用（654 / 735 / 2130 / 2211）全部是 transpose_B 的 DEBUG_SERIAL 调试路径，所以本算子今天已经没有走 N-tiling 的生产调用点——这条能力在本算子里的位置是「先由它解决 PV 的 L0B 溢出、跑通 4×128 的忠实切分，之后被更细的内核驱动拆分接管」；但作为通用 gemm_v0 对大 N 的必要修复，对任何 headDim 较大的 attention 或普通 GEMM caller，它依然是唯一能用的写法。

2.7　兼容性与坑

兼容性：老代码逐字节不变靠两个编译期默认值。第一，nTile = (transpose_B || N <= nMaxByL0B) ? N : nMaxByL0B——只要是转置 B 的 caller，或者 N 本来就小于等于单槽装得下的列数（half + kL0Size=128 时是 128），nTile 就等于 N，于是 nL0split = 1。第二，外层 N 循环只跑一轮时 bNOffset = 0、cNOffset = 0、tileIdx == kL0Idx，L0B 槽偏移 pp * (nTile * kL0Size) 退化回原来的 pp * (N * kL0Size)，mma 的模板参数 <M, nTile> 退化回 <M, N>，L0A/L0B 的 prime/drain 位置也和原来完全一致。所有既有 caller 走的都是和改动前同一条展开路径。唯一新增的是编译期检查：原来会在运行期把 cube 打崩的 shape，现在被 static_assert 在编译期挡下来——把静默故障变成显式失败，不影响任何本来就能正确跑的代码。

  • N 必须能被 nTile 整除，否则编译期报 "gemm_v0 N-tiling requires N divisible by the N tile size"。nTile 在 half + kL0Size=128 时是 128，所以大 N 的非转置 caller 实际要求 N 是 128 的倍数。
  • 只有 transpose_B == false 的路径会被切。转置 B 的 caller 一律保持 nTile == N，所以一个 N 很大的 transpose_B caller 不会被自动救回来，而是撞上 "gemm_v0: the (kL0Size x nTile) B tile does not fit its L0B ping-pong slot"——这是有意的，因为转置 B 在 L1 里的布局不同，列偏移公式不通用。
  • N-tile 的列偏移写死成 nL0Idx * nTile * roundUp16(K)，这是 zN 布局的 C1 stride。B 在 L1 里必须是默认的 zN 布局；换了别的 layout annotation，这个偏移就是错的，而且是静默算错、不报错。
  • M 方向不切。M * kL0Size * sizeof(T1) 必须 ≤ kL0Budget，M 太大会撞 L0A 那条 static_assert。想要更大的 M 只能调小 kL0Size（但那会同时把 nTile 调大，因为 nMaxByL0B = 32KB / (kL0Size * sizeof(T1))）。
  • 乒乓预算是分段的：只有一步（kNumSteps == 1）时单槽可以用满 64KB；一旦 (N-tile, K-tile) 步数大于 1、乒乓真的开始轮转，每槽就只剩 32KB。所以「多加了一个 K 片之后突然编译不过」是预期行为，不是回归。
  • 这条修复只在 ascendc 后端。PTO 后端有一套独立的 gemm_v0（src/tl_templates/pto/common.h 里的 gemm_v0_inner / TileMatL0B），没有被覆盖。
  • 和 n_actual（transpose_B 路径上运行期少算几列）是两个正交机制，不会同时生效：n_actual 只走转置路径，N-tiling 只走非转置路径。别指望用 n_actual 规避大 N 的 L0B 溢出。
L0C 没有 static_assert 兜底。L0C 容量是 device-dependent（A2/A3 128KB、A5 256KB），而 device 侧模板里拿不到 arch 宏（ASCEND_*_L0C_SIZE 只存在于 host 侧 codegen），写死一个字面量会误拒合法的 A5 caller、或者在 A2/A3 上放行超限的 caller，所以模板只在注释里写明约束、由 caller 自己保证。roundUp16(M) * N * sizeof(T2) 超了不会有任何提示。典型陷阱：M=128、N=512、fp32 累加需要 256KB，在 A2/A3 上会静默踩坏 L0C。
回归测试怎么设计的：test_tilelang_ascend_language_gemm_v0_n_tiling.py 构造一个最朴素的 C = A @ B（transpose_B=False），整块 A/B 一次拷进 L1，K 切分和 N 切分全部交给 gemm_v0 内部，再和 a @ b 的 torch 结果比。三个 shape 分工不同：(M=64, N=512, K=128) 就是真实的 PV shape，旧代码下 B 片 128KB、是整块 L0B 的两倍，必挂；(64, 512, 256) 把 K 也切成 2 段，专压拍平后的 (N-tile, K-tile) 流水（2 K-tile × 4 N-tile = 8 步），tileIdx 的乒乓槽映射或 cNOffset/bNOffset 算错时是静默错值而不是崩，只有真跑数值比对才抓得到；(128, 128, 128) 则是旧代码也能跑通的兼容对照组（K 只有一片、kNumSteps == 1，单槽预算就是整块 64KB，而 L0B 槽 128 * 128 * 2 = 32KB、L0A 槽同样 32KB），用来证明 nL0split == 1 时新循环和原路径等价。dtype 参数化跑 float16 和 bfloat16（bf16 在这个 N/K 下累加误差更大，容差放到 2e-2）。另有一个 session 级 clear_cache fixture 先清 tilelang 缓存——否则一个陈旧的已编译 kernel 会把 codegen 回归整个掩盖掉。M 取 64 而不是 128 也是刻意的：N=512、fp32 累加时 L0C 需要 roundUp16(M) * N * 4，M=128 就要 256KB，只有 A5 放得下。

3GM↔UB copy / atomic_add 的最内（列）维可以是运行期表达式了 —— 按真实窗口宽度搬，而不是按 buffer 定宽搬

Bugfix 已合入 2026-07-17 PR #1337

一句话：原来 GM↔UB 的 T.copy 只能切编译期常量宽度，一旦切成运行期宽度 [..., 0:tw] 就生成非法 C++、内核根本编译不过；现在运行期内层 extent 合法，能只搬真实有效的那几列。

3.1　这块机制是什么

昇腾非-PTO（ascendc）codegen 里，GM↔UB 的搬运最终落到 src/tl_templates/ascend/common.h 的两个 C++ 模板函数：copy_gm_to_ub<T, dstN, dstM> 和 copy_ub_to_gm<T, srcN, srcM>。列维 dstN / srcN 是编译期模板实参，语义是 UB tile 的行距 —— DataCopyPad 用 (dstN - maskShapeN) 算行间 stride；而这一次实际要搬多少列，由运行期函数参 maskShapeN 承载（lowering 里叫 validCol_src / validCol_dst，由 compute_valid_extent 从 region 的 extent 算出来）。

lowering 侧 AscendCopy::Lower 是用 std::stringstream 把模板名和模板实参拼成一个字符串常量，模板实参直接 ss << PrimExpr 打印进去。所以塞进模板的 PrimExpr 必须是编译期常量，否则打印出来的是 TVMScript 文本，原样进了生成的 .cpp。同一处的行维模板参 dstM（走 compute_blocklen）早就有 IsInstance<IntImmNode>() 守卫，唯独列维漏了。

3.2　原来写不出来的东西

cfa / scfa 的 vector softmax，UB 和 GM workspace 都是定宽 S2_BASE = 512，但真实窗口宽度（ori 窗 ≤ 128，cmp tile ≤ 512）只有运行期才知道。内核作者想对齐 Ascend C 参考实现 block_vector.h 的 DealBmm1（它只处理 columnCount = actualSingleProcessSInnerSizeAlign 列），把 T.copy 切成运行期宽度，只搬、只写有效的那几列。

改动前会这么写
# UB / GM workspace 都是定宽 S2_BASE = 512,
# 但真实窗口宽度只有 64 / 128, 且要等 seqused_kv / act_q_lens 读出来才知道
tw = tir.Select(is_first, win, T.min(S2_BASE, act_cmp - rel * S2_BASE))
tw_a = (tw + 15) // 16 * 16          # 纯运行期表达式

# 想只搬有效的那几列(= 参考实现 DealBmm1 的 columnCount)
T.copy(
    workspace_s[cid, buf, r0 : r0 + M_CHUNK, 0:tw],
    in_ub[ps, :, 0:tw],
)
# ...
T.copy(
    out_ub[:, 0:tw_a],
    workspace_p[cid, buf, r0 : r0 + M_CHUNK, 0:tw_a],
)
会发生什么编译期直接挂，不是算错、也不是挂死 —— 根本编不出二进制。lowering 把 region 的运行期 extent ss << 进了模板实参，PrimExpr 的流式输出是 TVMScript 文本，于是生成的 .cpp 里出现形如 copy_gm_to_ub<float, T.min(512, act_cmp - rel * 512), 16>(...) 的东西，编译器报 error: use of undeclared identifier 'T'。那个 T 是 TVMScript 的命名空间前缀被原样打印进 C++ 的结果，跟模板里的 typename T 无关，纯属撞名。

为什么在内核侧绕不过去：

  1. 数据通路本来就是通的，只差模板那一格。copy_gm_to_ub 的 C++ 侧早就有运行期的 maskShapeN 参数，compute_valid_extent 也一直在正确下传 validCol_*。但前端 T.copy 没有「只给 mask、不切 region」的口子：不切 → extent == shape == 512（满宽搬，零收益）；切 0:tw → 撞模板 bug。死结只能在 codegen 解。
  2. 没有别的原语能替代。copy_l0c_to_gm 是 L0C→GM，copy_pa 是 paged gather，都发不出「运行期宽度 + strided 连续 GM↔UB」这条 DMA。
  3. 退回定宽 512 搬的代价。读侧会把窗口外上一轮遗留在 workspace_s 里的 stale QK 一并读进 UB，而后面定长满宽的 reduce_max / reduce_sum 会读到这段区间 —— 吃的是上一轮的残留分数（内容不可控），lse 与 softmax 分母直接错；写侧每个 m-chunk 都按 512 列满宽写回，而 ori 窗那一支实际只有 ≤ 128 列有效，其余全是废数据 —— 满宽搬运下 msprof 上 CFA vector 的 mte2 是 Ascend C 的 1.8 倍（1722 vs 954）。
  4. 穷举编译期宽度也不行。tw 由 seqused_kv / act_q_lens 在运行期决定，1..512 全枚举成静态分支是代码爆炸。

3.3　编译器改了什么

  • src/op/ascend.cc · AscendCopy::Lower（gm2ub 分支）：模板列维改成「extent 是 IntImm 就照用（既有调用逐字节不变），是运行期表达式就回退用 dst->shape[last]」，回退值再 ICHECK 一次必须是编译期常量。
  • src/op/ascend.cc · AscendCopy::Lower（ub2gm 分支）：同一套 IntImm 判定，回退用 src->shape[last]。
  • src/op/ascend.cc · AscendAtomicAdd::Lower：抽出一个 atomic_tmpl_n 按同样规则算好，同时喂给 atomic_add_ub_to_gm<T, N, M> 和非-UB 分支的两处模板实参（原来那两处各自直接用 src_extents[last]）。
  • 新增 testing/python/language/test_tilelang_ascend_language_copy_runtime_extent.py（182 行）作为回归测试：GM 张量与 UB tile 全部定宽 512，运行期宽度 n 用 T.symbolic("n") 通过一个 dummy 输入的 shape 带进 kernel，gm2ub / ub2gm / atomic_add 三条路径都切 [..., 0:n]。旧 lowering 在编译阶段就崩，测试连跑都跑不到，不需要靠数值断言来抓。

API 签名一个字没改（T.copy 实为 copy_op.py:257 的 npu_copy_v2，T.tile.atomic_add 在 ascend_tile.py:191），变的只是这两个 API 能接受的 region 形态：方向是 GM→UB / UB→GM（含 atomic_add 的 UB→GM）时，region 的最内维 slice 上界现在可以是运行期 PrimExpr —— T.symbolic 变量、tir.Select / T.min 的结果、从 GM 标量算出来的 int32，都行。

需要这个能力的是撤掉 007 之后的这版实现，不是一开始就需要：改动前 vector 侧只能满宽搬（workspace_s[cid, buf, vid * G2 : (vid + 1) * G2, :] → s_ub，整整 128 列），窗口对齐由 007 原语 T.tile.softmax_flash_v2 的 columnCount = win_align / actualColumnCount = winm 两个参数承担，库自己只归约有效列。把 007 换成手拼的定长满宽 reduce_max / exp / reduce_sum 之后，窗口外那段就没人替内核挡了 —— 必须靠运行期 inner-extent copy 把搬运停在 winm，让 [winm:BI] 落一个可控的值，而不是上一轮遗留在 workspace_s 里的 stale QK。win_align 本身也是运行期表达式（(win + 15) // 16 * 16），今天仍然活着，只是退到了 cube 侧：kernel.py:3586 定义、喂给 QK mma 的 n_actual 和 kernel.py:3752 的 L0C→GM copy 宽度；第 3977 行注释里的「不再需要 win_align 对齐」只对 vector 侧这条 copy 成立。

这里的 -inf mask 其实不是靠 fill 保住的。注释说 [winm:BI] 保持前面 T.tile.fill 打的 -inf，但这一步实际没保住：copy_gm_to_ub（common.h:233-241）在 maskShapeN != dstN —— 正是运行期窄搬的情形 —— 会先 AscendC::Duplicate(dstTensor, padValue, dstM * dstN) 把整块刷一遍，padValue 来自 T.copy 的 pad_value 参数、默认 0，而 kernel.py 全文没有一处传过它，空档里落的是 0 而不是 -inf。结论仍然成立是靠另一条链：src/transform/ascend_tail_mask_propagation.cc 的第一批改写就包含 reduce（rewrite_reduce_ 默认 true，只对 PTO 后端关掉），满宽的 T.reduce_max / T.reduce_sum 会被改写成只覆盖有效区的 tail_reduce，空档压根不进归约；unary / binary / scalar 同样被改写成 tail_*，所以空档里的值既不会被逐元素算子破坏、也不会被归约读到。要点：窄搬之后往空档填哨兵值这套写法，真正生效的是 tail 传播而不是那次 fill。如果空档要被某个没有 tail 改写的满宽读者看见（cmp / select 之类），就必须显式传 pad_value，不能指望前面的 T.tile.fill。

softmax 算完、cast 成 fp16 的概率 P 要写回 workspace_p，交给 cube 做 PV。cube 那边的 mma 只吃 tw_a 列（K 就是窗口宽），多写的列纯属浪费。两个分支的宽度都要等 seqused_kv / act_q_lens 从 GM 读出来才知道：第一个 tile 走 ori 窗 tw = win，之后的 cmp tile 走 tw = min(S2_BASE, act_cmp - rel * S2_BASE)；tw_a 再向上 16 对齐 —— 16 是为了迁就 mma 的 K 分形，顺带在 fp16 下正好凑成 32B，满足 DataCopyPad 行间 stride 的块粒度。改动前这一行编不过，只能按定宽 S2_BASE = 512 写回，等于每个 m-chunk 白写 512 - tw_a 列废数据（ori 窗那一支 tw_a ≤ 128，即至少 384 列）；注释里的「= ref width, saves MTE3」就是说这行把宽度对齐到了 Ascend C 参考实现只处理 columnCount 列的行为。

3.7　兼容性与坑

兼容性：走的是「常量原路 / 运行期新路」的默认值机制 —— 模板列维先看 dst_extents[last]（或 ub2gm 的 src_extents[last]）是不是 IntImmNode，是就照用，ss << 出来的模板字符串逐字节不变。而现存所有调用的 inner extent 全是编译期常量：满维 copy 时 extent == shape，常量 slice 时是个常数宽度，所以老代码一行都不会走进新分支。更强的一条论证是反过来的：现有算子全都能编译，这本身就证明它们从没用过运行期 inner-extent（用了就会当场崩），因此回归验证的目标是「确认它们确实没用过」，而不是「验证行为变化是否符合预期」。

  • 回退用的是 buffer 的编译期 shape，也就是模板列维 = UB 的行距。所以 UB buffer 自己的最内维必须是编译期常量；把 buffer 声明成动态宽度仍然会撞 ICHECK（copy_gm_to_ub: the inner (column) dimension of the destination buffer shape must be a compile-time constant, but got ...，ub2gm / atomic_add 各有对应措辞的一条）—— 只是从「生成非法 C++、编译器报一句莫名其妙的 undeclared identifier」变成了「lowering 早失败、报一句人能看懂的话」。
  • 空档里的值由 pad_value 决定，不是 UB 里原有的内容。copy_gm_to_ub 在 maskShapeN != dstN（或 maskShapeM != dstM）时先 AscendC::Duplicate(dstTensor, padValue, dstM * dstN) 刷整个 tile，padValue 默认 0。想要 -inf mask 就显式传 pad_value=-T.infinity(dtype)，别指望 copy 之前自己 T.tile.fill(-inf) 的内容能活下来。这段 Duplicate 还自带 MTE2_V / MTE3_V / V_MTE2 三组 SetFlag·WaitFlag，是笔隐藏开销。
  • Duplicate 还有 dtype 白名单。common.h:41-45 的 IsDuplicateSupported_v 只含 int16_t / uint16_t / half / bfloat16_t / int32_t / uint32_t / float。int8、int64、fp8 这些同样合法的 copy 元素类型走的是「完全不填空档」的路径 —— 空档里既不是 pad_value 也不保证是原内容，满宽计算前必须自己保证它。
  • 只修了列维（最内维）。行维模板参 dstM / srcM 走 compute_blocklen，本来就有 IntImm 守卫；再往外的维度不是模板实参，不受影响。
  • 常量窄 slice 的行为没变（刻意的）。[:, 0:64] 这种编译期常量宽度仍然用 extent 当模板列维，于是模板 dstN == maskShapeN，行间 stride 被算成 0 —— 等于把 UB 当成按 64 紧密排列。只有运行期 extent 才会回退成 buffer shape（真实行距）。同一个 buffer 上混用两种写法时，两者的行布局语义不一样，要留神。
  • 32B 粒度：两个方向用的是同一个 floor 公式。写侧 copy_ub_to_gm 的 src stride 是 (srcN - maskShapeN) * sizeof(T) / 32（common.h:257），读侧 copy_gm_to_ub 的 dst 行间 gap 是 (dstN - maskShapeN) * sizeof(T) / 32（common.h:246），同型、同样截断。安全前提是 buffer 最内维的字节数本身是 32 的整数倍（512 个 fp32 = 2048B，成立）—— 此时 blockLen 向上取整补的那点正好等于 gap 被截掉的那点，行距还原成整行。DataCopyPadExtParams 里的 isPad / rightPadding 只管 GM 侧不足一整块时的补齐，跟 UB 行距的截断无关。kernel.py 的 copy-out 用 tw_a = (tw + 15) // 16 * 16（fp16 → 32B）是迁就 mma 分形顺带满足这条，上游单测也只取 64 / 128 / 256 这种 32B 对齐的 n。
  • 只覆盖 gm2ub / ub2gm / atomic_add 这三条路径，剩下的分支没有逐条穷举。同一个 AscendCopy::Lower 里的 UB→L1（ascend.cc:291，模板列维仍是 ss << src_extents[last]）和 UB→UB（ascend.cc:310，模板长度是 dst_extents 直接连乘）都还是老写法，运行期 inner-slice 一样会崩成同样的 undeclared-identifier。copy_gm_to_l1 / copy_l0c_to_gm / copy_l0c_to_ub 那几条本来就用 shape 填模板，不受影响 —— SWA cube 侧 kernel.py:3752 那句 workspace_s[cid, buf, :, 0:win_align] 就是靠这一点，早在本 PR 之前就能用运行期宽度。
  • 修的是非-PTO（ascendc）codegen。PTO 后端另有一套 copy_gm_to_ub_dynamic（src/tl_templates/pto/common.h:304），不在这次改动范围内，单测 docstring 也写明只针对 ascendc 后端。

4gemm_v0 / mma 的运行期输出列数 n_actual —— 让 QK 只算真实窗口那几列

Feat 已合入 2026-07-17 PR #1352

一句话：原来 gemm 的输出列数只能是编译期模板 N（只能整块 N 列算完，再在下游把尾巴掩掉），现在可以传一个运行期 n_actual，cube 只计算前 n_actual 列。

4.1　这块机制是什么

昇腾 cube 的一次矩阵乘由 MmadParams{m, n, k} 描述。TileLang 的 gemm_v0 把 M/N/K 全做成 C++ 模板参数 —— template<T1, T2, M, N, K, transpose_A, transpose_B, kL0Size>。其中 K 早就在调用时被拆成了运行期实参：模板内部按 kL0Size 把 K 切成若干 kL0 tile，每个 tile 把真实的 kSize 传给 mma。唯独 N 一直是编译期常量 —— copy_l1_to_l0b 按模板 N 列搬 L1→L0B，mma 里写死 mmadParams.n = N。

稀疏 attention 里，每个 query 块真正要打分的 KV 窗口长度是运行期算出来的：SWA 的滑窗长 win 依赖 seqused_kv / act_q_lens，CFA/SCFA 的压缩 tile 按 BI=128 切列块、末块列宽也是运行期值。这一节讲的就是把 N 也变成运行期值：改动前是「M/N 编译期、K 运行期」，改动后 N 跟 K 一样能接一个运行期实参。

4.2　原来写不出来的东西

内核作者想表达的是「这一轮 QK 只算真实窗口那几列」：SWA 里 ori_left = T.max(s_global - ori_win_left, 0)、win = s_global + 1 - ori_left，16 对齐成 win_align；CFA 里 ncols = T.min(BI, tw - cb * BI)、ncols_a = (ncols + 15) // 16 * 16。剩下的 [n_actual:N] 列根本不该进 cube。

          transpose_B=True, init=True)

会发生什么两种形态。① 直接传列数 → 前端签名里根本没有这个关键字，构造 prim_func 时就崩：TypeError: gemm_v0() got an unexpected keyword argument 'n_actual'。② 退回整 N 写法 → 编译能过、跑得动，但 cube 必然算满 128 列，而 [win_align:128] 那段 L0B 装的是这一轮压根没搬进来的内容（SWA 的分页窗口加载带 if pa_done < win 守卫，只填 win 行，其余是上一个 task 遗留在 ring slot 里的 KV），算出来的分数是垃圾值，必须在 softmax 之前额外掩一遍。更硬的一层：mmadParams.n 恒等于 N，就没有任何办法让配套的 0b11 融合 fixpipe 只搬窄带 —— fixpipe 的 nSize 一旦和 mma 的 n 不一致，fixpipe 会去等一批从来没有被任何 mma 标记 ready 的 cL0 列，cube 直接挂死。

为什么在内核侧绕不过去：列数是 C++ 模板参数。内核侧唯一的「变通」是按每个可能的窗口宽度实例化一套模板 —— 但窗口宽度是运行期值，做不到。剩下就只有整 N 算完再掩：多一轮 vector mask，窗口只有 32 列的极端行（序列最开头）白算 4 倍 cube、L1→L0B 白搬 4 倍数据。而 unitFlag 那条路是死的：0b11 的 mma↔fixpipe 融合要求 fixpipe 的 nSize == mma 的 n，n 被钉死成 N 时「窄带 fixpipe」整个封死，只能退回加屏障的串行写法。

省算力是次要收益。本仓 ori_win_left 取 api.py 的默认值 127，这也是它能取到的最大值（kernel.py 直接拒绝 ori_win_left + 1 > BI），于是 win = min(s_global + 1, 128)：只有序列最前面 127 个因果位置的 win 小于编译期 N=128，8K 序列下约 1.5% 的行才真正收窄。CFA/SCFA 那边 512 宽的 cmp tile 切成 4 个 128 列块，也只有末块可能不满。真正让 n_actual 成为必需品的是 0b11 融合 fixpipe 那条硬约束（见 4.7）。

4.8　兼容性与坑

兼容性：n_actual 在 C++ 侧是带默认值 = N 的尾参数，在 tilelang/language/ascend.py 的前端是 n_actual=None → N。老的 mma(A, B, C, init, K)、gemm_v0(A, B, C, l0a, l0b, clear)、T.gemm_v0(A, B, C, transpose_B=True, init=True) 一个字符都不用改，行为完全等价：mmadParams.n 仍然是编译期 N、copy_l1_to_l0b 仍然搬 N 列。非 transpose_B 路径更是显式传 nTile（不是 n_actual），从机制上完全不受影响。

注意「逐字节不变」的边界：逐字节不变的是 C++ 模板层的既有调用方（默认实参补上，展开结果一致）。ascendc 后端生成的 C++ 文本会变 —— GemmOpCodegen 无条件追加 args[5]，老调用现在发出的是 tl::ascend::gemm_v0<...>(a[..], b[..], c[..], ascend_l0a, ascend_l0b, true, 128);，尾部比改动前多一个值等于 N 的实参。本仓惯例是 dump codegen 做文本对拍，按「逐字节一致」去 diff 会被这个多出来的实参误导。

补 N 的只有 tilelang/language/ascend.py 这一个前端。同名的 PTO 前端 tilelang/language/pto.py 的 gemm_v0 签名里没有 n_actual，仍然只发 5 个实参给同一个 tl.ascend_gemm_v0，与 src/op/ascend.cc 的 set_num_inputs(6) 对不上；目前没有运行期后果，仅仅因为 codegen_ascend_pto.cc 只读 op->args[4]。一旦 PTO 侧发出的调用被非-PTO codegen（读 args[5]）处理就会越界。
  • 只在 transpose_B（QK）路径生效。非转置的 gemm_v0（PV 那条）传 n_actual 不会报错，但模板内部 mma 拿到的是 nTile、copy_l1_to_l0b 也照 N 搬 —— 想在非转置路径上收窄用的是另一套手段（PV 用 real_k / k_actual 限 K，N 保持编译期宽度）。
  • 必须是 16 的倍数（分形列粒度）且 0 < n_actual <= N。编译期常量会在前端被 assert 拦下；运行期 tir 表达式不会被检查，是调用方的责任 —— 所以 kernel.py 里到处是 (win + 15) // 16 * 16 / (ncols + 15) // 16 * 16 这种手工上取整。
  • [n_actual:N] 那段 cL0 列永远不会被写，停在 L0C 的初值或上一轮残留。下游必须自己 mask / 回避，绝不能当成 0 来用。
  • 配 unit_flag=0b11 用时，随后那个融合 fixpipe（T.copy(cL0 -> GM, unit_flag=0b11)）的 nSize 必须等于 mma 的 n_actual。不一致的话 fixpipe 会去等一批从没被任何 mma 标记 ready 的 cL0 列，cube 挂死（不是算错，是卡住）。
  • 单独改 mma 的 n 不够。L0B 的 nZ 分形从列数推 K-block stride，L1→L0B 的搬运必须同步带 real_n=<同一个值>（T.copy(..., transpose=True, real_k=..., real_n=...)），否则连算出来的那几列都是错的。gemm_v0 内部已经做了这一步；用解构的 T.mma 时要自己配对写。
  • C++ mma 的形参顺序是 (..., init, K, n_actual, unitFlag) —— n_actual 插在 unitFlag 前面。前端 T.mma 里 n_actual 和 unit_flag 只要有一个非 None，另一个就会被补成无害默认值（N / 0）一起以位置参数发出去。
  • 别把 n_actual 写成 T.if_then_else 之类会 lower 成语句的表达式：mma 的 codegen 是边写调用边打印实参，临时变量声明会插进半写好的调用中间，生成非法 C++。kernel.py 因此把 4 个 mma 逐条展开而不是写成 kk 循环。
  • PTO 后端（codegen_ascend_pto.cc）有自己的 gemm_v0 codegen，会忽略这个尾参；本仓的 Ascend codegen 永远走非-PTO 的 codegen_ascend.cc。

上游单测：testing/python/language/test_tilelang_ascend_language_gemm_v0_n_actual.py。真机（.npu()）跑 transpose_B 的 S = Q @ K^T，target 限定 ascendc，dtype 覆盖 float16 / bfloat16，参数化 4 组 (block_M, block_N, dim, n_actual)：(64,128,128,None) 是兼容基线、(64,128,128,64)、(64,128,128,112)（112 = 7×16，一个刁钻的非平凡部分宽）、(64,256,128,128)（N=256 时窗口 128）。断言只比对 s[:, :, :win]，因为 [win:N] 从没被 mma 写过 —— 这个「只比对窄带」的断言恰恰是 stride 一致性的探针：只要有人只改了 mma 的 n、忘了让 copy_l1_to_l0b 跟着传 n_actual，L0B 的 nZ 分形就会按 N 列去推 K-block stride，连被断言的前 win 列都会算错。把这个测试拿到改动前的仓上跑，四组（含 n_actual=None 的兼容基线，测试无条件以关键字传参）全部在构造 prim_func 时抛 TypeError: gemm_v0() got an unexpected keyword argument 'n_actual'，连带 fp16/bf16 共 8 个用例全崩 —— 关键字本身不存在，传 None 也不例外。默认路径的等价性由改动后 n_actual=None 这一组与旧仓上手工删掉该关键字的版本结果一致来体现。

5exp_experiment：带行 stride 的掩码 exp —— 在物理宽 512 的分数缓冲里，原地只 exp 窗口那 64 列

Feat 已合入 2026-07-17 PR #1399

一句话：原来 T.tile.exp 只有「从切片起点连续数 N 个元素」这一种语义，要在 [16, 512] 的分数缓冲上只 exp 每行前 64 列，就得逐行发 16 条 Exp；现在一条 T.tile.exp_experiment 覆盖所有行，行与行之间按 buffer 声明里的物理列数自动 stride。

5.1　这块机制是什么

在线 softmax 的分数缓冲在 kernel.py 里是 in_ub = T.alloc_ub([2, M_CHUNK, S2_BASE], accum_dtype)，物理宽度写死 S2_BASE = 512：宽 cmp tile 要用满这 512 列，而 cast、copy-out、p_l1 的行距也全按 512 算。但单个 tile 的真实有效列数 tw 常常只有滑窗那点宽度（tw <= BI = 128），剩下几百列是 padding。

昇腾的向量指令天生有 mask + repeat + repStride 三件套：一次 repeat 处理 64 个 fp32（= 8 个 32B block），repeat 之间按 repStride 个 block 往前跳。硬件本来就能表达「每行只算前 64 列、行间跳 512」，缺的只是把这套参数从 TileLang 前端发出去的通道。二元侧已经有 row_expand_sub_experiment / row_expand_div_experiment 这一族在这么干，一元的 exp 没有对应件 —— 这一节补的就是它。

5.2　原来写不出来的东西

内核作者想表达的是：在 [M_CHUNK, 512] 的分数缓冲上原地做 P = exp(S - rowmax)，但只碰前 tw_64 列（64 对齐后的有效窗口），既不整块算 512 列，也不把有效列压实到临时 tile。

改动前会这么写
# 写法 A：整块 exp —— 编得过、算得也对（padding 是 -inf，exp 后为 0），但纯浪费
T.tile.exp(in_ub[ps, :, :], in_ub[ps, :, :])

# 写法 B：直接切窄窗口 —— 也编得过，但答案是错的
for kc in range(BI // 64):
    if kc * 64 < tw_64:
        T.tile.exp(
            in_ub[ps, :, kc * 64 : kc * 64 + 64],
            in_ub[ps, :, kc * 64 : kc * 64 + 64],
        )

# 写法 C：压实到窄 tile 再 exp 再搬回 —— 多一块 UB + 两次搬运，in-place 拆成三步
T.copy(in_ub[ps, :, 0:tw_64], narrow_ub)
T.tile.exp(narrow_ub, narrow_ub)
T.copy(narrow_ub, in_ub[ps, :, 0:tw_64])

# 写法 D：逐行发 —— 语义正确，但一个 64 列 chunk 就是 M_CHUNK 条 Exp
for i in range(M_CHUNK):
    T.tile.exp(
        in_ub[ps, i, kc * 64 : kc * 64 + 64],
        in_ub[ps, i, kc * 64 : kc * 64 + 64],
    )
会发生什么写法 B 是静默算错：不报错、不挂、编译产物看起来完全正常。前端 unary_op（tilelang/language/ascend_tile.py:944-967）把 BufferRegion 的 extent 拍平成 size_0 = math.prod(dst_extent)，in_ub[ps, :, kc*64 : kc*64+64] 的 extent 是 [1, 16, 64]，size_0 = 1024；codegen 走 UnaryVecOpCodegen(op, "AscendC::Exp")，发出 AscendC::Exp(dst, src, 1024) —— 这是连续 count 语义，从切片起点一路往后数 1024 个 fp32。in_ub 的物理行距是 512，1024 / 512 恰好是整整两行：kc = 0 那次把第 0、1 行整行（窗口连同 padding）exp 掉，第 2~15 行一个元素都没碰。tw_64 = 128 要发第二块时（起点偏移 64、同样 1024 个元素），第 0、1 行已经 exp 过的部分被再 exp 一次成 exp(exp(x))，还溢进第 2 行的前 64 列。净效果是第 2 行起的「P」停在 score - rowmax 这个负数上，紧接着的 T.reduce_sum 把负数累进 denom —— denom 和 P 同时错，vs golden 的 mismatch 既不像 race 也不像 QK 的锅。覆盖到的行数随 dtype / chunk 变（= 切片 extent / 物理列数），fp16 的 128 列 chunk 是前 4 行。

为什么在内核侧绕不过去：分数缓冲的物理宽度不能改窄 —— 同一块 in_ub 还要给宽 cmp 路径用（tw 最大就是 512），cast、copy-out、p_l1 的行距也全按 512 算，声明成 [M, 64] 整条链都得跟着改。于是内核侧能写对的只剩三条：整块算 512 列（窄窗时白算 8 倍的超越函数）；压实到临时 tile（多一块 UB、两次搬运，还把原地操作拆成三步、要补跨 pipe 握手，V 流水更长）；或者写法 D 逐行发 —— 每行一条 AscendC::Exp(dst, src, 64)，语义完全正确，代价是一个 64 列 chunk 要发 M_CHUNK = 16 条 Exp，再乘上 NMC 个 m-chunk、每个 tile、每个 task，而且 Python 层的行循环会被展开，生成的 C++ 也跟着膨胀 16 倍。exp_experiment 把同一件事压成 1 条指令：16 个 repeat，跨行的偏移交给硬件的 repStride。

5.3　编译器改了什么

  • tilelang/language/ascend_tile.py：新增前端 exp_experiment(dst, src)，用 _handle_buffer_region_2d 把 ND 切片折成 [rows, cols]，校验 dst/src 形状相等、dtype 是 fp16/fp32、最后一维正好是 64（fp32）/ 128（fp16），发 tl.ascend_exp_experiment intrin。
  • src/op/ascend.cc + src/op/ascend.h：注册 ascend_exp_experiment builtin（set_num_inputs(-1)、kOpaque），与 row_expand_*_experiment 一族并列。
  • src/target/codegen_ascend.cc：新增 ExpExperimentCodegen —— 从 buffer_shapes_ 里读出 dst buffer 声明的最后一维（物理列数 cols）算 rep_stride = cols / elems_per_block，从 access_ptr 的 extent 算 repeat_time = extent / (8 * elems_per_block)（对 [rows, 64] 切片正好等于 rows），并对 extent、cols 的整除性下 ICHECK。
  • src/target/codegen_ascend.cc（dispatch）：VisitExpr_(CallNode*) 里加一条 else if (op->op.same_as(tl::ascend_exp_experiment())) ExpExperimentCodegen(op);。
  • src/tl_templates/ascend/common.h：新增 exp_mask<T>(dst, src, mask0, mask1, repeatTime, dstBlkStride, srcBlkStride, dstRepStride, srcRepStride)，内部拼 AscendC::UnaryRepeatParams 后调 AscendC::Exp<T, false> —— 就是 sub_mask / div_mask 的一元镜像。
  • src/transform/common/operation_config.h：登记 {"tl.ascend_exp_experiment", {{{1, "write"}, {2, "read"}}, "PIPE_V"}}，让同步插入 pass 知道它跑在 PIPE_V、arg1 写 arg2 读。
  • src/target/codegen_ascend.h：声明 void ExpExperimentCodegen(const CallNode *op);。

生成的 C++：tl::ascend::exp_mask<T>(dst, src, 0xFFFFFFFFFFFFFFFF, mask1, repeat_time, 1, 1, rep_stride, rep_stride);。其中 fp32 的 mask1 = 0、fp16 的 mask1 = 0xFFFFFFFFFFFFFFFF；elems_per_block = 256 / bits（fp32 8、fp16 16）；in_ub 的 512 列 fp32 → rep_stride = 64；[16, 64] 的切片 → repeat_time = 16，一行一个 repeat。前端只有两个位置参数，count / mask / stride 全部由 codegen 从 buffer 声明推。

5.4　现在可以这么写

最小例子
M, N = 16, 512   # 分数缓冲的物理宽度写死 512
CH = 64          # fp32 一次调用正好 64 列(fp16 是 128)
VALID = 40       # 本 tile 真实有效列数,故意不是 64 的整数倍

@T.prim_func
def main(A: T.Tensor((M, N), "float"), C: T.Tensor((M, N), "float")):
    with T.Kernel(1, is_npu=True) as (cid, vid):
        a_ub = T.alloc_ub([M, N], "float")
        row_max = T.alloc_ub([M, 1], "float")
        brcb = T.alloc_ub([M, 8], "float")
        # 满宽铺 -inf,再只搬有效的 40 列:[40:64] 这段对齐 slop 留 -inf
        T.tile.fill(a_ub, -T.infinity("float"))
        T.copy(A[:, 0:VALID], a_ub[:, 0:VALID])
        # rowmax 走满宽:padding 是 -inf,不影响 max
        T.reduce_max(a_ub, row_max, dim=-1)
        T.tile.brcb_experiment(brcb, row_max, M // 8, 1, 8)
        # 只有这两行是窄的:原地只碰 [0:64],行间按声明的 512 列 stride
        T.tile.row_expand_sub_experiment(a_ub[:, 0:CH], a_ub[:, 0:CH], brcb)
        T.tile.exp_experiment(a_ub[:, 0:CH], a_ub[:, 0:CH])
        T.copy(a_ub, C)   # [64:512] 仍是 fill 写下的 -inf

一段手拼的窄窗口 softmax：缓冲物理宽 512，真实有效只有 40 列。关键在最后那行 T.tile.exp_experiment —— 它对 16 行各发一个 repeat（每 repeat 64 个 fp32），行间按 buffer 声明的 512 列 stride 跳，一次调用就把 16 行的窗口全 exp 了，[64:512] 那 448 列一个字节不动。同一行换成 T.tile.exp(a_ub[:, 0:CH], a_ub[:, 0:CH]) 编译照样过，发出的却是 AscendC::Exp(dst, src, 1024) 连续 count，只覆盖前 2 行、其余 14 行原封不动。VALID = 40 是刻意的：exp_experiment 的 mask 传的是全 1（窄化靠 repeat 次数 × repStride，不是列掩码），所以 [40:64] 这段对齐 slop 一定会被 exp —— 开头「满宽 fill(-inf) → 只搬有效列」的顺序就是为了让 slop 保持 -inf，exp 后是 0，无害。反过来写成先 copy 满宽再 fill，fill 就成了死语句。

T.reduce_max 这里走的是满宽。窄列切片上的行归约（T.reduce_max(a_ub[:, 0:CH], ...) 按物理行宽 512 跨行）是另一件事，要 reduce codegen 认识物理行宽，那是 #1422 里的改动；只有 #1399 的树上那么写会拿第 0 行的若干块当成 16 行的 rowmax，静默算错。exp_experiment 本身不依赖它。上游单测里没有 fill / reduce，只有 T.copy + 分块 exp_experiment + T.copy，pass_configs 打开了 TL_ASCEND_AUTO_SYNC，fill(PIPE_V) → copy(MTE2) 之间的握手由 pass 插；手写同步的 kernel 要自己补 T.set_flag("v", "mte2", ev) / T.wait_flag(...)（kernel.py:1168-1169 那对）。

5.6　kernel.py 里的真实用法（二）：窄窗口的前置约定

kernel.py:1140-1149 · CFA · 每个 tile 的宽度对齐与窄/宽分路
tw_a = (tw + 15) // 16 * 16
# 64-align for strided masked ops (每 chunk = 64 fp32/
# repeat 上限)。[tw:tw_64] 留 fill(-inf) -> reduce 忽略、
# exp->0;tw_a<=tw_64 故 copy-out [0:tw_a] 全在算过的范围内。
tw_64 = (tw + 63) // 64 * 64
# softmax reduce/sub/exp 按**实际宽度**分路(不是 ori/cmp):
# 窄窗(tw<=BI:ori 窗 + cfa 的窄 cmp)走 strided-in-place 省
# padding;宽窗(tw>BI,只 scfa 的宽 cmp,最多 512)走满宽——
# 无 padding 可省时一条宽 op 胜过 tw_64/64 个 strided chunk。
is_narrow = tw <= BI

这十行是整段 strided softmax 的「合同」，直接决定 exp_experiment 用得对不对。tw 是本 tile 的真实有效列数；tw_a 是 16 对齐（cube 侧 PV 读 P 的宽度）；tw_64 是 64 对齐 —— 因为 exp_experiment / row_expand_sub_experiment 每次调用正好吃 64 列（fp32 一次 repeat 的上限），窗口必须先补到 64 的整数倍才能整块喂。mask 传的是全 1，窄化完全来自 repeat 次数 × repStride，所以 [tw, tw_64) 这段对齐 slop 一定会被算进去；kernel.py 的办法是整块先 T.tile.fill(in_ub[ps, :, :], -T.infinity(accum_dtype))、再只 copy 有效的 [0:tw] 列（kernel.py:1164-1181），slop 保持 -inf —— reduce_max 忽略它、exp 把它变成 0、reduce_sum 加 0，全链无害。tw_a <= tw_64 保证 copy-out 的那 tw_a 列全落在算过的范围内。最后一行 is_narrow = tw <= BI 是分路开关：按实际宽度而不是按 ori/cmp 语义分，窄的走 strided（省 padding），宽的走 T.tile.broadcast + T.tile.sub + T.tile.exp 满宽（kernel.py:1307-1318，没 padding 可省时一条宽 op 更划算）。

同文件 1267-1269 还有一句「ori: strided …；cmp: wide broadcast+sub+exp(满 512)」，与第 8 行那句「宽窗(tw>BI,只 scfa 的宽 cmp)」看起来矛盾 —— 两句都是简写，真正的分路条件只有 tw <= BI 这一个运行时比较，跟 ori / cmp 的语义无关，同一段 cmp 走哪条完全由 cmp_ratio 和 topk 决定。本仓的两组配置正好各占一边：CFA 用 cmp_ratio = 128（test_sparse_attn_sharedkv.py:191、214），8K 下 act_cmp = 8192 // 128 = 64、cmp_tiles = 1，那唯一一个 cmp tile 宽 64 ≤ BI，走的是窄路；SCFA 用 cmp_ratio = 4 加 topk_cmp = K = 512 截断（test_sparse_attn_sharedkv.py:114、122），cmp tile 满 512 宽，才是宽路的实际来源。所以 exp_experiment 这条窄路上跑的不只是 ori 滑窗，还有 CFA 的整段 cmp。

5.7　兼容性与坑

兼容性：纯新增，老代码逐字节不变。exp_experiment 全套都是新东西：新 builtin ascend_exp_experiment、新前端函数、新 codegen 分支 ExpExperimentCodegen、新模板函数 exp_mask。老路径 tl.ascend_exp → UnaryVecOpCodegen(op, "AscendC::Exp") → AscendC::Exp(d, s, n) 一行没动，内核里原来的 T.tile.exp 生成的 C++ 与合入前完全相同。operation_config.h 的改动是往表里加一行新 key，既有 key 的 pipe / 读写标注没碰。也没有任何 pass 会把 T.tile.exp 自动改写成 exp_experiment —— 要用必须显式改调用点，不会有「升级编译器后行为悄悄变了」这种事。

  • 最后一维必须正好是 64（fp32）/ 128（fp16），不是「不超过」。96 列的窗口不能一次调完，要么向上对齐到 128 分两块，要么自己处理尾巴。kernel.py 的做法是先 tw_64 = (tw + 63) // 64 * 64 再按 64 分块。
  • mask 传的是全 1（fp32：mask0 全 1、mask1 = 0；fp16：两个都全 1），每次 repeat 的 64 / 128 个 lane 全算。窄化来自 repeat 次数 × repStride，不是列掩码。[tw, tw_64) 这段 slop 必然被 exp，调用方要自己保证那里的值无害；若 slop 里是垃圾数，exp 会炸成 inf/NaN 再被 reduce_sum 吃进分母。
  • dst 与 src 的声明物理列数必须相同。rep_stride 只从 dst 的 buffer 声明推（src/target/codegen_ascend.cc:2321-2331），然后同一个值被打到 dstRepStride 和 srcRepStride（同文件 2354-2357）；前端只比切片形状，不看物理宽度。所以 T.tile.exp_experiment(narrow[:, 0:64], wide[ps, :, 0:64])（narrow 声明 [16, 64]、wide 声明 [2, 16, 512]）前端全过、codegen 无 ICHECK，src 却会按 8 个 block 而不是 64 个 block 跨行读 —— 静默读错行。kernel.py 里 dst 就是 src（原地），所以安全。
  • 行 stride 读的是静态声明，buffer 必须静态形状；extent 非 IntImm 直接 ICHECK(false) << "ExpExperimentCodegen: dynamic extent not supported"。反过来说，同一段窗口逻辑换一个物理宽度不同的 buffer，stride 会自动跟着变，调用点不用改。
  • exp_mask 的 repeatTime 与 dstRepStride / srcRepStride 都是 uint8_t：行数 ≤ 255，物理列数 ≤ 255 × (256/bits)（fp32 ≤ 2040、fp16 ≤ 4080）。codegen 不检查这两条，超了是静默截断。kernel.py 的 M_CHUNK = 16、512 列（rep_stride = 64）远在范围内。
  • 整除性是硬约束：物理列数必须被 elems_per_block（fp32 8 / fp16 16）整除，切片 extent 必须是 8 * elems_per_block（fp32 64 / fp16 128）的整数倍，否则 codegen 期 ICHECK 直接挂，错误里会打出 extent 和 cols 的实际值。
  • 窗口宽了就别用：每 64 列一次调用，tw 接近 512 时是 8 次 strided 调用 vs 1 次宽 exp，反而亏。kernel.py 用 is_narrow = tw <= BI（BI = 128）分路，宽 cmp tile 走满宽写法。
  • 只有 ascendc target 有，PTO 后端没有对应件（上游单测也不做 target 参数化）。按本仓惯例 SWA/CFA/SCFA 的 codegen 永远走非-PTO，所以不影响，但别指望它在 PTO 路径上能编出来。
  • 依赖跟踪靠 operation_config.h 里那条 {{{1, "write"}, {2, "read"}}, "PIPE_V"}。它和串在一起的 row_expand_sub_experiment、reduce_max / reduce_sum 同在 V 流水，V 内顺序由 pipe 语义保证；但跟 MTE2（copy-in）/ MTE3（copy-out）之间的 T.set_flag / T.wait_flag 握手仍然要自己写 —— 加这个原语不会替调用方补跨 pipe 的围栏。

上游单测：testing/python/language/test_tilelang_ascend_language_elementwise.py::test_exp_experiment，参数化 ("float", 64 / 128 / 256) 与 ("float16", 128 / 256)。固定 M, N = 16, 512，kernel 里按 for k in range(col // chunk) 循环调用 exp_experiment，参考值是 ref = a.clone(); ref[:, :col] = torch.exp(a[:, :col]) —— 也就是尾部 [col:N] 必须逐字节不变。这一条把两种旧写法都判死：整块 T.tile.exp 会把尾部一起 exp（尾部对不上 ref）；切片写法走连续 count，fp32 col=64 时把前 2 行整行 exp（尾部一起动）、第 2~15 行的窗口一个都没算，fp16 chunk=128 时是前 4 行，头尾同时对不上 ref。col = 256 那两档（fp32 要 4 次调用、fp16 要 2 次）专门覆盖调用方自己分块循环这个用法，顺带验多次调用的偏移推导。测试只跑 ascendc target。

6让内核自己驱动 L1→L0→mma→fixpipe：real_k / real_n / k_actual / n_actual / unit_flag 五件套

Feat + Bugfix 待合入 PR #1422

一句话：原来 L1→L0→mma 这三步只能整个交给 T.gemm_v0 一条辅助函数，内核自己发的 T.mma 里 K 和 N 都只能是编译期值，调用方另发的那条 fixpipe 跟 mma 之间只有软件握手；现在内核可以自己逐块载 L0、自己发累加型 mma，K 和 N 都可以是运行期值，最后一块到齐时用 unitFlag 与那条 fixpipe 做硬件配对。

6.1　这块机制是什么

昇腾 cube 的一次矩阵乘走四步：L1→L0A/L0B 按分形载入操作数、Mmad 计算、结果落在 L0C 累加器、再由 fixpipe 把 L0C 搬到 GM。TileLang 把前三步封在 T.gemm_v0 里（src/tl_templates/ascend/common.h:1103-1252）。第四步不在里面：模板签名上 C 那一行的注释原文就是 // this must be located in l0c，函数体只做 L1→L0 载入加 mma，结果留在调用方的 L0C，fixpipe 一直是调用方自己另发的一条 T.copy(L0C→GM)。

硬件另有一个 unitFlag 字段，可以把 mma 和它后面那条 fixpipe 串成硬件流水：0b10 表示结果留在 L0C，0b11 表示释放给配对的 fixpipe。它在 mma 模板里一直是被注释掉的死钩子。

当内核需要自己维护 KV 环形缓冲、或者把 contraction 拆成不同时刻才到齐的若干块时，L1→L0→mma 这三步必须由内核自己走完，并且要能和那条 fixpipe 配上对——否则每一块都得回到模板里跑一遍完整序列。

6.2　原来写不出来的东西

以 SWA 的 PV 阶段为例：内核自己维护一个 3 槽 KV 环，V 的每个 D-tile 是逐页 gather 进来的，到齐一块就想立刻发一条 mma 把它算掉，算完直接用配对的 fixpipe 把 L0C 冲进 workspace。而且 contraction 长度 winm（真实窗口行数）是运行期值，不是 L0 缓冲的编译期宽度 BI=128。

改动前会这么写
# 意图（SWA PV）：P[G, winm] @ V[winm, 128]，winm 是运行期窗口行数

# 尝试 1：整块载入 L0 再发 mma —— 当时唯一写得出来的形式
T.copy(p_l1[:, :], p_l0a[pp, :, :])                 # 只能按 p_l0a 的 [G, BI] 全宽载入
T.copy(kv_ring[slot, :, 0:PV_NW], v_l0b[pp, :, :])  # 同上，全宽

会发生什么尝试 1 编译过、跑得动、答案静默错。K 恒等于 A 最后一维，也就是 L0 缓冲的编译期宽度 BI=128：窗口只有 winm < 128 行时，P 的 [:, winm:128] 列和 V 的 [winm:128, :] 行——两块都是上一轮留下的陈旧内容——一并被 contract 进去。输出的每一行都被污染，没有任何告警。
会发生什么尝试 2 走不通。npu_gemm 的 retrieve_ptr 用 buffer.access_ptr(offset=..., extent=...) 取指针，符号 extent 拿不到具体值（customize.py 的 docstring 原话：Passing a symbolic slice instead (a_l0[pp, :, 0:k]) is not an option, since access_ptr would need a concrete extent）。就算这一层放行，tl.ascend_mma 当时在 src/op/ascend.cc 里注册成 .set_num_inputs(6)，前端根本没有第 7、第 8 个位置可以放 n_actual / unit_flag。
会发生什么尝试 3 能算对，但整条 L1→L0→mma 被关进模板，内核插不进自己的环形缓冲 flag；而且每次调用都带一层 pipe drain（见下）。

为什么在内核侧绕不过去：退回 T.gemm_v0 是唯一能算对的写法，但它有两个内核侧绕不过去的代价。

第一，模板每次调用首尾各有两对「SetFlag(e) 紧跟 WaitFlag(e)」的同一 event 自配对围栏：开头是 MTE2_MTE1 和 FIX_M（common.h:1187-1190），结尾是 MTE1_MTE2 和 M_FIX（common.h:1248-1251），四对都用 L0AB_EVENT = 0。自配对不是在给某个 buffer 排序，而是 drain 整条 pipe：每次 gemm 都强迫 MTE2 全部做完 MTE1 才能开始，并把下一页 KV 的搬运卡到当前 L1→L0 结束之后。这恰好就是内核那个 3 槽 KV 环存在的意义，等于把 overlap 全废掉。

第二，contraction 长度 winm 是运行期值，模板的 K 只能来自编译期模板参数或 A 的最后一维，切窗口这件事在内核侧无法表达。

6.4　现在可以这么写

最小例子（对应上游单测 testing/python/language/test_tilelang_ascend_language_kernel_driven_mma.py，注释译为中文）
M, N, K = 64, 128, 256
KL0 = 128  # kL0 tile 宽度：K 拆成 2 块

@T.prim_func
def main(A: T.Tensor([M, K], "float16"),
         B: T.Tensor([N, K], "float16"),   # K^T 布局，算的是 A @ B^T
         nact: T.Tensor([1], "int32"),     # 运行期输出列数
         C: T.Tensor([M, N], "float")):
    with T.Kernel(1, is_npu=True) as (cid, _):
        a_l1 = T.alloc_L1([M, K], "float16")
        b_l1 = T.alloc_L1([N, K], "float16")
        a_l0 = T.alloc_L0A([2, M, KL0], "float16")   # 两个 ping-pong 槽
        b_l0 = T.alloc_L0B([2, N, KL0], "float16")
        c_l0 = T.alloc_L0C([1, M, N], "float")
        with T.Scope("C"):
            n_act = nact[0]
            T.barrier_all()
            T.copy(A, a_l1)
            T.copy(B, b_l1)
            T.barrier_all()
            # K 块 0：初始化 L0C 槽，结果留在 L0C（0b10）
            T.copy(a_l1[:, 0:KL0], a_l0[0, :, :])
            T.copy(b_l1[:, 0:KL0], b_l0[0, :, :],
                   transpose=True, real_k=KL0, real_n=n_act)
            T.barrier_all()
            T.mma(a_l0[0, :, :], b_l0[0, :, :], c_l0[0, :, :],
                  init=True, k_actual=KL0, n_actual=n_act, unit_flag=0b10)
            T.barrier_all()
            # K 块 1：累加进同一个槽；最后一块释放给配对的 fixpipe（0b11）
            T.copy(a_l1[:, KL0 : 2 * KL0], a_l0[1, :, :])
            T.copy(b_l1[:, KL0 : 2 * KL0], b_l0[1, :, :],
                   transpose=True, real_k=KL0, real_n=n_act)
            T.barrier_all()
            T.mma(a_l0[1, :, :], b_l0[1, :, :], c_l0[0, :, :],
                  init=False, k_actual=KL0, n_actual=n_act, unit_flag=0b11)
            T.barrier_all()
            # 配对的那条 fixpipe：列数必须 == mma 的 n_actual，
            # 且这里刻意不加同步——被顶掉的正是软件 M_FIX 握手
            T.copy(c_l0[0, :, :], C[:, 0:n_act], unit_flag=0b11)
            T.barrier_all()

真正「删掉就错」的是三处：B 侧两条 real_n=n_act 让 L0B 按 mma 真正会读的形状载入分形（少了它，nZ 分形的 K-block stride 会按缓冲全宽算，第二块 K 就寻址到错误的 K-block）；两条 mma 的 init=True/False 把 K 累加进同一个 L0C 槽；unit_flag 从 0b10 变成 0b11 标出「最后一块到齐了」，最后那条 T.copy(..., unit_flag=0b11) 就是与它配对的 fixpipe。n_act 从一个 int32 输入张量取，是运行期值——单测参数化跑 128 / 96 / 32 三档，编译期常量蒙不过去。两块 K 也是必须的：只跑一块的话它的 K-block 偏移恰好是 0，real_n 写错也照样算对。

这个例子里的 real_k / k_actual 是恒等写法。A 侧 K 本来就是 a_l0 的末维 KL0；B 侧 realK 覆写的是 dst->shape[-2]，而在这个 A @ B^T 的布局里那一维就是 N=128=KL0。删掉这两个参数，发出的 TIR 和 C++ 逐字节不变。运行期 K 的真实用例见 6.5（三）。另外这个布局有个隐含约束：npu_gemm 会断言 K == K_B，而 A 侧 K 取 A_shape[-1]、B 侧取 B_shape[-2]，所以 b_l0 声明成 [2, N, KL0] 时必须 N == KL0，非方形要改 L0B 的声明顺序。
例子里每个流水边界的 T.barrier_all() 都是显式写出来的。L1→L0 的 copy 在 PIPE_MTE1 上、mma 在 PIPE_M 上（src/transform/common/operation_config.h），两条流水之间没有同步就是 L0A/L0B 的 RAW 竞态。自动同步 pass tl.ascend_auto_sync 默认关闭（src/transform/ascend_sync_insert.cc 不开就直接 return），kernel.py 走的正是默认关闭这条路，同步全由内核自己发。要保住 overlap 就把全屏障换成定向 flag（T.set_flag("mte1", "m", EV) / T.wait_flag("mte1", "m", EV)），也就是 6.5–6.7 里 kernel.py 的写法。唯独最后那条 mma 与配对 fixpipe 之间不要加同步——那正是 0b11 要顶掉的握手。

四块 kL0 tile 里的最后一块。前三块是 (T,0b10)(F,0b10)(F,0b10)，这块是 (F,0b11)：init=False 继续往同一个 cL0 槽累加，unit_flag=0b11 告诉硬件「这个槽算完了，放给配对的 fixpipe」。紧接着那条 T.copy(cL0 → workspace_s, unit_flag=0b11) 就是被配对的 fixpipe，对应 Ascend C 参考实现里两个 D-half 之后的那一条 Fixpipe。

另外这四块是逐条展开写的，而不是 kk 循环——因为最后一块的 unit_flag 不一样，而 mma 的实参里放运行期条件会把 codegen 写坏（MmaCodegen 边写调用边把实参 PrintExpr 进同一个 stream，if_then_else 的临时变量声明会落在半截调用中间）。

6.8　兼容性与坑

兼容性：这一节所有新能力都是「不给参数就等于没这回事」。前端 T.copy 的 unit_flag / real_k / real_n 和 T.mma 的 n_actual / unit_flag / k_actual 全部默认 None，只有真的被设置时 copy_op.py / customize.py 才往 intrin 的实参列表尾部追加，所以旧调用发出的 TIR 逐字节不变（T.mma 仍是 6 参）。C++ 侧 AscendCopy 把 unitFlag / realK / realN 一律初始化成 Integer(0)：realK==0 时 l0_dst_split 仍旧 push dst->shape 的最后两维，unitFlag==0 就是一条独立 fixpipe。模板层 mma 的 n_actual 默认 N、unitFlag 默认 0，copy_l0c_to_gm 的 unitFlag 默认 0。ascend_mma 的 set_num_inputs 从 6 改成 -1 只是把定长放宽成变参，不拒绝任何旧调用。唯一一处无条件改变行为的是 Bugfix 那条 mmadParams.cmatrixSource = false;——但它原本是未初始化字段，等于把一处未定义行为钉死成所有累加型调用方本来就想要的语义（「C 从 L0C 取源」）。

  • L0 分形的 K 必须和 mma 的 K 严格相等（real_k == k_actual）。不等不会报错，会静默算错：第一个 M-block（第 0..15 行）因为偏移是 0 恰好正确，16 行往后全错。调试时很容易误判成竞态。
  • real_k 和 real_n 的失败签名不一样，别混：real_k 不匹配对应「寻址到错误的 M-block」，real_n 不匹配对应「寻址到错误的 K-block」（copy_op.py 的 docstring 里就是这么分的）。
  • 配对的那条 fixpipe 的列数必须等于 mma 的 n_actual。否则带 unitFlag 的 fixpipe 会去等一批没有任何 mma 标记就绪的 L0C 列，表现为挂死而不是错数。
  • real_n 只对 wmma.matrix_b 生效——matrix_a 是 [M, K]，没有 N 这根轴，给了也会被忽略。反过来 real_k 对 A / B 都生效，但覆写的维不同（matrix_a 覆写末维，matrix_b 覆写首维）。
  • unit_flag 不同的那几条 mma 必须逐条展开写，不能写成 kk 循环加 if_then_else。MmaCodegen 是边写调用边把实参 PrintExpr 进同一个 stream 的，if_then_else 产生的临时变量声明会落在半截调用中间，直接写坏 codegen。
  • 前端的尾参是位置参数：real_n 非空但 real_k 为空时，realK 会被物化成 0（=「从 dst 取」）；unit_flag 非空但 n_actual 为空时，n_actual 会被物化成模板 N。想只改外层那个，也得清楚内层被填了什么默认值。
  • 解构掉 gemm 模板换来的是 pipe overlap，但也把 L0AB 的 flag 生命周期整个交给了内核：("mte1", "m") 那对（载入完成→允许 mma）、("m", "mte1") 那对（mma 用完→允许下一轮覆写这个 L0AB 槽），以及参考实现按运行期 n 门控的那条 PipeBarrier<PIPE_M>，都得自己复现，漏一条就是竞态。
改这块编译器代码的人要注意两处工程约束。其一，unitFlag 在 AscendCopy::Lower 的 new_args 里必须追加到末尾，因为这条参数列表是与 PTO codegen 共用的、PTO 按固定下标读其中的 pad enum 和 relu flag。其二，CopyCodegen 里按 kCopyOpExtraArgs 查表读出的个数被 clamp 到 args.size() - 3：查表值是「一条调用最多能带多少参数」，而任何自建参数列表的生产者（例如 ascend_workspace_reduction 把 copy_l0c_to_ub 就地改写成 copy_l0c_to_gm）都可能只带它关心的那几个。这条 clamp 是防御性上限，不是被现有某条改写路径逼出来的——那条路径恰好凑齐 6 个实参，正好等于 extra_args = 3 需要的数量。
上游单测：testing/python/language/test_tilelang_ascend_language_kernel_driven_mma.py，即 6.4 那段代码。它设计成「旧代码必挂」有三处：① n_act 通过 int32 输入张量传进来，是运行期值（参数化跑 128 / 96 / 32 三档），real_n / n_actual 必须走运行期路径；② 必须用两块 K——单独一块的 K-block 偏移恰好是 0，就算 real_n 写错也照样算对，只有第二块才会暴露寻址到错误 K-block；③ 第二条 mma 是 init=False 的累加路径，正是硬件会去读 cmatrixSource 的那条路径，所以只要 unitFlag 接通而 cmatrixSource 没修，这个测试直接把 cube 挂死而不是报数值不符。

7多维 buffer 的标量下标要跨维展平 —— block_table[b, lg] 不再别名到第 0 行

Bugfix 待合入 PR #1422

一句话：原来对多维 buffer 做标量读写只发最内层下标、前导维被整个丢掉（table[b, i] 恒等于 table[0, i]，静默算错且 B=1 时完全正常），现在按 buffer 形状跨全维展平成一个线性偏移，ori_block_table[b, pa_lg] 这类「按批次取物理页号」才真正取到第 b 行。

PR #1422 本身的标题是 [Feat] Ascend: let the kernel drive the L1->L0->mma->fixpipe sequence（fork 分支 feat/ascend-kernel-driven-cube @ 3a546966，改 11 个文件），主体是 real_k / real_n / n_actual / k_actual / unit_flag 这几个可选运行期参数。本节只讲这条 feat 顺带修掉的一个缺陷 —— 标量索引展平，落在 codegen_ascend.cc 的两个 hunk 上。

7.1　这块机制是什么

昇腾侧生成出来的 Ascend C 里，UB / GM 上的 buffer 分别是 LocalTensor<T> 和 GlobalTensor<T>，这两个类型只提供线性下标的标量接口 GetValue(i) 和 SetValue(i, v) —— 没有多维下标的重载。所以 TileLang 里写 buf[i, j] 这种「取一个标量进标量寄存器」的访问，codegen 必须自己把多维下标折算成一个线性偏移再发下去。

这跟整块搬运是两条完全不同的路：T.copy 走 CopyCodegen，把 dstM / dstN / stride 当参数显式发给搬运指令，多维信息不经过下标；只有标量访问依赖 codegen 自己做展平。TVM 侧现成的工具是 Buffer::OffsetOf(indices)，它按 buffer 的 shape / strides 把一组下标折成物理偏移，对一维 buffer 是恒等映射。

7.2　原来写不出来的东西

分页 KV（PagedAttention）的页表是二维的 [batch, num_logical_blocks] —— 第 b 行是第 b 个 batch 的逻辑块号到物理块号的映射。内核里要按当前 batch 取物理页号：先算逻辑块号 pa_lg = pa_cur // ori_block_size，再 pa_ph = ori_block_table[b, pa_lg]，然后拿 pa_ph 去 ori_kv[pa_ph, ...] 做真正的 GM→L1 搬运。这是分页 KV 最基本的一步，当前 main（692edac）上三个 kernel 合计 9 处页表查表，外加 SCFA 一处三维的 cmp_indices 标量读，共 10 处多维标量访问。

改动前会这么写
# 页表按 vLLM/FlashAttention 的约定就是二维 [B, max_blocks]
ori_block_table: T.Tensor([batch, ori_table_len], idx_dtype)

...
    b = T.cast(pid // max_seq, "int32")       # 当前 batch
    pa_lg = pa_cur // ori_block_size          # 逻辑块号
    pa_ph = ori_block_table[b, pa_lg]         # 二维标量读
    T.copy(ori_kv[pa_ph, ...], kv_ring[...])  # 用物理块号搬运

# 改动前 codegen（src/target/codegen_ascend.cc）:
#   os << var_name << ".GetValue(" << PrintExpr(op->indices.back()) << ")";
# 发出来的 Ascend C 是：
#   ori_block_table.GetValue(pa_lg);          // b 不见了
# 应该是：
#   ori_block_table.GetValue(b * ori_table_len + pa_lg);
会发生什么编译不报错、运行不挂，纯静默错。op->indices.back() 只打印最内层下标，前导的 b 被整条丢掉，于是 ori_block_table[b, pa_lg] 发成 ori_block_table.GetValue(pa_lg) —— 不管 b 是几，读回来的都是页表第 0 行的物理页号。B=1 时 b 恒为 0，block_table[0, lg] 与 block_table[lg] 读的就是同一处内存，完全正常；B=2 时 batch 1 的每一次 KV 搬运都去搬 batch 0 的物理页，Output / LSE 整批对不上 golden（不是个别 token 偏差，是这个 batch 全错）。BufferStore 侧同理：buf[b, i] = v 只把 v 写进 buf.SetValue(i, v)，B 行互相覆盖到第 0 行。

为什么在内核侧绕不过去：第一层是「根本不知道要绕」—— 错得完全静默，而算子仓两套测试（test_sparse_attn_sharedkv.py / test_sparse_attn_sharedkv_fast.py）当时所有场景都是 B=1，b 恒为 0，写对写错逐字节等价，全绿。第二层是「知道了也别扭」：TileLang 的 layout 推断硬性要求下标个数跟 buffer 维度一致（src/transform/layout_inference.cc:97 的 ICHECK(indices.size() == buffer->shape.size())），不能在内核里直接写 ori_block_table[b * ori_table_len + pa_lg]；要手工展平只能把内核签名从 T.Tensor([batch, ori_table_len]) 改成一维，再让 api.py 把外部传进来的 [B, max_blocks] 页表 .view(-1) —— 而 [B, max_blocks] 是 vLLM/FA 那套 block_table 的公开约定，api.py 的入参就是它，改成一维等于把展平责任推给调用方；9 处调用点都要手写 stride 乘法，SCFA 的 cmp_indices 还是三维 [total_tokens, N2, topk_cmp]（kernel.py:1811），手写展平后基本不可读。UB 上的临时 buffer 更没有 host 侧 view 可用。

这条 bug 一直可达，只是没人看得见。首个 SWA kernel a4f2f72 的 kernel.py:234 就是 page = ori_block_table[b, pos // ori_block_size]；45c6a32（SCFA 场景 3）引入了 vector 侧的 cmp_indices / cmp_block_table 两处并一直存活到今天。5ecdbc4（把页表查表从 copy_pa 的 C++ helper 挪到内核前端）不是「让 bug 变得可达」，而是把站点数从 5 处放大到 10 处。真正让它暴露的是 b2f06ee —— 补上 B=2 场景。

7.3　编译器改了什么

  • src/target/codegen_ascend.cc:481-482：VisitExpr_(const BufferLoadNode *) 里 .GetValue() 的下标从 PrintExpr(op->indices.back()) 改成 PrintExpr(op->buffer.OffsetOf(op->indices).back())，按 buffer 的 shape / strides 跨全维折成一个线性偏移。
  • src/target/codegen_ascend.cc:495：VisitStmt_(const BufferStoreNode *) 里 .SetValue(index, value) 的 index 做同样替换，读写两侧对齐（否则会出现读对了写还错的半修）。
  • 两处都留了 WHY 注释，点明「OffsetOf 对一维 buffer 是恒等，所以既有一维访问逐字节不变」，避免以后被当成冗余改动删掉。
  • testing/python/language/test_tilelang_ascend_language_scalar_index.py：新增多 batch 标量读写回环单测，B 全部 ≥ 2。
  • src/target/codegen_ascend_pto.cc:780 / 798 的同名缺陷本次刻意不动：PTO 的全局 buffer 寻址要另外看，单测 docstring 明确写了只覆盖 ascendc、pto 侧单开 issue 跟踪。

7.4　现在可以这么写

没有新原语、没有新参数、签名不变 —— 本条是纯语义修复（同 PR 里的 real_k / real_n / k_actual / unit_flag 属另一条目）。写法就是原来那个写法：对任意维度的 GM / UB buffer，x = buf[i0, i1, ..., ik]（标量读，发 GetValue）和 buf[i0, i1, ..., ik] = v（标量写，发 SetValue）现在按 OffsetOf(indices).back() 展平，等价于 i0 * s0 + i1 * s1 + ... + ik（si 由 buffer 形状推出）。

7.6　兼容性与坑

兼容性：老代码逐字节不变，靠的是 OffsetOf 在一维 buffer 上是恒等映射，和原来的 op->indices.back() 输出同一个表达式。算子仓里的一维标量取值全部落在这一类 —— act_q_lens[b]（kernel.py:456）、seqused_kv[b]（458）、q_prefix[b]（485）、act_q_lens[bm]（858）、seqused_kv[bm]（859）、q_prefix[v0b]（2598），生成的 Ascend C 一个字符都不变。真正变了的只有原本就写错的多维标量访问 —— 而这些访问在前导维恒为 0 的场景（B=1）下新旧数值也一样，只是表达式从 GetValue(lg) 变成 GetValue(0 * L + lg) 再被编译器折掉。换句话说：升级编译器后，要么行为不变，要么把一个原本就错的结果改对，不存在「老代码被改坏」的第三种情况。

只修了 ascendc codegen。src/target/codegen_ascend_pto.cc:780 / 798 里的同名缺陷本次没动（单测 docstring 明确说明），所以 target 走 pto 的路径上多维标量下标仍然只发最内层。本算子的 Ascend codegen 恒走 tilelang_ascend（非 pto），不受影响；但切到 pto 的 kernel 别以为这条已经好了。

  • 展平用的是 buffer 的逻辑形状。UB buffer 的行宽会被 ascend_collect_buffer_shape.cc 的 AlignInnerDim 按 32B 向上对齐，而对齐后的形状是挂在 kLogicBufferShapes 属性上给分配和声明用的，BufferLoad 手里的 op->buffer 仍是逻辑形状。逻辑行宽不是 32B 整数倍时，buf[i, j] 展平出来的 i * N + j 与实际分配的对齐行宽未必一致。上游单测挑的 N 是 int32 的 8 / 16 / 32（正好 32 / 64 / 128 字节）全是 32B 整数倍，越出这个范围的 UB 多维标量下标要自己核一遍生成的 C。GM buffer 没有这层对齐，不受影响。

8窄行归约按物理行宽 stride —— 只归约宽 tile 里的一段列

Bugfix 待合入 PR #1422

一句话：原来对一块 [M, 512] 的 UB tile 做「每行只归约其中 64 列」会静默算错（只有第 0 行是对的），现在直接写切片 ub[:, a:b] 或 real_shape=[M, 64] 就是对的。

8.1　这块机制是什么

昇腾 vector 单元的行归约有两族指令。tl::ascend::reduce_* 落到的是 AscendC::Reduce* 的 {M, N} shape 重载：它把 src 当成一块连续的 M×N 数据顺序读完，这个重载的语义里没有「行间步长」。AscendC::WholeReduce* 则按 repeat 组织 —— 每个 repeat 处理一段（最多 256 字节），带一个显式的 srcRepStride（以 32 字节 block 计）说明相邻两个 repeat 之间跳多远。

TileLang 的 T.reduce_max/min/sum 在 ascendc target 上一直只落到前者。只要归约的逻辑宽度就是 buffer 的整行宽，两者完全等价；一旦逻辑宽度只是宽行里的一段，前者就无法表达 —— 而它不会报错。

8.2　原来写不出来的东西

内核作者想表达的是：对一块 [16, 512] 的 fp32 UB tile，只归约每行的一段列（[0:64] 或 [64:128]），拿到 [16] 的每行结果。在线 softmax 的 rowmax / rowsum 只对窗口内的有效列有意义，宽 tile 里剩下的是 padding。

改动前会这么写
ub = T.alloc_ub([16, 512], "float")   # 物理行宽 512
acc = T.alloc_ub([16], "float")

# 写法 A: 用切片说明列范围
T.reduce_max(ub[:, 64:128], acc, dim=-1)

# 写法 B: 整块进, 用 real_shape 说明逻辑宽度
T.reduce_max(ub, acc, dim=-1, real_shape=[16, 64])

# 唯一能跑对的绕法: deprecated 的 wholereduce, srcRepStride 手算
# (512 fp32 = 2048B = 64 个 32B block)
T.tile.wholereducemax(
    acc, ub[:, 64:128], 64, 16, 1, 1, 64,
    ReduceOrder="ORDER_ONLY_VALUE",
)
会发生什么编译通过、运行不报错、结果静默错。写法 A 和 B 都落到 tl::ascend::reduce_max<float, 16, 64, -1>(...) → AscendC::ReduceMax<float, AscendC::Pattern::Reduce::AR>(dst, src, tmp, shape={16, 64}, true)。该重载把 src 当连续的 M×N 块读：写法 B 读的是 flat 元素 [0, 16*64)，也就是第 0 行整行 512 个 + 第 1 行整行 512 个，然后把这 16 个 64 列 chunk 当成 16 行的结果吐出来 —— out[0] 是第 0 行的第一个 chunk（恰好对），out[1..7] 是第 0 行其余 7 个 chunk，out[8..15] 是第 1 行的 8 个 chunk。写法 A 同理，只是起点挪到 flat 偏移 64。第 0 行永远是对的（它的起点就是切片起点），其余行全错；没有编译错误、没有 warning、没有 runtime 报错。接在在线 softmax 里的表现是「除第一行外整片输出偏掉」，非常容易被误判成同步 race 或者 QK 本身算错。

为什么在内核侧绕不过去：现成能用的逐行归约里，能表达行间步长的只有 T.tile.wholereducemax / wholereducemin / wholereducesum（tilelang/language/ascend_tile.py:1351/1368/1385）。这三个都带 @deprecated() 装饰器，docstring 明写 Currently, this implementation does not support pto target，而且要内核作者自己把物理行宽换算成 32 字节 block 数（512 fp32 → 64）手填进 7 个位置参数里 —— 一个纯硬件层面的量泄漏进算子代码，还把这段代码钉死在非-pto 后端上。kernel.py 在 692edac 之前正是这么写的。另一条绕法是先把窄列段 T.copy 成一块紧凑的 [16, 64] UB 再归约，但这是在 online softmax 的最内层，每个 KV tile × 每个 m-chunk 都要多付一次搬运和一块额外 UB，纯粹是为编译器缺陷买单。

同族里还有一个非 deprecated 的 T.tile.reduce_sum_mask_experiment(dst, src, mask, repeatTime, srcRepStride)（ascend_tile.py:2558），它确实带 srcRepStride 并发到 AscendC::ReduceSum 的 mask/repeat 重载。但它的语义是把全部元素求成一个总和，不是逐行归约，当不了这个场景的绕法。

8.3　编译器改了什么

  • tilelang/language/reduce_ascend.py：reduce() 在逻辑宽 N 与源 buffer 的物理行宽（src_buffer.shape[-1]）不一致时，往 tl.ascend_reduce 调用的尾部追加一个 int32 物理行宽。
  • tilelang/language/reduce_ascend.py：这个尾参只在 M、逻辑 N、物理行宽三者全是编译期常量且行宽 != N 时才追加（if None not in (m_const, n_const, row_const) and row_const != n_const），因为 codegen 要把 M/N 从模板参数里读回来算 stride。
  • src/target/codegen_ascend.cc：ReduceOpCodegen 先从参数尾部剥掉这个「非 bool 的 IntImm」，值 > 0 就走新分支，改发 tl::ascend::reduce_{max,min,sum}_narrow<T>(dst, src, mask=N, repeatTime=M, srcRepStride=物理行宽*elem_bytes/32)。
  • src/target/codegen_ascend.cc：新分支带一圈 ICHECK 守卫 —— dim 必须是 -1、N*elem_bytes ≤ 256、clear 必须为 true、物理行宽*elem_bytes 必须是 32 的整数倍；元素字节按 dtype 名逐个枚举，认不出的类型直接 ICHECK 失败而不是默认成 4 字节把 stride 悄悄缩放错。
  • src/tl_templates/ascend/common.h：新增 reduce_max_narrow / reduce_min_narrow / reduce_sum_narrow 三个模板，转发到 AscendC::WholeReduceMax/Min/Sum（dstRepStride=1、srcBlkStride=1、显式 srcRepStride；max/min 用 ReduceOrder::ORDER_ONLY_VALUE）。老的 reduce_max / reduce_min / reduce_sum 模板一行没动。
  • src/target/codegen_ascend_pto.cc：PTO 的 tile 本来就带物理 stride 视图、结果是对的，所以只在找 clear 时把下标往前跳过这个可能存在的尾参（clear_idx--）。

CFA 每个 KV tile 在线 softmax 的第一步：对刚算出的 QK 打分 in_ub[ps]（一块 [16, 512] 的 fp32 tile）求每行最大值 m_i。真正有效的列只有运行时的 tw 列（滑窗宽度或 compressed tile 的实际宽度）—— 第 1164 行先对整块 in_ub[ps, :, :] 满宽 fill(-inf)，再只把 [0:tw] 搬进来，所以 [tw:512] 全是 -inf padding；归约宽度按 64 对齐到 tw_64，多出来的 [tw:tw_64] 正是靠这段 -inf 才允许被算进来（reduce_max 忽略、exp 后为 0）。窄窗分支（is_narrow = tw <= BI，BI=128）不愿为 512 列全宽付代价，于是先把 m_i 填成 -inf，再按 64 列一块对 in_ub[ps, :, kc*64 : kc*64+64] 做 T.reduce_max，用 T.tile.max 把每块结果无条件合并进 m_i —— 每一次 T.reduce_max 都是一次窄归约：逻辑宽 64、物理行宽 512、M=16。为什么必须切成 64 列块而不是一次归约 tw_64 列：一个 repeat 上限 256 字节，fp32 就是 64 个元素，这正是 for kc in range(BI // 64) 和 tw_64 = (tw + 63) // 64 * 64 的由来。改动之前这条只能写成 deprecated 的 T.tile.wholereducemax(part, ..., 64, M_CHUNK, 1, 1, 64, ReduceOrder="ORDER_ONLY_VALUE")，把 srcRepStride 手算成 64 个 32B block 填进去（见 692edac 的 diff）；换成 T.reduce_max 后实参从 8 个（7 个位置参数 + ReduceOrder）降到 3 个，硬件量不再泄漏进算子。紧跟其后的 else 一路（宽 cmp tile，512 列全有效、无 padding 可省）反而走普通全宽 T.reduce_max，因为 8 块的标量合并比一条宽指令贵。

8.6　kernel.py 里的真实用法（二）：CFA 在线 softmax 的分母

kernel.py:1350-1367 · CFA · 在线 softmax 的 denom += sum(P)
if is_narrow:
    T.tile.fill(sumP, T.float32(0.0))
    for kc in range(BI // 64):
        if kc * 64 < tw_64:
            T.reduce_sum(
                in_ub[
                    ps,
                    :,
                    kc * 64 : kc * 64 + 64,
                ],
                part,
                dim=-1,
            )
            T.tile.add(sumP, sumP, part)
else:
    T.reduce_sum(
        in_ub[ps, :, :], sumP, dim=-1
    )

同一块 in_ub[ps] 此时装的已经是减完 rowmax、取过 exp 的概率 P，在线 softmax 的分母要把这一 tile 的行和累加进 denom。结构与 rowmax 完全同构，只是算子换成 T.reduce_sum、合并算子换成 T.tile.add、初值 fill 成 0 而不是 -inf。它被放在 cast 之后，是因为 reduce_sum 会破坏 src，必须等 cast 先把 P 读完（中间隔了一个 T.pipe_barrier("v")）。这里窄归约的价值和 rowmax 略有不同：padding 经过 exp 之后是 0，求和其实不怕读到它 —— 真正怕的是读错行。改动之前照直写 T.reduce_sum(in_ub[ps, :, kc*64 : kc*64+64], part, dim=-1)，第 0 行之外全错（out[1..7] 是第 0 行其余 chunk 的和、out[8..15] 是第 1 行的 8 个 chunk），denom 一错整条 softmax 的输出就全错，而且不报任何错。SCFA 侧第 2826-2846 / 2965-2982 行是逐字节相同的两份；SCFA 的 tile 宽度还额外被 topk 压缩过，宽窄两条路在同一次运行里都会真的走到。

8.7　兼容性与坑

兼容性：尾参只在逻辑宽 != 物理行宽且 M / 逻辑 N / 物理行宽三者都是编译期常量时才追加。整块归约（N 就等于物理行宽，也就是绝大多数既有调用）走的参数列表和改动前逐字节一致，codegen 剥参数时看到尾部是 bool（clear）就直接落进老分支。common.h 里老的 reduce_max / reduce_min / reduce_sum / reduce_sum_half 模板一个字符没动，新增的是三个独立的 *_narrow 模板。

但窄路径本身不是纯增量：一批「改动前能编过、只是答案静默错」的写法（列归约 dim=0 的窄切片、clear=False 的窄归约、逻辑宽超过一个 repeat 的窄归约）现在会变成硬 ICHECK 失败。从静默错答案换成编译报错是好事，但升级编译器后原本「能跑」的代码可能编不过。

PTO 侧只在不追加尾参时逐字节不变。追加尾参后 codegen_ascend_pto.cc 已经会跳过它，但 src/transform/allocate_tmp_buffer.cc:276-280（NeedReduceOutputTmp）和 :371-374 仍按 args.back() 是不是 const-false bool 来找 clear，尾参一加就失配 → reduce_out tmp buffer 既不分配也不注入 → 走到 codegen_ascend_pto.cc:3553 的 ICHECK(op->args.size() >= 6 && op->args[4].as<CallNode>()) 编译失败，报 PTO reduce(clear=False) expects an injected temporary output buffer.。也就是说 PTO + 窄归约 + clear=False 是一条改动前可用、改动后编译失败的路。最小复现：把 examples/reduce/example_row_reduce_max_slice_buffer.py（target="pto"、[4,8] buffer、real_shape=[4,4]）那句 reduce 加上 clear=False。仓内现有的 clear=False 单测用的是满宽 (64,64) 归约、不追加尾参，CI 碰不到这条路。
  • 一个 vector repeat 最多 256 字节 → 逻辑宽度 × 元素字节 ≤ 256（fp32 只能 64 列，fp16 128 列）。超了 ICHECK 直接报错并提示自己切段合并 —— kernel.py 里那圈 for kc in range(BI // 64) 加 T.tile.max / T.tile.add 就是这么被逼出来的。
  • 只支持 dim == -1（行归约）。dim == 0 的列归约走不到这条路，会撞 ICHECK narrow reduce is only implemented for a row reduce。
  • 不支持 clear == False：不能把窄归约的结果并进目标，必须归约到 scratch 再自己合并（kernel.py 用的就是 part + T.tile.max / T.tile.add）。
  • 物理行宽 × 元素字节必须是 32 的整数倍（srcRepStride 以 32B block 计），否则整除会截断、stride 走短甚至变 0。ICHECK 守着这一条。
  • 只有 M / 逻辑宽 / 物理行宽三者全是编译期常量时才会走窄路径。任何一个是符号量，前端就不追加尾参，直接退回老的连续块走法且不报任何错 —— 也就是又变回静默错答案。所以窄归约的宽度千万别写成运行时变量；kernel.py 的做法是宽度定死 64、用运行时的 tw_64 去 guard 哪些 chunk 参与。
  • 切片起点 a 没有任何一层校验：它会原样变成 WholeReduce* 源操作数的元素偏移，而向量指令要求操作数起始地址 32 字节对齐（fp32 → a 是 8 的倍数，fp16 → 16 的倍数）。ub[:, 4 : 68] 这种偏移前端和 codegen 都会照单全收、不报错。上游单测的 offset 只覆盖 0 / 64 / 448，kernel.py 用的是 64 的倍数，未对齐的起点没有被验证过。
  • 元素字节是按 codegen 侧的 dtype 名枚举的：float / int32_t / uint32_t = 4，half / bfloat16_t / int16_t / uint16_t = 2，int8_t / uint8_t / float8_e4m3_t / float8_e5m2_t = 1。int32 用不了：前端 tilelang/language/ascend.py::_dtype 的 type_map 把 int32 发成 int，而枚举表里写的是 int32_t，对不上 —— int32 buffer 做窄归约会撞 ICHECK(elem_bytes > 0)，报 narrow reduce does not know the element size of 'int' 直接编译失败（int32_t 那一支是死代码）。int64 / uint64 / int4 同样不在表里。至少这是硬报错而不是默认 4 字节把 stride 悄悄缩放错。
  • 窄路每个 chunk 要一次 reduce 加一次标量合并。当整行都有效时（比如 512 列全参与）走普通宽 reduce 更快 —— kernel.py 用 is_narrow = tw <= BI 分路，注释里写明「8-chunk 的标量开销太贵」。
  • 归约 dst 留在 UB 里用 [M, 1] 没问题（kernel.py 的 part / sumP 就是）。但要把每行标量直接 UB→GM 导出时得用一维 [M]，否则会撞上 32B block 对齐重排的老坑 —— 上游单测的输出就是特意写成 [M] 的。

上游单测：testing/python/language/test_tilelang_ascend_language_narrow_reduce.py（PR 新增，128 行）。M=16、buf_n=512，两种写法各一组 parametrize：test_narrow_reduce_real_shape（op ∈ {sum, max, min} × valid ∈ {64, 32, 16}）和 test_narrow_reduce_column_slice（op ∈ {sum, max, min} × offset ∈ {0, 64, 448}），后者顺便覆盖了切片起点贴近 buffer 尾部的情况。测试是设计成旧代码必挂的：_data() 把归约窗口之外的列灌成 poison（sum → 7.0、max → 100.0、min → -100.0），一旦按连续 M×N 块乱走就一定会读进 poison，靠运气蒙对不可能。更关键的一手是 poison 只灌到 data[1:, ...]，第 0 行故意留干净 —— 因为第 0 行在错误走法下也恰好是对的，如果连它也灌 poison，就分不清到底是哪些行真的走错了；docstring 里把这一点写明了。

9.3　解构之后反而更快

把 QK 从 gemm_v0_fixp 模板解构成内核驱动的 T.mma 序列（第 6 节），性能不是持平，是直接快了约 12 个百分点。原因藏在模板内部：

src/tl_templates/ascend/common.h · gemm_v0 首尾
SetFlag<HardEvent::MTE2_MTE1>(L0AB_EVENT);    // 1187
WaitFlag<HardEvent::MTE2_MTE1>(L0AB_EVENT);   // 1188  <-- set 紧跟 wait，同一个 event
...
SetFlag<HardEvent::MTE1_MTE2>(L0AB_EVENT);    // 1248
WaitFlag<HardEvent::MTE1_MTE2>(L0AB_EVENT);   // 1249  <-- 同样

判据：SetFlag(e) 紧跟 WaitFlag(e)（同一 event）不是给某个 buffer 定序，而是 drain 整条 pipe。效果等同一条 pipe barrier。

于是模板每被调用一次，就强制「所有 MTE2 完成才能开始 MTE1」，并且挡住下一次 KV 页加载在当前这次 gemm 的 L1→L0 结束之前发出 —— 正好把内核精心搭的 3 槽 KV 环存在的全部意义废掉。解构之后这两对围栏自然消失（内核自己的 per-slot MTE2_MTE1 / MTE1_MTE2 是定向的，守同样的 hazard 但不 drain 整条 pipe）。
