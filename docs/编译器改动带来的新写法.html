编译器改动带来的新写法

六个上游 PR、八处改动 —— 每一处让 TileLang 里原来写不出来（或者写出来编不过、算错、挂死）的东西，现在能写成什么样子

这篇文档回答一个问题：这些编译器 bugfix 和特性增加，让 TileLang 里多出了哪些写法。每一节的结构都一样 —— 原来想写什么、当时会怎样（编不过 / 静默算错 / 挂死）、现在能怎么写（最小例子）、kernel.py 里的真实用法、以及用的时候要注意什么。

读者设定：写过 TileLang kernel、知道昇腾的 GM / L1 / L0A / L0B / L0C / UB 各级内存和 Cube / Vector / MTE 各条流水，这些不再赘述。不需要懂编译器内部 —— 全文不涉及编译器实现，只讲写法；想追改动细节的话，每节标题旁有 PR 链接。

1整数标量 max/min 发成三元表达式 —— 让运行期的 max/min 在昇腾后端编得过

Bugfix 已合入 2026-07-07 PR #1309

一句话：原来 kernel 里只要出现一个运行期（编译期折不掉）、且两个操作数整数宽度不一致的 T.max / T.min，生成的 AscendC 就会被 bisheng 顶回来 call to 'max' is ambiguous，整个 kernel 编不出来；现在整数标量 max/min 一律打成 (a > b ? a : b)，运行期的 max/min 可以随便写。

1.1　原来写不出来的东西

昇腾后端最后交给 bisheng 的是一份 AscendC C++ 源码。改动前，T.max(a, b) / T.min(a, b) 在这份源码里就是一个不带命名空间的裸 max(a, b) / min(a, b) 函数调用，类型决议整个甩给 C++ 重载；而这份源码的 prelude 既没有 <algorithm>，也没有 using namespace std，裸 max 解析到哪个重载完全取决于 CANN 头文件链恰好拖进来了什么。

还有一个前提决定了这个坑什么时候才现形：编译期能算出结果的 max/min 会被提前折成常量，根本不会出现在生成的源码里。只有依赖运行期值（张量取值、grid 变量、T.alloc_var 游标、T.symbolic 形状）的 max/min 才会真的被打印出来。

内核作者想写的正是「拿运行期的值取 max/min」——滑动窗口的左边界不能小于序列开头 0、分页 KV 一次 DMA 能搬多少行要在「本物理页剩余行」和「窗口还差多少行」之间取小、tile 尾段的实际列宽要在 tile 宽和剩余 token 数之间取小。这些量依赖 seqused_kv / act_q_lens / block_table 或 grid 变量，编译期一个都定不下来。

改动前会这么写
BI, ORI_WIN_LEFT = 128, 127        # 编译期常量放 prim_func 外层

@T.prim_func
def main(K: T.Tensor((S, D), "float16"), O: T.Tensor((S, D), "float16")):
    with T.Kernel(core_num, is_npu=True) as (cid, vid):
        k_ub = T.alloc_ub((BI, D), "float16")
        # cid 是 grid block 变量,生成的源码里是 auto cid = AscendC::GetBlockIdx(),
        # auto 取的是 GetBlockIdx() 的 int64 返回类型。它对 int 字面量 0 取整数
        # max,两个操作数的宽度就不一致了。
        ori_left = T.max(cid * BI - ORI_WIN_LEFT, 0)   # <-- 这一行就是雷
        T.copy(K[ori_left, 0], k_ub)
        T.copy(k_ub, O[cid * BI, 0])
会发生什么不是算错，是根本编不出来。生成的源码把这个 max 内联成裸调用 max((cid * 128 - 127), 0)，直接塞进 DMA 的行偏移表达式里；bisheng 编这份源码时报 error: call to 'max' is ambiguous —— max(int64_t, int) 的多个重载同样可行，决议不出来。报错点落在生成的 C++ 临时文件上，回溯不到 Python 源码的那一行，第一次撞上只看得到一坨陌生的行号。

触发条件比「任何运行期整数 max/min」窄：歧义来自两个操作数整数宽度不一致的组合，典型就是 auto cid = AscendC::GetBlockIdx() 拿到的 int64 对上 int 字面量。同宽的运行期 max/min（两个 int32）在改动前也能编过，所以这个坑一直潜伏着，直到某处少了一个 T.cast 才炸。

为什么绕不过去：

  1. max/min 的结果紧接着就要当 T.copy / copy_pa 的切片上下界和搬运行数用（ori_kv[pv_ph, pv_rem : pv_rem + pv_run, ...]），必须是表达式。T.if_then_else 会 lower 成语句（int32_t condval; if {...}），塞不进 copy 的参数表 —— kernel.py 自己在 1961-1969 行把这条写进注释了。
  2. 唯一的表达式级替代是每处手写 tir.Select(a > b, a, b)。kernel.py 里 T.min( / T.max( 合计 37 处（T.min 28 + T.max 9），人肉替换啰嗦，漏一个就炸。
  3. 而且手写 tir.Select 只能覆盖内核里自己写的那些：内核侧一个 T.max 都不写，也不保证生成的源码里没有整数取大取小 —— UB 的分配与地址规划自己就会造出来，尺寸超 int32 时以 int64 出现，正好落在混宽那一类里。

1.2　现在可以这么写

没有新 API，签名一个字都没变。写的还是 T.max(a, b) / T.min(a, b)，变的只是它们在昇腾后端被打印成什么：从裸函数调用变成内联三元 (a > b ? a : b) / (a < b ? a : b)，没有重载需要决议。生效条件严格限定为「整数或无符号整数」且「标量（不是向量）」。

最小例子
import tilelang
import tilelang.language as T

M, N, BM = 1024, 256, 128
M_NUM = M // BM        # 8 个 block
ROWS = BM // 2         # 编译期常量必须留在 prim_func 外层

VEC_PASS_CONFIGS = {
    tilelang.PassConfigKey.TL_ASCEND_AUTO_CV_COMBINE: True,
    tilelang.PassConfigKey.TL_ASCEND_AUTO_CV_SYNC: True,
    tilelang.PassConfigKey.TL_ASCEND_AUTO_SYNC: True,
    tilelang.PassConfigKey.TL_ASCEND_MEMORY_PLANNING: True,
}

@T.prim_func
def main(A: T.Tensor((M, N), "float"), C: T.Tensor((M, N), "float")):
    with T.Kernel(M_NUM, is_npu=True) as (cid, vid):
        # cid 是 int64 的 grid block 变量,和 int 字面量 0 混宽做整数 max。
        src = T.max(cid - 1, 0)
        a_ub = T.alloc_ub((ROWS, N), "float")
        voff = vid * ROWS
        T.copy(A[src * BM + voff, 0], a_ub)     # 读左邻 tile(首块读自己)
        T.copy(a_ub, C[cid * BM + voff, 0])     # 写自己这块 tile

func = tilelang.compile(main, out_idx=[-1],
                        pass_configs=VEC_PASS_CONFIGS, target="ascendc")

重点只有 src = T.max(cid - 1, 0) 这一行：cid 是 int64 的 grid 变量，0 是 int 字面量，混宽标量整数 max。改动前这行打成裸的 max(cid - 1, 0)，bisheng 报 call to 'max' is ambiguous，整个 kernel 编译失败；改动后打成 ((cid - 1) > 0 ? (cid - 1) : 0)，结果就是「每块读它左边那块、第 0 块读自己」。另外三处不是凑数：把 src 用作 DMA 行偏移，是为了让这个 max 活到生成源码里（只算不用会被优化掉，问题就复现不出来）；M_NUM / ROWS 必须定义在 @T.prim_func 外层，否则会被绑成 Let-Var、把 a_ub 的 shape 变成符号量而不是常量；pass_configs 也不能省 —— target="ascendc" 默认不带任何 pass config，不显式开 AUTO_SYNC 的话 MTE2 和 MTE3 之间不插 flag，两条 DMA 无依赖就是数据竞争。

1.3　kernel.py 里的真实用法（一）：SWA 滑窗左边界

kernel.py:3538-3544 · SWA · QK 打分
act_q = act_q_lens[b]
if s < act_q:
    act_kv = seqused_kv[b]
    tok = q_prefix[b] + s
    s_global = act_kv - act_q + s
    ori_left = T.max(s_global - ori_win_left, 0)
    win = s_global + 1 - ori_left

这是 SWA 每个 query token 进 QK 之前必须先算出来的两件事：它能看到的 KV 区间从哪开始、有多长。s_global = act_kv - act_q + s 是该 token 在整条 KV 序列上的全局位置，由 seqused_kv[b]（该 batch 真实 KV 长度）和 act_q_lens[b]（该 batch 真实 query 长度）算出，两者都是运行期从张量里读的。滑窗左边界本该是 s_global - ori_win_left（滑窗左半径，本仓断言 ≤127），但对序列开头那几个 token 这个值是负的，必须限制到不小于 0，否则 win 会算成大于 128、后面 T.copy 的切片直接越界读到别的 batch 的 KV。取完 max 之后 win = s_global + 1 - ori_left 就是该 token 真实的窗口长度：序列开头短、往后稳定在 128。这两个值一路喂给 copy_pa 的搬运行数、gemm 的 real_k / k_actual、以及在线 softmax 的有效列数，是整个 SWA 的骨架尺寸。

这里的操作数目前都在 int32 域内（3504-3505 行 b = T.cast(pid // max_seq, "int32") / s = T.cast(pid % max_seq, "int32") 已经把 grid 变量降到 int32），所以这条具体的 T.max 在改动前也能编过。改动的价值在于把「能不能编过」从「有没有记得 cast」这个约定里解耦出来 —— 少写一个 cast，或者哪个中间环节自己造出一个 int64 操作数，都不再是编译失败。

1.4　kernel.py 里的真实用法（二）：分页 KV 一次能搬多少行

kernel.py:3867-3889 · SWA · PV 分页搬运
pv_done = pv_cur - pv_start
if pv_done < winm:
    pv_lg = pv_cur // ori_block_size
    pv_ph = ori_block_table[bm, pv_lg]
    pv_rem = pv_cur % ori_block_size
    pv_run = T.min(
        ori_block_size - pv_rem,
        winm - pv_done,
    )
    T.copy(
        ori_kv[
            pv_ph,
            pv_rem : pv_rem + pv_run,
            0,
            nl * PV_NW : nl * PV_NW + PV_NW,
        ],
        kv_ring[
            slot,
            pv_done : pv_done + pv_run,
            0:PV_NW,
        ],
    )
    pv_cur += pv_run

这是 PagedAttention 的分页走查循环：窗口内的 V 在显存里不连续，按 ori_block_size 行一页散落在 ori_kv 里，页的物理编号要经 ori_block_table[bm, pv_lg] 查表。每轮搬一段连续行，段长 pv_run 同时受两个上界约束 —— ori_block_size - pv_rem 是「当前物理页从 pv_rem 往后还剩多少行」（跨页得换 pv_ph，不能一口气搬过去），winm - pv_done 是「本 token 的窗口还差多少行没搬完」（搬多了会把窗口外的 KV 混进 PV 累加，结果静默错）。取小之后 pv_run 同时出现在源切片、目的切片和游标推进三个位置 —— 源和目的两侧都切，运行期长度才真的落到 DMA 的 validRow 上。三个操作数里只有 ori_block_size 是编译期常量，pv_rem / winm / pv_done 全是运行期值，这个 min 折不掉、必然出现在生成的源码里。这也是「绕不过去」最直白的例子：pv_run 是切片边界，只能是表达式，T.if_then_else lower 成语句就塞不进 T.copy 的参数表。

1.5　kernel.py 里的真实用法（三）：SCFA 压缩域尾 tile 列宽

kernel.py:1984-2006 · SCFA · 压缩域 tile 划分
is_ori = tile == 0
ori_left = T.max(s_global - ori_win_left, 0)
win = s_global + 1 - ori_left
rel = tile - 1
# tile column width. Inner cmp-tail conditional
# is T.min (last tile: act_cmp-rel*512 < 512;
# else >=512 -> 512); the ori/cmp pick is a
# tir.Select. Both lower to an INLINE ternary
# (SelectNode); a runtime T.if_then_else instead
# lowers to a statement (int32_t condval; if{}),
# which the codegen cannot inline into a load
# or mma arg list (bad C++). For ori (rel=-1)
# the cmp branch = min(512, act_cmp+512)=512,
# unused.
tw = tir.Select(
    is_ori,
    win,
    T.min(
        S2_BASE,
        T.min(act_cmp, topk_cmp)
        - rel * S2_BASE,
    ),
)

SCFA 的 KV 分两段：一段是原始滑窗（is_ori，tile 0），一段是被 topk 选中并合并过的压缩 KV（tile ≥1，每块 512 宽）。上面 1951-1954 行先算 act_cmp = (s_global + 1) // cmp_ratio（该 token 在压缩域上可见的 token 数）和 cmp_tiles = (T.min(act_cmp, topk_cmp) + S2_BASE - 1) // S2_BASE，topk_cmp 是 topk 只保留 K 个压缩块的上限，取小才是真正要算的压缩 token 数，直接决定这个 token 要跑几轮 tile 循环。这里的 tw 是当前 tile 的实际列宽：压缩分支里外层 T.min(S2_BASE, ...) 处理尾 tile 不足 512 的情况，tw 随后在 2010 行被切成 ncols = T.min(BI, tw - cb * BI) 的 128 宽列块喂给 gemm（CFA 的 505 行是同款代码，属于另一个 kernel）。

代码里那段原生注释顺带解释了为什么只能用内联三元类的东西（tir.Select / T.min）而不能用 T.if_then_else —— 后者 lower 成语句，没法内联进 copy_pa 的参数表，生成的是编不过的 C++。同时这也是全仓嵌套最深的一处 max/min，直接对应 1.6 里的三元膨胀问题。

1.6　用的时候要注意什么

  • 只管标量整数。浮点标量的 T.max / T.min 仍然发裸 max(a, b)，同样的歧义风险原样还在，只是本仓还没撞到；lanes() > 1 的向量 max/min 也走老路径，NaN 语义和向量语义一个字没碰。真要对浮点做运行期 max/min，优先用向量原语（ascend_max / clamp_max 一类），或自己写 tir.Select(a > b, a, b)。
  • 三元把每个操作数各打印两遍，嵌套按 2 的幂膨胀。生成的源码里 a 和 b 各出现两次，所以每嵌一层，内层的文本翻一倍。上面 1973-1977 行的 T.min(S2_BASE, T.min(act_cmp, topk_cmp) - rel * S2_BASE)：外层把操作数 b 打两遍，内层 min 的三元文本出现 2 次，它的叶操作数 act_cmp / topk_cmp 各出现 4 次；再往里嵌一层，最内层的叶操作数就是 8 次。写多层嵌套的 max/min 时留意生成源码的体积和可读性，必要时先把内层结果落到 T.alloc_var 标量上。（tir.Select 每个分支只印一次，不额外放大。）
  • dump 出来看不到三元 ≠ 改动没生效。编译期能折的 max/min 早就成了常量，压根到不了后端。要验证这条改动，操作数里必须至少有一个是真的运行期值（张量取值、grid 变量、alloc_var 游标）。
  • 只对 target="ascendc"（非-PTO）生效。pto 编出来的是另一份源码，走的不是这条路径。本仓 SWA / CFA / SCFA 全部走非-PTO，所以这条是刚需；在 pto 上验证是验不出差别的。
  • 三元只解决重载决议，没解决整型提升。a 和 b 的符号扩展 / 整型提升还是 C 的老规矩：int64 的 grid 变量对 int 字面量取 max，结果类型是 int64。取完 max 要当 int32 索引用的话，仍然建议显式 T.cast(..., "int32") —— kernel.py:3536-3537 的两个 T.cast 就是先把 grid 变量降到 int32、让后面所有 max/min 都留在 int32 域的写法，值得照抄。
  • 生成的表达式外层已经带括号，可以直接嵌进更大的算式，不用自己再包一层。
  • 别指望它顺带修好整数除法。生成的源码里 // / % 发成 C 的 / 和 %（截断语义），负被除数下和 floor 语义差 1，那是另一个独立的坑。用 max/min 限范围、用 // 取页号的组合里，如果操作数可能为负，先确认除法那一侧。

2gemm_v0 在模板内部按列切 N，让 headDim=512 的 PV 矩阵乘能一句写完

Bugfix 已合入 2026-07-14 PR #1312

一句话：原来 gemm_v0 只切 K 不切 N，N 一大（PV 的 headDim=512）就要把 128KB 的 B 片塞进整块只有 64KB 的 L0B，运行期把 cube 打崩；现在超宽的 B 会自动按 128 列切开，一句 T.gemm_v0 就能写完整个 PV。

2.1　原来写不出来的东西

昇腾 cube 单元做一次 mma，两个操作数必须先从 L1 搬到 L0A / L0B 这两块各 64KB 的片上缓冲，累加器落在 L0C。T.gemm_v0 是块级矩阵乘：给它 A、B、C 三个 buffer，它把 <M, N, K> 的 gemm 拆成若干次 mma，并用 L0A/L0B 双缓冲（乒乓）把「L1→L0 搬运」和「上一片的 mma」重叠起来。单槽预算是分段的：整条切分序列只有一步时，单槽可以占满整块 64KB；一旦步数大于 1、乒乓真的开始轮转，每槽就只剩 32KB。

改动前它只在 K 方向切（每片 K 长度就是前端参数 kL0Size，默认 128），B 在 L0B 里占的大小恒等于 N * kL0Size——N 有多大就占多大，装不装得下没人兜底。

attention 的第二个矩阵乘 PV（O = P @ V）想按 Ascend C 参考的 ComputeMm2 那样一句写完：M=G=64、N=headDim=512、K=window≤128，P 和整块 V 各在一个 L1 buffer 里，结果累加进一块 [64, 512] 的 L0C。

改动前会这么写
# 想写成这样（sparse_attn_sharedkv 早期 SWA，一句 PV）：
kv_l1 = T.alloc_L1([BI, D], dtype)        # [128, 512] 整窗 V
p_l1  = T.alloc_L1([G, BI], dtype)        # [64, 128]  P
acc_o_l0c = T.alloc_L0C([G, D], accum_dtype)
T.copy(workspace_p[cid, :, :], p_l1)
T.gemm_v0(p_l1, kv_l1, acc_o_l0c, init=True)   # M=64, N=512, K=128

# 实际当年被迫写成这样（K 劈两半的绕法）：
kv_l1_0 = T.alloc_L1([BI // 2, D], dtype)
kv_l1_1 = T.alloc_L1([BI // 2, D], dtype)
p_l1_0  = T.alloc_L1([G, BI // 2], dtype)
p_l1_1  = T.alloc_L1([G, BI // 2], dtype)
T.copy(workspace_p[cid, :, 0 : BI // 2], p_l1_0)
T.copy(workspace_p[cid, :, BI // 2 : BI], p_l1_1)
T.gemm_v0(p_l1_0, kv_l1_0, acc_o_l0c, init=True)
T.gemm_v0(p_l1_1, kv_l1_1, acc_o_l0c, init=False)
会发生什么第一种写法编译得过，运行期挂死：aicore 报 CCU instruction address check error。不是逻辑错，是地址越界——B 片 = K * N * sizeof(half) = 128 * 512 * 2 = 128KB，而整块 L0B 只有 64KB，超了 2 倍。（这个 shape 的 K=128 恰好只切出一片，乒乓不轮转，单槽拿到的已经是整块 64KB 的最宽松预算，仍然差一倍；K 再大一点、乒乓真的轮转起来，每槽只剩 32KB，差得更多。）写穿 L0B 之后 cube 的指令/地址被冲掉，表现为硬件异常而不是干净的报错。

同一个 kernel 的 QK 反而没事：它走 transpose_B 路径，N = block_I = 128、B 片 128 * 128 * 2 = 32KB，K=D2=256 分两个 K 片、乒乓轮转，32KB 正好卡在预算里。所以这个坑只有 N 大的那一侧才踩得到。

为什么绕不过去：内核侧只能拿 K 换 N——把 K=BI 的收缩拆成两个 BI/2 半，让每个 B 片降到 512 * 64 * 2 = 64KB。传一个切片进去在语法上可行，但 L1 上的 B 走 zN 分形布局：沿 K 切出来的后半段在 zN 下根本不是一段连续内存，用一个指针偏移表达不出来，写出来也不报错、是静默指到错地址。所以「切一刀」必须真的多申请一组完整 L1 buffer。代价是三层的：

  1. L1 占用翻倍，且窗口被人为劈成两段；
  2. workspace 的读写从 1 次变 2 次，cube 侧和 vector 侧的 cross-core sync 点数要逐个配平（半 0 写↔半 0 读、半 1 写↔半 1 读），错一个就死锁；
  3. 为了配平，vector 侧的 s_ub / P 也得按列半段搬，而列半段的 UB 子块拷贝会把 s_ub 的 stride 搞错、逐行污染 LSE。

更要命的是：这么切之后 tiling 已经和 Ascend C 参考的 ComputeMm2（按 N_SPLIT_SIZE 切 N）对不上了，忠实复刻这个目标本身就废了。真正的切分点在 N 上，而 L0B 的槽划分、乒乓事件、L0C 的列带偏移都不在内核这一层——内核能做的只有把窗口手工劈成几段、再多申请几组 L1 buffer，够不着真正该切的那一刀。

2.2　现在可以这么写

最小例子
import tilelang
import tilelang.language as T

CUBE_CONFIGS = {
    tilelang.PassConfigKey.TL_ASCEND_AUTO_SYNC: True,
    tilelang.PassConfigKey.TL_ASCEND_MEMORY_PLANNING: True,
}

@T.prim_func
def pv(P: T.Tensor((64, 128), "float16"),
       V: T.Tensor((128, 512), "float16"),
       O: T.Tensor((64, 512), "float16")):
    with T.Kernel(1, is_npu=True) as (cid, _):
        P_L1 = T.alloc_L1((64, 128), "float16")     # M=64,  K=128
        V_L1 = T.alloc_L1((128, 512), "float16")    # K=128, N=headDim=512
        O_L0 = T.alloc_L0C((64, 512), "float")
        with T.Scope("C"):
            T.copy(P[0, 0], P_L1)
            T.copy(V[0, 0], V_L1)
            # 改动前：这一句要把 K*N*2 = 128*512*2 = 128KB 的 B 片一次塞进
            #   L0B（整块 64KB）--> 运行期 aicore 报 CCU instruction
            #   address check error，不是编译报错。
            # 改动后：N 自动切成 4 段、每段 128 列，每段 B 片
            #   128*128*2 = 32KB 正好装下，分别累加进 O_L0 对应的那条
            #   列带。内核这一行一个字都不用改。
            T.gemm_v0(P_L1, V_L1, O_L0, init=True)
            T.copy(O_L0, O[0, 0])

f = tilelang.compile(pv, out_idx=[-1], pass_configs=CUBE_CONFIGS, target="ascendc")

关键在 T.gemm_v0(P_L1, V_L1, O_L0, init=True) 这一行——它在改动前后逐字节相同，区别全在它展开出来的指令：改动前它编译得过、跑起来把 cube 打崩；改动后它按 4 段 × 128 列自动切分，正好复现 Ascend C 参考里 ComputeMm2 的 N_SPLIT_SIZE 切法。pass_configs 那两行不能省：tl.ascend_auto_sync 默认是关的，关掉之后 MTE2(GM→L1) → MTE1(L1→L0) → M(mma) → FIX(L0C→GM) 之间一条 set_flag/wait_flag 都不会插，是裸的跨 pipe RAW 竞争。另有两个容易撞的形状约束：V_L1 必须是 (K, N) 且走默认 zN 布局（自动切出来的列偏移是按 zN 算的）；M=64 也不是随便取的，L0C 要整块放下 M（按 16 向上取整）× N × 4 字节 = 128KB，M 再大就超出 A2/A3 的 L0C 了。

前端签名一个参数都没加。切 N 是为了塞进 L0B 而做的内部行为，不该让内核作者操心。前端仍然是 T.gemm_v0(A, B, C, transpose_A=False, transpose_B=False, init=False, kL0Size=128, n_actual=None)（其中 n_actual 是后来 #1352 加的，与本条无关），M/N/K 照旧从 A/B/C 三个 buffer 的 shape 推出来。变化只在语义上：以前「N 必须小到能整个塞进一个 L0B 槽」是一条没写出来的隐式前提，越界就是运行期硬件异常；现在超出单槽的非转置 B 路径会自动切段，真正装不下的情况改由编译期断言明确拒绝。

CFA 的 QK 打分：把这个 token 的 64 个 query head（q_l1 的第 0 个 D 半段）和 KV 环里 slot s0 的 K 做 Q @ K 转置乘，结果累加进 cL0 的第 cs 个槽。DEBUG_SERIAL 是一条排障开关：打开时用整块 T.gemm_v0 加 T.barrier_all() 串行跑，关掉时（生产路径，DEBUG_SERIAL = False）走手工解构的 mma 流水。列在这里是为了展示 #1312 的兼容性保证，而不是它的切分能力：这一句是 transpose_B=True，转置 B 的路径一律不切 N，展开成和加 N 切分之前完全同一条单趟路径。M、N 由 cL0[cs] 的 shape [G=64, BI=128] 推出，B 片 128 * 128 * 2 = 32KB，本来就装得下——这也正是 QK 侧从来没踩过 L0B 溢出的原因。

2.5　用的时候要注意什么

  • N 必须能被自动切出的列宽整除，否则编译期报 "gemm_v0 N-tiling requires N divisible by the N tile size"。half + kL0Size=128 时列宽是 128，所以大 N 的非转置 caller 实际要求 N 是 128 的倍数。
  • 只有 transpose_B == false 的路径会被切。一个 N 很大的 transpose_B caller 不会被自动救回来，而是撞上 "gemm_v0: the (kL0Size x nTile) B tile does not fit its L0B ping-pong slot"——这是有意的，因为转置 B 在 L1 里的布局不同，列偏移公式不通用。
  • 列偏移是按 zN 分形布局算死的。B 在 L1 里必须是默认的 zN 布局；换了别的 layout annotation，这个偏移就是错的，而且是静默算错、不报错。
  • M 方向不切。M * kL0Size * sizeof(T1) 必须装进 L0A 的单槽预算，M 太大编译期就被挡下。想要更大的 M 只能调小 kL0Size——但那会同时把自动切出的列宽调大（列宽 = 32KB / (kL0Size * 元素字节数)），等于把压力从 L0A 挪到 L0B。
  • 乒乓预算是分段的：整条切分序列只有一步时，单槽可以用满 64KB；一旦步数大于 1、乒乓真的开始轮转，每槽就只剩 32KB。所以「多加了一个 K 片之后突然编译不过」是预期行为，不是回归。
  • n_actual（transpose_B 路径上运行期少算几列）和 N 切分是两个正交机制，不会同时生效：n_actual 只走转置路径，N 切分只走非转置路径。别指望用 n_actual 规避大 N 的 L0B 溢出。
L0C 容量编译期不检查。L0C 大小是 device-dependent（A2/A3 128KB、A5 256KB），编译期拿不到 arch 信息，写死一个字面量会误拒合法的 A5 caller、或者在 A2/A3 上放行超限的 caller，所以这条约束完全由内核作者自己算：M（按 16 向上取整）× N × 累加元素字节数 超了不会有任何提示。典型陷阱：M=128、N=512、fp32 累加需要 256KB，在 A2/A3 上会静默踩坏 L0C。

3GM↔UB copy / atomic_add 的最内（列）维可以是运行期表达式了 —— 按真实窗口宽度搬，而不是按 buffer 定宽搬

Bugfix 已合入 2026-07-17 PR #1337

一句话：原来 GM↔UB 的 T.copy 只能切编译期常量宽度，一旦切成运行期宽度 [..., 0:tw] 内核根本编译不过；现在最内（列）维可以给运行期宽度，只搬真实有效的那几列。

3.1　原来写不出来的东西

GM↔UB 的一次搬运是「M 行 × N 列」的二维形状：每行搬 N 列有效数据，行与行之间按 UB tile 的行距往下跳。行距由 UB buffer 声明时的最内维定死，是编译期常量；每行真正搬多少列则是另一个量，硬件本来就允许它比行距小（差出来的那段就是行间跳过的空档）。改动前，前端要求这两个量都是编译期常量，于是「行距 512、这一次只搬 winm 列」这种形状在 TileLang 里根本表达不出来。

cfa / scfa 的 vector softmax 正好卡在这里：UB 和 GM workspace 都是定宽 S2_BASE = 512，但真实窗口宽度（ori 窗 ≤ 128，cmp tile ≤ 512）只有运行期才知道。内核作者想对齐 Ascend C 参考实现 block_vector.h 的 DealBmm1（它只处理 columnCount = actualSingleProcessSInnerSizeAlign 列），把 T.copy 切成运行期宽度，只搬、只写有效的那几列。

改动前会这么写
# UB / GM workspace 都是定宽 S2_BASE = 512,
# 但真实窗口宽度只有 64 / 128, 且要等 seqused_kv / act_q_lens 读出来才知道
tw = tir.Select(is_first, win, T.min(S2_BASE, act_cmp - rel * S2_BASE))
tw_a = (tw + 15) // 16 * 16          # 纯运行期表达式

# 想只搬有效的那几列(= 参考实现 DealBmm1 的 columnCount)
T.copy(
    workspace_s[cid, buf, r0 : r0 + M_CHUNK, 0:tw],
    in_ub[ps, :, 0:tw],
)
# ...
T.copy(
    out_ub[:, 0:tw_a],
    workspace_p[cid, buf, r0 : r0 + M_CHUNK, 0:tw_a],
)
会发生什么编译期直接挂，不是算错、也不是挂死 —— 根本编不出二进制。生成的 .cpp 里会出现形如 copy_gm_to_ub<float, T.min(512, act_cmp - rel * 512), 16>(...) 的东西，编译器报 error: use of undeclared identifier 'T'。这个 T 跟模板里的 typename T 无关，纯属撞名 —— 认这条报错就够了，顺着模板往下查是白费力气。

为什么绕不过去：

  1. 前端没有「只给有效列数、不切 region」的口子。不切 → extent == shape == 512，满宽搬，零收益；切 0:tw → 撞上面那条编译错误。两条路都堵死。
  2. 没有别的原语能替代。copy_l0c_to_gm 是 L0C→GM，copy_pa 是 paged gather，都发不出「运行期宽度 + strided 连续 GM↔UB」这条 DMA。
  3. 退回定宽 512 搬的代价。读侧会把窗口外上一轮遗留在 workspace_s 里的 stale QK 一并读进 UB，而后面定长满宽的 reduce_max / reduce_sum 会读到这段区间 —— 吃的是上一轮的残留分数（内容不可控），lse 与 softmax 分母直接错；写侧每个 m-chunk 都按 512 列满宽写回，而 ori 窗那一支实际只有 ≤ 128 列有效，其余全是废数据。代价在 profile 上可量：满宽搬运时 CFA 场景 vector 核的 mte2 耗时是 Ascend C 参考实现的 1.8 倍（msprof 台账记录 1722 vs 954）。
  4. 穷举编译期宽度也不行。tw 由 seqused_kv / act_q_lens 在运行期决定，1..512 全枚举成静态分支是代码爆炸。

需要这个能力的是撤掉 softmax_flash_v2 提案之后的这版实现，不是一开始就需要：改动前 vector 侧只能满宽搬（workspace_s[cid, buf, vid * G2 : (vid + 1) * G2, :] → s_ub，整整 128 列），窗口对齐由 T.tile.softmax_flash_v2 这个原语 的 columnCount = win_align / actualColumnCount = winm 两个参数承担，库自己只归约有效列。把它换成手拼的定长满宽 reduce_max / exp / reduce_sum 之后，窗口外那段就没人替内核挡了 —— 必须靠运行期 inner-extent copy 把搬运停在 winm，让 [winm:BI] 落一个可控的值，而不是上一轮遗留在 workspace_s 里的 stale QK。win_align 本身也是运行期表达式（(win + 15) // 16 * 16），今天仍然活着，只是退到了 cube 侧：kernel.py:3620 定义、喂给 QK mma 的 n_actual 和 kernel.py:3785 的 L0C→GM copy 宽度；第 4011 行注释里的「不再需要 win_align 对齐」只对 vector 侧这条 copy 成立。

这里的 -inf mask 其实不是靠 fill 保住的。注释说 [winm:BI] 保持前面 T.tile.fill 打的 -inf，但这一步实际没保住：GM→UB 的窄搬（真实列数 ≠ UB 行距，正是这里的情形）会先拿 pad_value 把整块 UB tile Duplicate 刷一遍，而 pad_value 是 T.copy 的可选参数、默认 0，kernel.py 全文没有一处传过它 —— 空档里落的是 0 而不是 -inf，前面那次 fill 被整块冲掉了。结论仍然成立是靠另一条链：满宽的 T.reduce_max / T.reduce_sum 以及 unary / binary / scalar 算子会被自动收窄到有效区，空档压根不进归约、也不会被逐元素算子破坏。要点：窄搬之后先 fill 一个哨兵值再 copy，这套写法真正生效的不是那次 fill。空档一旦要被某个不会被收窄的满宽读者看见（cmp / select 之类），就必须在 T.copy 上显式传 pad_value。

3.5　用的时候要注意什么

  • 只有最内（列）维能给运行期表达式。行数（次内维）和更外层的维度仍然必须是编译期常量。方向上也只有 GM→UB、UB→GM（含 T.tile.atomic_add 的 UB→GM）三条路通；同一个 T.copy 写成 UB→L1 或 UB→UB，运行期 inner-slice 一样会崩成同样的 undeclared-identifier 报错。copy_gm_to_l1 / copy_l0c_to_gm / copy_l0c_to_ub 那几条不受这条限制（SWA cube 侧 kernel.py:3785 那句 workspace_s[cid, buf, :, 0:win_align] 就是靠这一点）。
  • UB buffer 自己的最内维必须是编译期常量 —— 运行期宽度只能出现在 slice 上界，不能出现在 T.alloc_ub 的 shape 里。把 buffer 声明成动态宽度会在编译阶段直接报一句 copy_gm_to_ub: the inner (column) dimension of the destination buffer shape must be a compile-time constant, but got ...（ub2gm / atomic_add 各有对应措辞的一条），比原来那句莫名其妙的 undeclared identifier 好认。
  • 运行期宽度自己要落在 buffer 里。行间跳过的空档是「buffer 最内维 − 这次搬的列数」，宽度算大了这个差值会变成负数，搬运直接踩进相邻行。tw 是内核自己从 seqused_kv / act_q_lens 算出来的，编译期没人替它做边界检查，clamp 要自己写（kernel.py 里的 T.min(S2_BASE, ...) 就是干这个的）。
  • 空档里的值由 pad_value 决定，不是 UB 里原有的内容。真实列数与 UB 行距不一致时，copy 会先用 pad_value 把整个 tile Duplicate 刷一遍，默认值是 0。想要 -inf mask 就显式传 pad_value=-T.infinity(dtype)，别指望 copy 之前自己 T.tile.fill(-inf) 的内容能活下来。这段 Duplicate 还自带三组 SetFlag·WaitFlag，是笔隐藏开销 —— 窄搬省下的 MTE2 列数不多时，未必划算。
  • Duplicate 有 dtype 白名单：只有 int16_t / uint16_t / half / bfloat16_t / int32_t / uint32_t / float 会被填。int8、int64、fp8 这些同样合法的 copy 元素类型走的是「完全不填空档」的路径 —— 空档里既不是 pad_value 也不保证是原内容，满宽计算之前必须自己保证它。
  • 常量窄 slice 的行为没变，跟运行期窄 slice 语义不同。[:, 0:64] 这种编译期常量宽度仍然按「行距 = 64」处理，等于把 UB 当成按 64 紧密排列；只有运行期宽度才按 buffer 声明的真实行距跳。同一个 buffer 上混用两种写法时，两者的行布局不一样，要留神。
  • 32B 粒度。行间空档换算成块数时是向下取整（(行距 − 有效列数) * sizeof(T) / 32），读写两个方向同型。安全前提是 buffer 最内维的字节数本身是 32 的整数倍（512 个 fp32 = 2048B，成立）—— 此时搬运长度向上取整补的那点正好等于空档被截掉的那点，行距还原成整行。kernel.py 的 copy-out 用 tw_a = (tw + 15) // 16 * 16（fp16 → 32B）是迁就 mma 分形顺带满足这条；自己挑运行期宽度时也照 32B 的倍数挑最省心。

4gemm_v0 / mma 的运行期输出列数 n_actual —— 让 QK 只算真实窗口那几列

Feat 已合入 2026-07-17 PR #1352

一句话：原来 gemm 的输出列数只能是编译期模板 N（只能整块 N 列算完，再在下游把尾巴掩掉），现在可以传一个运行期 n_actual，cube 只计算前 n_actual 列。

4.1　原来写不出来的东西

昇腾 cube 的一次矩阵乘由 m / n / k 三个尺寸描述。gemm_v0 的三个尺寸里，K 早就能接运行期实参 —— 模板内部按 kL0Size 把 K 切成若干 tile，每个 tile 把真实的 K 长度交给 mma。唯独 N 一直是编译期常量：L1→L0B 按编译期 N 列搬分形，mma 的 n 也写死成同一个 N。而稀疏 attention 里，每个 query 块真正要打分的 KV 窗口长度是运行期算出来的 —— SWA 的滑窗长 win 依赖 seqused_kv / act_q_lens，CFA/SCFA 的压缩 tile 按 BI=128 切列块、末块的列宽同样是运行期值。

内核作者想表达的是「这一轮 QK 只算真实窗口那几列」：SWA 里 ori_left = T.max(s_global - ori_win_left, 0)、win = s_global + 1 - ori_left，16 对齐成 win_align；CFA 里 ncols = T.min(BI, tw - cb * BI)、ncols_a = (ncols + 15) // 16 * 16。剩下的 [n_actual:N] 列根本不该进 cube。

会发生什么两种形态。① 直接传列数 → gemm_v0 的签名里根本没有这个关键字，构造 prim_func 时就崩：TypeError: gemm_v0() got an unexpected keyword argument 'n_actual'。② 退回整 N 写法 → 编译能过、跑得动，但 cube 必然算满 128 列，而 [win_align:128] 那段 L0B 装的是这一轮压根没搬进来的内容（SWA 的分页窗口加载带 if pa_done < win 守卫，只填 win 行，其余是上一个 task 遗留在 ring slot 里的 KV），算出来的分数是垃圾值，必须在 softmax 之前额外掩一遍。更硬的一层：mma 的 n 恒等于 N，就没有任何办法让配套的 0b11 融合 fixpipe 只搬窄带 —— fixpipe 的 nSize 一旦和 mma 的 n 不一致，fixpipe 会去等一批从来没有被任何 mma 标记 ready 的 cL0 列，cube 直接挂死。

为什么绕不过去：列数是编译期模板参数。内核侧唯一的「变通」是按每个可能的窗口宽度实例化一套模板 —— 但窗口宽度是运行期值，做不到。剩下就只有整 N 算完再掩：多一轮 vector mask，窗口只有 32 列的极端行（序列最开头）白算 4 倍 cube、L1→L0B 白搬 4 倍数据。而 unitFlag 那条路是死的：0b11 的 mma↔fixpipe 融合要求 fixpipe 的 nSize == mma 的 n，n 被钉死成 N 时「窄带 fixpipe」整个封死，只能退回加屏障的串行写法。

省算力是次要收益。本仓 ori_win_left 取 api.py 的默认值 127，这也是它能取到的最大值（kernel.py 直接拒绝 ori_win_left + 1 > BI），于是 win = min(s_global + 1, 128)：只有序列最前面 127 个因果位置的 win 小于编译期 N=128，8K 序列下约 1.5% 的行才真正收窄。CFA/SCFA 那边 512 宽的 cmp tile 切成 4 个 128 列块，也只有末块可能不满。真正让 n_actual 成为必需品的是 0b11 融合 fixpipe 那条硬约束（见 4.5）。

@T.prim_func
def qk(Q: T.Tensor([1, 64, 128], "float16"),
       K: T.Tensor([1, 128, 128], "float16"),
       S: T.Tensor([1, 64, 128], "float")):
    with T.Kernel(1, is_npu=True) as (cid, vid):
        q_l1 = T.alloc_L1([64, 128], "float16")
        k_l1 = T.alloc_L1([128, 128], "float16")
        l0c = T.alloc_L0C([64, 128], "float")
        T.annotate_layout({
            q_l1: make_zn_layout(q_l1),
            k_l1: make_nz_layout(k_l1),
        })
        T.copy(Q[0, :, :], q_l1[:, :])
        T.copy(K[0, :, :], k_l1[:, :])

        T.copy(l0c, S[0, :, :])

关键只在 n_actual=112 这一行：模板 N 依旧是 l0c 的 128 列（buffer、L0C 排布都不变），但 L1→L0B 只搬 112 列、mma 的 n 也是 112，S[:, :, 112:128] 停留在 L0C 的初值，由调用方自行 mask。把 n_actual=112 删掉就退回整 128 列的老行为，与改动前完全等价。112 = 7×16 必须是 16 的倍数（分形列粒度）；这里写的是编译期常量、写错会当场被拦下报错，实际算子里传的是 (win + 15) // 16 * 16 这种运行期表达式。

4.6　用的时候要注意什么

  • 取值必须满足 0 < n_actual <= N（N 是编译期模板列数），且必须是 16 的倍数（分形列粒度）。编译期常量写错会当场被拦下；运行期表达式不做检查，是调用方的责任 —— 所以 kernel.py 里到处是 (win + 15) // 16 * 16 / (ncols + 15) // 16 * 16 这种手工上取整。
  • 不传 n_actual 就是改动前的老行为（等价于取编译期 N），既有的 T.gemm_v0(A, B, C, transpose_B=True, init=True) 一个字符都不用改。
  • 只在 transpose_B（QK）路径生效。非转置的 gemm_v0（PV 那条）传 n_actual 不会报错，但模板内部拿到的仍是整块的 nTile、L1→L0B 也照 N 搬 —— 想在非转置路径上收窄用的是另一套手段（PV 用 real_k / k_actual 限 K，N 保持编译期宽度）。
  • [n_actual:N] 那段 cL0 列永远不会被写，停在 L0C 的初值或上一轮残留。下游必须自己 mask / 回避，绝不能当成 0 来用。
  • 配 unit_flag=0b11 用时，随后那个融合 fixpipe（T.copy(cL0 -> GM, unit_flag=0b11)）的 nSize 必须等于 mma 的 n_actual。不一致的话 fixpipe 会去等一批从没被任何 mma 标记 ready 的 cL0 列，cube 挂死（不是算错，是卡住）。
  • 单独改 mma 的 n 不够。L0B 的 nZ 分形从列数推 K-block stride，L1→L0B 的搬运必须同步带 real_n=<同一个值>（T.copy(..., transpose=True, real_k=..., real_n=...)），否则连算出来的那几列都是错的。gemm_v0 内部已经做了这一步；用解构的 T.mma 时要自己配对写。
  • 别把 n_actual 写成 T.if_then_else 之类会 lower 成语句的表达式，会生成非法 C++、编不过。kernel.py 因此把 4 个 mma 逐条展开而不是写成 kk 循环。

5exp_experiment：带行 stride 的掩码 exp —— 在物理宽 512 的分数缓冲里，原地只 exp 窗口那 64 列

Feat 已合入 2026-07-17 PR #1399

一句话：原来 T.tile.exp 只有「从切片起点连续数 N 个元素」这一种语义，要在 [16, 512] 的分数缓冲上只 exp 每行前 64 列，就得逐行发 16 条 Exp；现在一条 T.tile.exp_experiment 覆盖所有行，行与行之间按 buffer 声明里的物理列数自动 stride。

5.1　原来写不出来的东西

在线 softmax 的分数缓冲在 kernel.py 里是 in_ub = T.alloc_ub([2, M_CHUNK, S2_BASE], accum_dtype)，物理宽度写死 S2_BASE = 512：宽 cmp tile 要用满这 512 列，而 cast、copy-out、p_l1 的行距也全按 512 算。但单个 tile 的真实有效列数 tw 常常只有滑窗那点宽度（tw <= BI = 128），剩下几百列是 padding。

昇腾的向量指令天生有 mask + repeat + repStride 三件套：一次 repeat 处理 64 个 fp32（= 8 个 32B block），repeat 之间按 repStride 个 block 往前跳。硬件本来就能表达「每行只算前 64 列、行间跳 512」—— 二元侧的 row_expand_sub_experiment / row_expand_div_experiment 一族早就在这么用，缺的只是一元 exp 的对应写法。

内核作者想表达的是：在 [M_CHUNK, 512] 的分数缓冲上原地做 P = exp(S - rowmax)，但只碰前 tw_64 列（64 对齐后的有效窗口），既不整块算 512 列，也不把有效列压实到临时 tile。

改动前会这么写
# 写法 A：整块 exp —— 编得过、算得也对（padding 是 -inf，exp 后为 0），但纯浪费
T.tile.exp(in_ub[ps, :, :], in_ub[ps, :, :])

# 写法 B：直接切窄窗口 —— 也编得过，但答案是错的
for kc in range(BI // 64):
    if kc * 64 < tw_64:
        T.tile.exp(
            in_ub[ps, :, kc * 64 : kc * 64 + 64],
            in_ub[ps, :, kc * 64 : kc * 64 + 64],
        )

# 写法 C：压实到窄 tile 再 exp 再搬回 —— 多一块 UB + 两次搬运，in-place 拆成三步
T.copy(in_ub[ps, :, 0:tw_64], narrow_ub)
T.tile.exp(narrow_ub, narrow_ub)
T.copy(narrow_ub, in_ub[ps, :, 0:tw_64])

# 写法 D：逐行发 —— 语义正确，但一个 64 列 chunk 就是 M_CHUNK 条 Exp
for i in range(M_CHUNK):
    T.tile.exp(
        in_ub[ps, i, kc * 64 : kc * 64 + 64],
        in_ub[ps, i, kc * 64 : kc * 64 + 64],
    )
会发生什么写法 B 是静默算错：不报错、不挂、编译产物看起来完全正常。T.tile.exp 只认「元素总数」这一个量 —— in_ub[ps, :, kc*64 : kc*64+64] 的 extent 是 [1, 16, 64]，被拍平成 1024，发出去的是 AscendC::Exp(dst, src, 1024)。这是连续 count 语义，从切片起点一路往后数 1024 个 fp32。in_ub 的物理行距是 512，1024 / 512 恰好是整整两行：kc = 0 那次把第 0、1 行整行（窗口连同 padding）exp 掉，第 2~15 行一个元素都没碰。tw_64 = 128 要发第二块时（起点偏移 64、同样 1024 个元素），第 0、1 行已经 exp 过的部分被再 exp 一次成 exp(exp(x))，还溢进第 2 行的前 64 列。净效果是第 2 行起的「P」停在 score - rowmax 这个负数上，紧接着的 T.reduce_sum 把负数累进 denom —— denom 和 P 同时错，vs golden 的 mismatch 既不像 race 也不像 QK 的锅。覆盖到的行数随 dtype / chunk 变（= 切片 extent / 物理列数），fp16 的 128 列 chunk 是前 4 行。

为什么绕不过去：分数缓冲的物理宽度不能改窄 —— 同一块 in_ub 还要给宽 cmp 路径用（tw 最大就是 512），cast、copy-out、p_l1 的行距也全按 512 算，声明成 [M, 64] 整条链都得跟着改。于是内核侧能写对的只剩三条：整块算 512 列（窄窗时白算 8 倍的超越函数）；压实到临时 tile（多一块 UB、两次搬运，还把原地操作拆成三步、要补跨 pipe 握手，V 流水更长）；或者写法 D 逐行发 —— 每行一条 AscendC::Exp(dst, src, 64)，语义完全正确，代价是一个 64 列 chunk 要发 M_CHUNK = 16 条 Exp，再乘上 NMC 个 m-chunk、每个 tile、每个 task，而且 Python 层的行循环会被展开，生成的 C++ 也跟着膨胀 16 倍。exp_experiment 把同一件事压成 1 条指令：16 个 repeat，跨行的偏移交给硬件的 repStride。

5.2　现在可以这么写

签名只有两个位置参数：T.tile.exp_experiment(dst, src)。count、mask、行 stride 全部由 buffer 的声明形状和切片 extent 推出来，调用点不填、也没法填。

  • dst —— 写入目标，形如 buf[..., :, c0 : c0 + 64] 的切片。最后一维必须正好是 64（fp32）/ 128（fp16），也就是一次 repeat 的满额 lane 数；切片一共几行，就发几个 repeat。
  • src —— 输入切片，形状必须与 dst 完全相同，dtype 相同，且只支持 fp16 / fp32。dst 可以就是 src（原地），kernel.py 用的正是这种写法。
  • 行间跳距不出现在签名里，它从 dst 所属 buffer 声明的最后一维换算：声明 [2, 16, 512] 的 fp32 buffer，行距就是 512 个元素 = 64 个 32B block，每个 repeat 往前跳 64 个 block。同一段窗口逻辑挪到物理宽度不同的 buffer 上，stride 自动跟着变，调用点一个字都不用改。
  • mask 写死全 1（fp32：低 64 lane 全开；fp16：128 lane 全开），没有参数能窄化列。窄化完全来自「repeat 次数 × 行间跳距」这一维，不是列掩码。
最小例子
M, N = 16, 512   # 分数缓冲的物理宽度写死 512
CH = 64          # fp32 一次调用正好 64 列(fp16 是 128)
VALID = 40       # 本 tile 真实有效列数,故意不是 64 的整数倍

@T.prim_func
def main(A: T.Tensor((M, N), "float"), C: T.Tensor((M, N), "float")):

        brcb = T.alloc_ub([M, 8], "float")

        T.tile.fill(a_ub, -T.infinity("float"))
        T.copy(A[:, 0:VALID], a_ub[:, 0:VALID])
        # rowmax 走满宽:padding 是 -inf,不影响 max
        T.reduce_max(a_ub, row_max, dim=-1)

        T.tile.row_expand_sub_experiment(a_ub[:, 0:CH], a_ub[:, 0:CH], brcb)
        T.tile.exp_experiment(a_ub[:, 0:CH], a_ub[:, 0:CH])
        T.copy(a_ub, C)   # [64:512] 仍是 fill 写下的 -inf

一段手拼的窄窗口 softmax：缓冲物理宽 512，真实有效只有 40 列。关键在最后那行 T.tile.exp_experiment —— 它对 16 行各发一个 repeat（每 repeat 64 个 fp32），行间按 buffer 声明的 512 列 stride 跳，一次调用就把 16 行的窗口全 exp 了，[64:512] 那 448 列一个字节不动。同一行换成 T.tile.exp(a_ub[:, 0:CH], a_ub[:, 0:CH]) 编译照样过，发出的却是 AscendC::Exp(dst, src, 1024) 连续 count，只覆盖前 2 行、其余 14 行原封不动。VALID = 40 是刻意的：exp_experiment 的 mask 传的是全 1（窄化靠 repeat 次数 × repStride，不是列掩码），所以 [40:64] 这段对齐 slop 一定会被 exp —— 开头「满宽 fill(-inf) → 只搬有效列」的顺序就是为了让 slop 保持 -inf，exp 后是 0，无害。反过来写成先 copy 满宽再 fill，fill 就成了死语句。

T.reduce_max 这里走的是满宽。窄列切片上的行归约（T.reduce_max(a_ub[:, 0:CH], ...) 按物理行宽 512 跨行）是另一件事，属于 #1422 的范围（本文第 8 节）；在只有 #1399、没有 #1422 的编译器上那么写，会拿第 0 行的若干块当成 16 行的 rowmax，静默算错。exp_experiment 本身不依赖它。另外这个例子里 fill 跑在 PIPE_V、copy-in 跑在 MTE2，两者之间的握手要么靠 pass_configs 打开 TL_ASCEND_AUTO_SYNC 自动插，要么自己补 T.set_flag("v", "mte2", ev) / T.wait_flag(...)（kernel.py:1184-1185 那对）。

这十行是整段 strided softmax 的「合同」，直接决定 exp_experiment 用得对不对。tw 是本 tile 的真实有效列数；tw_a 是 16 对齐（cube 侧 PV 读 P 的宽度）；tw_64 是 64 对齐 —— 因为 exp_experiment / row_expand_sub_experiment 每次调用正好吃 64 列（fp32 一次 repeat 的上限），窗口必须先补到 64 的整数倍才能整块喂。mask 传的是全 1，窄化完全来自 repeat 次数 × repStride，所以 [tw, tw_64) 这段对齐 slop 一定会被算进去；kernel.py 的办法是整块先 T.tile.fill(in_ub[ps, :, :], -T.infinity(accum_dtype))、再只 copy 有效的 [0:tw] 列（kernel.py:1180-1197），slop 保持 -inf —— reduce_max 忽略它、exp 把它变成 0、reduce_sum 加 0，全链无害。tw_a <= tw_64 保证 copy-out 的那 tw_a 列全落在算过的范围内。最后一行 is_narrow = tw <= BI 是分路开关：按实际宽度而不是按 ori/cmp 语义分，窄的走 strided（省 padding），宽的走 T.tile.broadcast + T.tile.sub + T.tile.exp 满宽（kernel.py:1323-1334，没 padding 可省时一条宽 op 更划算）。

同文件 1267-1269 还有一句「ori: strided …；cmp: wide broadcast+sub+exp(满 512)」，与第 8 行那句「宽窗(tw>BI,只 scfa 的宽 cmp)」看起来矛盾 —— 两句都是简写，真正的分路条件只有 tw <= BI 这一个运行时比较，跟 ori / cmp 的语义无关，同一段 cmp 走哪条完全由 cmp_ratio 和 topk 决定。本仓的两组配置正好各占一边：CFA 用 cmp_ratio = 128（test_sparse_attn_sharedkv.py:191、214），8K 下 act_cmp = 8192 // 128 = 64、cmp_tiles = 1，那唯一一个 cmp tile 宽 64 ≤ BI，走的是窄路；SCFA 用 cmp_ratio = 4 加 topk_cmp = K = 512 截断（test_sparse_attn_sharedkv.py:114、122），cmp tile 满 512 宽，才是宽路的实际来源。所以 exp_experiment 这条窄路上跑的不只是 ori 滑窗，还有 CFA 的整段 cmp。

5.5　用的时候要注意什么

  • 最后一维必须正好是 64（fp32）/ 128（fp16），不是「不超过」。96 列的窗口不能一次调完，要么向上对齐到 128 分两块，要么自己处理尾巴。kernel.py 的做法是先 tw_64 = (tw + 63) // 64 * 64 再按 64 分块。
  • mask 传的是全 1（fp32：mask0 全 1、mask1 = 0；fp16：两个都全 1），每次 repeat 的 64 / 128 个 lane 全算。窄化来自 repeat 次数 × repStride，不是列掩码。[tw, tw_64) 这段 slop 必然被 exp，调用方要自己保证那里的值无害；若 slop 里是垃圾数，exp 会炸成 inf/NaN 再被 reduce_sum 吃进分母。
  • dst 与 src 的声明物理列数必须相同。行间跳距只从 dst 的 buffer 声明推，然后同一个值同时当作 dst 和 src 的行距；形状校验只比切片 extent，不看物理宽度。所以 T.tile.exp_experiment(narrow[:, 0:64], wide[ps, :, 0:64])（narrow 声明 [16, 64]、wide 声明 [2, 16, 512]）编译期一路全过，src 却会按 8 个 block 而不是 64 个 block 跨行读 —— 静默读错行。kernel.py 里 dst 就是 src（原地），所以安全。
  • 行 stride 读的是静态声明，buffer 必须静态形状；extent 不是编译期常量会直接挂在 ExpExperimentCodegen: dynamic extent not supported。反过来说，同一段窗口逻辑换一个物理宽度不同的 buffer，stride 会自动跟着变，调用点不用改。
  • 指令里的 repeat 次数与行间跳距都只有 8 位：行数 ≤ 255，物理列数 ≤ 255 × 一个 32B block 的元素数（fp32 ≤ 2040、fp16 ≤ 4080）。这两条没有检查，超了是静默截断。kernel.py 的 M_CHUNK = 16、512 列（行距 64 个 block）远在范围内。
  • 整除性是硬约束：物理列数必须被一个 32B block 的元素数（fp32 8 / fp16 16）整除，切片 extent 必须是 fp32 64 / fp16 128 的整数倍，否则编译期直接挂，错误里会打出 extent 和 cols 的实际值。
  • 窗口宽了就别用：每 64 列一次调用，tw 接近 512 时是 8 次 strided 调用 vs 1 次宽 exp，反而亏。kernel.py 用 is_narrow = tw <= BI（BI = 128）分路，宽 cmp tile 走满宽写法。
  • 它跑在 PIPE_V，和串在一起的 row_expand_sub_experiment、reduce_max / reduce_sum 同一条流水，V 内的先后由 pipe 语义保证；但跟 MTE2（copy-in）/ MTE3（copy-out）之间的 T.set_flag / T.wait_flag 握手仍然要自己写 —— 这个原语不替调用方补跨 pipe 的围栏。
  • 纯新增、不会自动生效：老的 T.tile.exp 生成的指令一行没变，也没有任何自动改写会把它换成 exp_experiment。升级编译器不会让既有 kernel 的行为悄悄改，反过来要用就必须显式改调用点。
  • 名字里的 experiment 标的是上游的实验性接口 —— 与 row_expand_sub_experiment / brcb_experiment 同一族，参数与命名后续仍有可能调整，调用点最好收在少数几处，别散在整个 kernel 里。

6让内核自己驱动 L1→L0→mma→fixpipe：real_k / real_n / k_actual / n_actual / unit_flag 五件套

Feat + Bugfix 待合入 PR #1422

一句话：原来 L1→L0→mma 这三步只能整个交给 T.gemm_v0 一条辅助函数，内核自己发的 T.mma 里 K 和 N 都只能是编译期值，调用方另发的那条 fixpipe 跟 mma 之间只有软件握手；现在内核可以自己逐块载 L0、自己发累加型 mma，K 和 N 都可以是运行期值，最后一块到齐时用 unit_flag 与那条 fixpipe 做硬件配对。

6.1　原来写不出来的东西

昇腾 cube 的一次矩阵乘走四步：L1→L0A/L0B 按分形载入操作数（走 PIPE_MTE1）、Mmad 计算（走 PIPE_M）、结果落在 L0C 累加器、再由 fixpipe 把 L0C 搬到 GM。TileLang 把前三步封在 T.gemm_v0 里；第四步不在里面——T.gemm_v0 算完把结果留在调用方的 L0C，fixpipe 一直是调用方自己另发的一条 T.copy(L0C→GM)。

硬件另有一个 unitFlag 字段，可以把一条 mma 和它后面那条 fixpipe 串成硬件流水：0b10 表示结果先留在 L0C，0b11 表示这个 L0C 槽算完了、释放给配对的 fixpipe。以前内核发不出这个字段。而当内核需要自己维护 KV 环形缓冲、或者把 contraction 拆成不同时刻才到齐的若干块时，L1→L0→mma 这三步必须由内核自己走完，并且要能和那条 fixpipe 配上对——否则每一块都得回到 T.gemm_v0 里跑一遍完整序列。

以 SWA 的 PV 阶段为例：内核自己维护一个 3 槽 KV 环，V 的每个 D-tile 是逐页 gather 进来的，到齐一块就想立刻发一条 mma 把它算掉，算完直接用配对的 fixpipe 把 L0C 冲进 workspace。而且 contraction 长度 winm（真实窗口行数）是运行期值，不是 L0 缓冲的编译期宽度 BI=128。

改动前会这么写

为什么绕不过去：退回 T.gemm_v0 是唯一能算对的写法，但它有两个内核侧绕不过去的代价。第一，T.gemm_v0 每次调用的首尾各有两对「SetFlag(e) 紧跟同一个 event 的 WaitFlag(e)」的自配对围栏：开头是 MTE2_MTE1 和 FIX_M，结尾是 MTE1_MTE2 和 M_FIX，四对用的还是同一个 event 号。自配对不是在给某个 buffer 排序，而是 drain 整条 pipe：每次 gemm 都强迫 MTE2 全部做完 MTE1 才能开始，并把下一页 KV 的搬运卡到当前 L1→L0 结束之后。这恰好就是内核那个 3 槽 KV 环存在的意义，等于把 overlap 全废掉。这是内核作者选择把这条序列拆开自己驱动的唯一理由。

第二，contraction 长度 winm 是运行期值，T.gemm_v0 的 K 只能来自编译期常量或 A 的最后一维，切窗口这件事在内核侧无法表达。

6.2　现在可以这么写

最小例子
M, N, K = 64, 128, 256
KL0 = 128  # kL0 tile 宽度：K 拆成 2 块

@T.prim_func
def main(A: T.Tensor([M, K], "float16"),
         B: T.Tensor([N, K], "float16"),   # K^T 布局，算的是 A @ B^T
         nact: T.Tensor([1], "int32"),     # 运行期输出列数
         C: T.Tensor([M, N], "float")):
    with T.Kernel(1, is_npu=True) as (cid, _):
        a_l1 = T.alloc_L1([M, K], "float16")
        b_l1 = T.alloc_L1([N, K], "float16")
        a_l0 = T.alloc_L0A([2, M, KL0], "float16")   # 两个 ping-pong 槽
        b_l0 = T.alloc_L0B([2, N, KL0], "float16")
        c_l0 = T.alloc_L0C([1, M, N], "float")
        with T.Scope("C"):
            n_act = nact[0]
            T.barrier_all()
            T.copy(A, a_l1)
            T.copy(B, b_l1)
            T.barrier_all()

            T.copy(a_l1[:, 0:KL0], a_l0[0, :, :])
            T.copy(b_l1[:, 0:KL0], b_l0[0, :, :],
                   transpose=True, real_k=KL0, real_n=n_act)
            T.barrier_all()

                  init=True, k_actual=KL0, n_actual=n_act, unit_flag=0b10)
            T.barrier_all()

            T.copy(a_l1[:, KL0 : 2 * KL0], a_l0[1, :, :])
            T.copy(b_l1[:, KL0 : 2 * KL0], b_l0[1, :, :],
                   transpose=True, real_k=KL0, real_n=n_act)
            T.barrier_all()
            T.mma(a_l0[1, :, :], b_l0[1, :, :], c_l0[0, :, :],
                  init=False, k_actual=KL0, n_actual=n_act, unit_flag=0b11)
            T.barrier_all()
            # 配对的那条 fixpipe：列数必须 == mma 的 n_actual，
            # 且这里刻意不加同步——被顶掉的正是软件 M_FIX 握手
            T.copy(c_l0[0, :, :], C[:, 0:n_act], unit_flag=0b11)
            T.barrier_all()

真正「删掉就错」的是三处：B 侧两条 real_n=n_act 让 L0B 按 mma 真正会读的形状载入分形（少了它，nZ 分形的 K-block stride 会按缓冲全宽算，第二块 K 就寻址到错误的 K-block）；两条 mma 的 init=True/False 把 K 累加进同一个 L0C 槽；unit_flag 从 0b10 变成 0b11 标出「最后一块到齐了」，最后那条 T.copy(..., unit_flag=0b11) 就是与它配对的 fixpipe。n_act 从一个 int32 输入张量取，是运行期值，编译期常量蒙不过去。拆两块 K 也不是凑数：只有一块时它的 K-block 偏移恰好是 0，real_n 写错也照样算对，第二块才会暴露寻址到错误的 K-block。

五个参数各自的位置和职责——载入和 mma 必须用同一套分形几何，real_k 与 k_actual、real_n 与 n_actual 必须成对相等，不等会寻址到别的块上而且不报错：

参数给谁是什么什么时候必须传
real_kT.copy（L1→L0A / L1→L0B）这块 L0 分形按多少 K 排布：A 侧覆写末维，B 侧覆写首维contraction 长度是运行期值、或者短于 L0 缓冲的编译期宽度时
real_nT.copy（L1→L0B）L0B 分形按多少 N 排布，也就是 nZ 分形拿来推 K-block stride 的那个列数输出列数是运行期值、或者短于 L0B 缓冲宽度时；A 侧没有 N 这根轴，给了也无效
k_actualT.mma这条 mma 真正 contract 多少 K，顶替由 A 最后一维推出来的值同 real_k，并且必须与 real_k 相等
n_actualT.mma这条 mma 真正算多少列，顶替编译期推出来的 N同 real_n，并且必须与 real_n 相等；配对 fixpipe 的列数也要等于它
unit_flagT.mma 和配对的 T.copy（L0C→GM）0b10 = 结果留在 L0C；0b11 = 冲刷这个 L0C 槽并与配对的 fixpipe 融合想让 mma→fixpipe 走硬件配对时；一串 K 累加里只有最后一条给 0b11，前面全给 0b10
这个例子里的 real_k / k_actual 是恒等写法。A 侧 K 本来就是 a_l0 的末维 KL0；B 侧 real_k 覆写的是 L0B 的首维，而在这个 A @ B^T 的布局里那一维就是 N=128=KL0。删掉这两个参数，发出的代码逐字节不变。运行期 K 的真实用例见 6.5（三）。另外这个布局有个隐含约束：T.mma 要求 A 侧的 K 与 B 侧的 K 相等，而 A 侧 K 取形状末维、B 侧取形状倒数第二维，所以 b_l0 声明成 [2, N, KL0] 时必须 N == KL0，非方形要改 L0B 的声明顺序。
例子里每个流水边界的 T.barrier_all() 都是显式写出来的。L1→L0 的 copy 在 PIPE_MTE1 上、mma 在 PIPE_M 上，两条流水之间没有同步就是 L0A/L0B 的 RAW 竞态。自动同步默认是关的，kernel.py 走的正是默认关闭这条路，同步全由内核自己发。要保住 overlap 就把全屏障换成定向 flag（T.set_flag("mte1", "m", EV) / T.wait_flag("mte1", "m", EV)），也就是 6.3–6.5 里 kernel.py 的写法。唯独最后那条 mma 与配对 fixpipe 之间不要加同步——那正是 0b11 要顶掉的握手。

B 侧（K 的转置）必须给 real_n=win_align，因为窗口宽度 win_align 是运行期算出来的（win 向上取整到 16），而 v_l0b 缓冲是编译期的 BI=128 全宽——L0B 的 nZ 分形从列数推它的 K-block stride，全宽载入配一条 n=win_align 的 mma 会寻址到错误的 K-block。A 侧连 real_k 都不用给：p_l0a 的 K 维本来就等于 KL0，kernel.py 3650-3651 行的注释原话是 real_k is unnecessary on the A side (p_l0a's K dim already IS kL0Size)。

四块 kL0 tile 里的最后一块。前三块是 (T,0b10)(F,0b10)(F,0b10)，这块是 (F,0b11)：init=False 继续往同一个 cL0 槽累加，unit_flag=0b11 告诉硬件「这个槽算完了，放给配对的 fixpipe」。紧接着那条 T.copy(cL0 → workspace_s, unit_flag=0b11) 就是被配对的 fixpipe，对应 Ascend C 参考实现里两个 D-half 之后的那一条 Fixpipe。

它的列数 0:win_align 必须和 mma 的 n_actual 严格相等，否则这条带 unit_flag 的 fixpipe 会去等一批没有任何 mma 标记就绪的 L0C 列。这一对 0b11 就是 mma→fixpipe 硬件流水的入口：不需要软件的 M_FIX / FIX_M 握手，tile i 的 fixpipe 就能和 tile i+1 的 mma 在两槽 cL0 上乒乓重叠。

中间那条 if win_align < 40: T.pipe_barrier("m") 复现的是 Ascend C 参考实现按运行期 n 门控的 PIPE_M 屏障（(m/16)*(n/16) < 10，M=G=64 时即 win_align <= 32）。TileLang 这边没有等价物：T.gemm_v0 是每个 tile 无条件发一次 PIPE_M 屏障，而单发 T.mma 一次都不发。解构到内核之后这个门得内核自己复现。

另外这四块是逐条展开写的，而不是 kk 循环——因为最后一块的 unit_flag 跟前三块不一样，而 T.mma 的实参里不能放 T.if_then_else（见 6.6）。

6.6　用的时候要注意什么

  • 载入和 mma 必须用同一套分形几何：real_k == k_actual、real_n == n_actual。不等不会报错，会静默算错：第一个 M-block（第 0..15 行）因为偏移是 0 恰好正确，16 行往后全错，调试时很容易误判成竞态。两者的失败签名还不一样，别混——real_k 不匹配对应「寻址到错误的 M-block」（L0A 的 M-block stride 依赖 K），real_n 不匹配对应「寻址到错误的 K-block」（L0B 的 K-block stride 依赖 N）。
  • real_n 只对 B 侧（L0B）生效——A 侧是 [M, K]，没有 N 这根轴，给了会被忽略。反过来 real_k 对 A / B 都生效，但覆写的维不同（A 侧覆写末维，B 侧覆写首维）。
  • 配对的那条 fixpipe 的列数必须等于 mma 的 n_actual。否则带 unit_flag 的 fixpipe 会去等一批没有任何 mma 标记就绪的 L0C 列，表现为挂死而不是错数。
  • T.mma 的实参里不能放 T.if_then_else：它会被 lower 成语句，插进半截调用中间，直接编译失败。所以 unit_flag 不同的那几条 mma 必须逐条展开写，不能写成 kk 循环加条件；编译期分支一律展开成字面量。
  • L1 切片的偏移不需要额外加 layout 标注：L1→L0 的搬运本来就走分形寻址，写 a_l1[:, KL0 : 2 * KL0] 这样的切片就够了，不用再去声明行优先或别的排布。
  • 解构掉 T.gemm_v0 换来的是 pipe overlap，但也把 L0AB 的 flag 生命周期整个交给了内核：("mte1", "m") 那对（载入完成→允许 mma）和 ("m", "mte1") 那对（mma 用完→允许下一轮覆写这个 L0AB 槽），每一块 tile 前后都要自己配好，漏一条就是竞态。
  • 小 tile 有 hazard：mma 的 (M/16)*(n/16) 太小时（参考实现的门限是 < 10）要自己补一条 T.pipe_barrier("m")。T.gemm_v0 每个 tile 无条件发一次，单发 T.mma 一次都不发。
  • 最后那条 mma 与它配对的 fixpipe 之间不要加同步——那正是 0b11 要顶掉的软件握手，加了就把硬件配对变回串行。
  • 这五个参数全部可选，不给就等于没这回事，旧的 T.copy / T.mma 写法发出的代码逐字节不变。但它们是位置参数：只给 real_n 不给 real_k 时，real_k 会被填成默认值（＝从缓冲形状取）；只给 unit_flag 不给 n_actual 时，n_actual 会被填成编译期的 N。想只改外层那个，也得清楚内层被填了什么。

7多维 buffer 的标量下标要跨维展平 —— block_table[b, lg] 不再别名到第 0 行

Bugfix 待合入 PR #1422

一句话：原来对多维 buffer 做标量读写只发最内层下标、前导维被整个丢掉（table[b, i] 恒等于 table[0, i]，静默算错且 B=1 时完全正常），现在按 buffer 形状跨全维展平成一个线性偏移，ori_block_table[b, pa_lg] 这类「按批次取物理页号」才真正取到第 b 行。

7.1　原来写不出来的东西

昇腾侧生成出来的 Ascend C 里，GM / UB 上的 buffer 只有线性下标的标量接口 —— GetValue(i) 取一个数进标量寄存器、SetValue(i, v) 写回去，没有多维下标的重载。所以 TileLang 里写 buf[i, j] 这种「取一个标量」的访问，多维下标必须先折成一个线性偏移。整块搬运不走这条路：T.copy 的 dstM / dstN / stride 是当参数显式发给搬运指令的，多维信息不经过下标；只有标量访问依赖下标展平。

分页 KV（PagedAttention）的页表是二维的 [batch, num_logical_blocks] —— 第 b 行是第 b 个 batch 的逻辑块号到物理块号的映射。内核里要按当前 batch 取物理页号：先算逻辑块号 pa_lg = pa_cur // ori_block_size，再 pa_ph = ori_block_table[b, pa_lg]，然后拿 pa_ph 去 ori_kv[pa_ph, ...] 做真正的 GM→L1 搬运。这是分页 KV 最基本的一步，当前 main 上三个 kernel 合计 9 处页表查表，外加 SCFA 一处三维的 cmp_indices 标量读，共 10 处多维标量访问。

改动前会这么写
# 页表按 vLLM/FlashAttention 的约定就是二维 [B, max_blocks]
ori_block_table: T.Tensor([batch, ori_table_len], idx_dtype)

...

# 改动前发出来的 Ascend C 是：
#   ori_block_table.GetValue(pa_lg);          // b 不见了
# 应该是：
#   ori_block_table.GetValue(b * ori_table_len + pa_lg);
会发生什么编译不报错、运行不挂，纯静默错。标量读只打印最内层下标，前导的 b 被整条丢掉，于是 ori_block_table[b, pa_lg] 发成 ori_block_table.GetValue(pa_lg) —— 不管 b 是几，读回来的都是页表第 0 行的物理页号。B=1 时 b 恒为 0，block_table[0, lg] 与 block_table[lg] 读的就是同一处内存，完全正常；B=2 时 batch 1 的每一次 KV 搬运都去搬 batch 0 的物理页，Output / LSE 整批对不上 golden（不是个别 token 偏差，是这个 batch 全错）。标量写同理：buf[b, i] = v 只把 v 写进 buf.SetValue(i, v)，B 行互相覆盖到第 0 行。这条从第一版 SWA kernel 起就一直可达，直到回归里补上一个 B=2 场景才露头。

为什么绕不过去：第一层是「根本不知道要绕」—— 错得完全静默，而算子仓两套测试（test_sparse_attn_sharedkv.py / test_sparse_attn_sharedkv_fast.py）当时所有场景都是 B=1，b 恒为 0，写对写错逐字节等价，全绿。第二层是「知道了也别扭」：下标个数必须跟 buffer 维度一致，内核里不能直接写 ori_block_table[b * ori_table_len + pa_lg] 把二维下标自己算成一维；要手工展平只能把内核签名从 T.Tensor([batch, ori_table_len]) 改成一维，再让 api.py 把外部传进来的 [B, max_blocks] 页表 .view(-1) —— 而 [B, max_blocks] 是 vLLM/FA 那套 block_table 的公开约定，api.py 的入参就是它，改成一维等于把展平责任推给调用方；9 处调用点都要手写 stride 乘法，SCFA 的 cmp_indices 还是三维 [total_tokens, N2, topk_cmp]（kernel.py:1835），手写展平后基本不可读。UB 上的临时 buffer 更没有 host 侧 view 可用。

7.2　现在可以这么写

没有新原语、没有新参数、内核签名也不用改 —— 写法就是原来那个写法。对任意维度的 GM / UB buffer，x = buf[i0, i1, ..., ik]（标量读，发 GetValue）和 buf[i0, i1, ..., ik] = v（标量写，发 SetValue）现在按 buffer 形状跨全维展平，等价于 i0 * s0 + i1 * s1 + ... + ik（si 由 buffer 形状推出），读写两侧行为一致。

最小例子

7.4　用的时候要注意什么

  • 什么形状的下标会触发。只有「标量」访问受影响：x = buf[i0, i1, ...]（取一个数进标量寄存器）和 buf[i0, i1, ...] = v，且 buffer 至少两维。带冒号的切片写法 buf[i, a : a + n, :] 走 T.copy 那条路，从来没错过。一维标量取值也一直是对的 —— act_q_lens[b]（kernel.py:469）、seqused_kv[b]（471）、q_prefix[b]（499）、act_q_lens[bm]（874）、seqused_kv[bm]（875）、q_prefix[v0b]（2631）生成的 Ascend C 一个字符都不变。自查范围就是「多维 buffer 上不带冒号的下标」。
  • 单 batch 测不出来。前导维恒为 0 时新旧数值完全一样（GetValue(lg) 与 GetValue(0 * L + lg) 等价），别拿 B=1 的回归当作「这条跟我无关」的证据。要验证必须构造前导维非 0 的场景，而且让不同前导维的数据真正不同：页号先做一次全局 permutation 再按 batch 切片，两个 batch 拿到的物理页集合不相交；每个 batch 用同一套页时，读串了也看不出来。
  • 本仓受影响的站点。SWA 两处 ori 页表查表（kernel.py:3589 / 3836）；CFA 在 cube 侧碰 ori + cmp 两张表共四处（560 / 607 / 928 / 975）；SCFA 在 cube 侧碰 ori 表两处（2094 / 2441），cmp 表只从 vector 侧的 topk gather 碰一次（2694），并在同一段里多踩一次三维的 cmp_indices（2689）。
  • UB buffer 的行宽要留意。展平用的是 buffer 的逻辑形状，而 UB 上真正分配出来的行宽会按 32B 向上对齐。逻辑行宽是 32B 整数倍时两者一致，buf[i, j] 展平出来的 i * N + j 落点没问题；不是整数倍的多维 UB 标量下标要自己核一遍生成的 Ascend C。GM buffer 没有这层对齐，不受影响。
  • T.alloc_var 不算 buffer。它拿到的是标量寄存器，直接按变量名使用、压根没有下标，既不受这条影响也享受不到它。要做多维标量索引必须是真 buffer：GM 参数，或 T.alloc_ub / T.alloc_L1 之类。
  • 多出一次标量乘加。展平后每次多维标量访问都要算 i0 * s0 + ...，落在标量流水上。每搬一页才查一次表可以忽略；放在最内层循环里高频取表就要算上这笔开销 —— 原来那份「便宜」是错的。

8窄行归约按物理行宽 stride —— 只归约宽 tile 里的一段列

Bugfix 待合入 PR #1422

一句话：原来对一块 [M, 512] 的 UB tile 做「每行只归约其中 64 列」会静默算错（只有第 0 行是对的），现在直接写切片 ub[:, a:b] 或 real_shape=[M, 64] 就是对的。

8.1　原来写不出来的东西

昇腾 vector 单元的行归约有两种组织方式。一种把源当成一块连续的 M×N 数据顺序读完，这种语义里根本没有「行间步长」这回事；另一种按 repeat 组织 —— 每个 repeat 处理一段（最多 256 字节），相邻两个 repeat 之间跳多远，由一个显式的行间步长（以 32 字节 block 计）说明。只要归约的逻辑宽度就是 buffer 的整行宽，两者完全等价；一旦逻辑宽度只是宽行里的一段，前一种就表达不出来 —— 而它不会报错。

内核作者想表达的是：对一块 [16, 512] 的 fp32 UB tile，只归约每行的一段列（[0:64] 或 [64:128]），拿到 [16] 的每行结果。在线 softmax 的 rowmax / rowsum 只对窗口内的有效列有意义，宽 tile 里剩下的是 padding。

改动前会这么写
ub = T.alloc_ub([16, 512], "float")   # 物理行宽 512
acc = T.alloc_ub([16], "float")

# 写法 A: 用切片说明列范围
T.reduce_max(ub[:, 64:128], acc, dim=-1)

# 写法 B: 整块进, 用 real_shape 说明逻辑宽度
T.reduce_max(ub, acc, dim=-1, real_shape=[16, 64])

# 唯一能跑对的绕法: deprecated 的 wholereduce, 行间步长手填
# (物理行宽 512 fp32 = 2048B = 64 个 32B block)
T.tile.wholereducemax(
    acc, ub[:, 64:128], 64, 16, 1, 1, 64,
    ReduceOrder="ORDER_ONLY_VALUE",
)
会发生什么编译通过、运行不报错、结果静默错。写法 A 和 B 都落到「把源当成连续 M×N 块」的那条归约上：写法 B 从 ub 的起点开始，顺序读 flat 元素 [0, 16*64)，也就是第 0 行整行 512 个 + 第 1 行整行 512 个，然后把这 16 个 64 列 chunk 当成 16 行的结果吐出来 —— out[0] 是第 0 行的第一个 chunk（恰好对），out[1..7] 是第 0 行其余 7 个 chunk，out[8..15] 是第 1 行的 8 个 chunk。写法 A 同理，只是起点挪到 flat 偏移 64。第 0 行永远是对的（它的起点就是切片起点），其余行全错；没有编译错误、没有 warning、没有 runtime 报错。接在在线 softmax 里的表现是「除第一行外整片输出偏掉」，非常容易被误判成同步 race 或者 QK 本身算错。

为什么在内核侧绕不过去：现成能用的逐行归约里，能表达行间步长的只有 T.tile.wholereducemax / wholereducemin / wholereducesum。这三个都带 @deprecated() 装饰器，docstring 明写 Currently, this implementation does not support pto target，而且要内核作者自己把物理行宽换算成 32 字节 block 数（512 fp32 → 64）手填进 7 个位置参数里 —— 一个纯硬件层面的量泄漏进算子代码，还把这段代码钉死在非-pto 后端上。另一条绕法是先把窄列段 T.copy 成一块紧凑的 [16, 64] UB 再归约，但这是在 online softmax 的最内层，每个 KV tile × 每个 m-chunk 都要多付一次搬运和一块额外 UB。

同族里还有一个非 deprecated 的 T.tile.reduce_sum_mask_experiment，它确实带行间步长参数，但语义是把全部元素求成一个总和、不是逐行归约，当不了这个场景的绕法。

8.2　现在可以这么写

最小例子
M, BUF_N, W = 16, 512, 64

@T.prim_func
def main(
    src: T.Tensor([M, BUF_N], "float"),
    out_slice: T.Tensor([M], "float"),
    out_real: T.Tensor([M], "float"),
):
    with T.Kernel(1, is_npu=True) as (_, vid):
        ub = T.alloc_ub([M, BUF_N], "float")
        acc_slice = T.alloc_ub([M], "float")
        acc_real = T.alloc_ub([M], "float")
        if vid == 0:
            T.copy(src, ub)
            # 写法 A: 切片直接说明列范围(起点可以是任意合法列)
            T.reduce_max(ub[:, 0:W], acc_slice, dim=-1)
            # 写法 B: 整块进, real_shape 说明逻辑宽度(只能从第 0 列起)
            T.reduce_max(ub, acc_real, dim=-1, real_shape=[M, W])
            T.copy(acc_slice, out_slice)
            T.copy(acc_real, out_real)

关键在 T.reduce_max 的第一个参数：逻辑宽 64 与 ub 的物理行宽 512 不一致，归约就改成一行一个 repeat、行与行之间跨 512 个 fp32（= 64 个 32B block）跳过去，而不再把 [16, 512] 当连续块读。签名里没有任何新参数，切片和 real_shape 两种写法都会被识别；这里让它们归约同一段 [0:W]，所以 out_slice 与 out_real 应当逐位相同（两条写法各自一个 dst 和一个输出，写成同一个 dst 会让先写的那条变成死代码，什么都验不出来）。把 W 改成 128 会直接编译报错：128×4 > 一个 repeat 的 256 字节上限。M, BUF_N, W 定义在 prim_func 外层，是为了避开体内常量会变成符号 Var 的老坑。

CFA 每个 KV tile 在线 softmax 的第一步：对刚算出的 QK 打分 in_ub[ps]（一块 [16, 512] 的 fp32 tile）求每行最大值 m_i。真正有效的列只有运行时的 tw 列（滑窗宽度或 compressed tile 的实际宽度）—— 第 1180 行先对整块 in_ub[ps, :, :] 满宽 fill(-inf)，再只把 [0:tw] 搬进来，所以 [tw:512] 全是 -inf padding；归约宽度按 64 对齐到 tw_64，多出来的 [tw:tw_64] 正是靠这段 -inf 才允许被算进来（reduce_max 忽略、exp 后为 0）。窄窗分支（is_narrow = tw <= BI，BI=128）不愿为 512 列全宽付代价，于是先把 m_i 填成 -inf，再按 64 列一块对 in_ub[ps, :, kc*64 : kc*64+64] 做 T.reduce_max，用 T.tile.max 把每块结果无条件合并进 m_i —— 每一次 T.reduce_max 都是一次窄归约：逻辑宽 64、物理行宽 512、M=16。为什么必须切成 64 列块而不是一次归约 tw_64 列：一个 repeat 上限 256 字节，fp32 就是 64 个元素，这正是 for kc in range(BI // 64) 和 tw_64 = (tw + 63) // 64 * 64 的由来。改动之前这条只能写成 deprecated 的 T.tile.wholereducemax(part, ..., 64, M_CHUNK, 1, 1, 64, ReduceOrder="ORDER_ONLY_VALUE")，把行间步长手算成 64 个 32B block 填进第 7 个位置参数；换成 T.reduce_max 后实参从 8 个（7 个位置参数 + ReduceOrder）降到 3 个，物理行宽这个纯硬件量不再泄漏进算子代码。紧跟其后的 else 一路（宽 cmp tile，512 列全有效、无 padding 可省）反而走普通全宽 T.reduce_max，因为 8 块的标量合并比一条宽指令贵。

8.4　kernel.py 里的真实用法（二）：CFA 在线 softmax 的分母

kernel.py:1366-1383 · CFA · 在线 softmax 的 denom += sum(P)
if is_narrow:
    T.tile.fill(sumP, T.float32(0.0))
    for kc in range(BI // 64):
        if kc * 64 < tw_64:
            T.reduce_sum(
                in_ub[
                    ps,
                    :,
                    kc * 64 : kc * 64 + 64,
                ],
                part,
                dim=-1,
            )
            T.tile.add(sumP, sumP, part)
else:
    T.reduce_sum(
        in_ub[ps, :, :], sumP, dim=-1
    )

同一块 in_ub[ps] 此时装的已经是减完 rowmax、取过 exp 的概率 P，在线 softmax 的分母要把这一 tile 的行和累加进 denom。结构与 rowmax 完全同构，只是算子换成 T.reduce_sum、合并算子换成 T.tile.add、初值 fill 成 0 而不是 -inf。它被放在 cast 之后，是因为 reduce_sum 会破坏 src，必须等 cast 先把 P 读完（中间隔了一个 T.pipe_barrier("v")）。这里窄归约的价值和 rowmax 略有不同：padding 经过 exp 之后是 0，求和其实不怕读到它 —— 真正怕的是读错行。改动之前照直写 T.reduce_sum(in_ub[ps, :, kc*64 : kc*64+64], part, dim=-1)，第 0 行之外全错（out[1..7] 是第 0 行其余 chunk 的和、out[8..15] 是第 1 行的 8 个 chunk），denom 一错整条 softmax 的输出就全错，而且不报任何错。SCFA 侧第 2826-2846 / 2965-2982 行是逐字节相同的两份；SCFA 的 tile 宽度还额外被 topk 压缩过，宽窄两条路在同一次运行里都会真的走到。

8.5　用的时候要注意什么

  • 一个 vector repeat 最多 256 字节 → 逻辑宽度 × 元素字节 ≤ 256（fp32 只能 64 列，fp16 128 列）。超了直接编译报错并提示自己切段合并 —— kernel.py 里那圈 for kc in range(BI // 64) 加 T.tile.max / T.tile.add 就是这么被逼出来的。
  • 只支持 dim == -1（行归约）。dim == 0 的列归约走不到这条路，会报 narrow reduce is only implemented for a row reduce。
  • 不支持 clear == False：不能把窄归约的结果并进目标，必须归约到 scratch 再自己合并（kernel.py 用的就是 part + T.tile.max / T.tile.add）。
  • 物理行宽 × 元素字节必须是 32 的整数倍（行间步长以 32B block 计），否则整除会截断、stride 走短甚至变 0。这一条有守卫，会编译报错。
  • 切片起点 a 没有任何一层校验：它会原样变成源操作数的元素偏移，而向量指令要求操作数起始地址 32 字节对齐（fp32 → a 是 8 的倍数，fp16 → 16 的倍数）。ub[:, 4 : 68] 这种偏移会被照单全收、不报错。kernel.py 用的是 64 的倍数，未对齐的起点没有被验证过。
  • 只有 M / 逻辑宽 / 物理行宽三者全是编译期常量时才会走窄路径。任何一个是符号量，就直接退回老的连续块走法且不报任何错 —— 也就是又变回静默错答案。所以窄归约的宽度千万别写成运行时变量；kernel.py 的做法是宽度定死 64、用运行时的 tw_64 去 guard 哪些 chunk 参与。
  • 元素宽度按 dtype 认：fp32 / uint32 = 4 字节，fp16 / bf16 / int16 / uint16 = 2，int8 / uint8 / fp8 = 1。int32 用不了 —— int32 buffer 做窄归约会直接编译失败，报 narrow reduce does not know the element size of 'int'；int64 / uint64 / int4 同样不支持。至少这是硬报错，而不是默认成 4 字节把 stride 悄悄缩放错。
  • 窄路每个 chunk 要一次 reduce 加一次标量合并。当整行都有效时（比如 512 列全参与）走普通宽 reduce 更快 —— kernel.py 用 is_narrow = tw <= BI 分路，注释里写明「8-chunk 的标量开销太贵」。
  • 归约 dst 留在 UB 里用 [M, 1] 没问题（kernel.py 的 part / sumP 就是）。但要把每行标量直接 UB→GM 导出时得用一维 [M]，否则会撞上 32B block 对齐重排的老坑 —— 每行标量按 [M, 1] 走 GM 会被按 32 字节块跳读打乱、行与行错位。
  • 升级编译器后，一批「改动前能编过、只是答案静默错」的写法（dim=0 的窄切片、clear=False 的窄归约、逻辑宽超过一个 repeat 的窄归约）会变成硬编译报错。从静默错答案换成编译报错是好事，但原本「能跑」的代码可能编不过。

9.3　解构之后反而更快

把 QK 从一句 T.gemm_v0 解构成内核驱动的 T.mma 序列（第 6 节），性能不是持平，是直接快了约 12 个百分点。原因不在算得更少，而在同步：

一句 T.gemm_v0 在首尾各含一对「set 紧跟 wait、同一个 event」的 flag。自己配对的 flag 不是给某个 buffer 定序，而是 drain 整条 pipe —— 效果等同一条 pipe barrier。

于是每调用一次 gemm，就强制「所有 MTE2 完成才能开始 MTE1」，并挡住下一次 KV 页加载在当前这次 gemm 的 L1→L0 结束之前发出 —— 正好把内核精心搭的 3 槽 KV 环存在的全部意义废掉。自己驱动之后这两对围栏就不存在了：内核发的 per-slot MTE2_MTE1 / MTE1_MTE2 是定向的，守同样的 hazard 但不 drain 整条 pipe。

对写 kernel 的人意味着什么：如果一段 cube 代码依赖多缓冲/环形 buffer 做 overlap，那么每一句整块的 T.gemm_v0 都会在它前后各插一道全 pipe 栅栏，overlap 拿不回来。这种场景才值得拆成第 6 节的写法；不靠 overlap 的地方，一句 T.gemm_v0 仍然是更好的选择。
