KV Cache · 内存管理

PagedAttention 原理详解

把 KV Cache 像操作系统管内存那样「分页」管理 —— 完全没接触过神经网络也能读懂

目标读者：完全没接触过神经网络也能读懂。你只需要会加减乘除，以及整数除法「//」和取余「%」：比如 7 // 4 = 1（7 里有 1 个完整的 4），7 % 4 = 3（余 3）。

全文用同一个小例子贯穿到底，每一步都能拿纸笔自己验算。神经网络、注意力是什么，第 0 节用大白话从零讲起，不需要任何前置知识。

0先说人话：注意力、KV 到底是什么

你可能听过 ChatGPT 这类「大模型」。它们的核心零件叫 Transformer，而 Transformer 里最关键的一步操作，叫 注意力（attention）。我们先把这两个词翻译成人话。

0.1 注意力：读到「他」的时候，回头看看「他」是谁

想象你在读一本侦探小说，读到第 100 页，看到一句：

「凶手就是他。」

要搞懂「他」指谁，你的大脑会回头看前面出现过的人物，并且重点关注最相关的那几个（比如第 12 页登场的管家），忽略无关的路人。

模型处理文字时干的是同一件事：

处理当前这个词的时候，回头看前面所有的词，给每个词打一个「该关注多少」的分数，再把这些词携带的信息，按分数加权混合起来，作为这一步的输出。

「该关注多少」这组分数怎么算、怎么变成加起来等于 1 的权重，是另一篇文档讲的（见 OnlineSoftmax 原理详解）。本文只关心一件事：那些「前面的词」的信息，到底存在哪、怎么存、怎么取。

0.2 K 和 V：每个词都存成两样东西

模型会把每个读过的词，变成两个一串数字的「向量」存起来：

  • Key（键，简称 K）：像这个词的「检索标签」。当前词拿自己的「查询」去跟它比对，判断该不该关注这个词。
  • Value（值，简称 V）：这个词真正携带的「内容信息」。一旦决定关注它，就把这份 V 取出来，按关注度混进结果里。

0.3 KV Cache：为什么要把 K、V 存起来

关键事实：一个词的 K 和 V，一旦算出来就固定不变了。

而模型生成文字是一个词一个词往外蹦的。每多蹦一个新词，注意力都要回头看前面所有词的 K、V。如果每蹦一个词就把前面所有词的 K、V 重算一遍，那就是巨大的重复劳动。

聪明的做法当然是：算一次，存起来，以后反复用。这块专门存放历史 K、V 的存储，就叫 KV Cache（KV 缓存）。

它会越堆越多。 对话越长、序列越长，存的 K、V 就越多，一段长对话能堆到好几个 GB。

于是问题就变成了一道内存管理题：这么多、还在不停增长的 K、V，怎么在显存里摆放最省、最快、最不浪费？PagedAttention（分页注意力） 就是目前公认最漂亮的答案。下面我们就从「老办法哪里不好」讲起。

1问题的根源：KV Cache 是个内存大户

先建立一点数量感。我们采用这样的设定：

  • 每个词的 K 向量是 512 个数字，V 向量也是 512 个数字；
  • 用的是 bfloat16，每个数字占 2 字节。

所以一个词的 K + V 大约占：

(512 + 512) 个数字 × 2 字节 = 2048 字节 ≈ 2 KB

听起来不多？但是：

  • 一条长序列可能有几千上万个词 → 几十 MB；
  • 服务器要同时处理很多条请求（很多用户、很多 batch）→ 几个 GB。

而显存（NPU/GPU 上的高速内存）是很贵、很有限的资源。所以「这几个 GB 怎么摆」直接决定了一台机器能同时服务多少用户。浪费一点，成本就翻一截。

2老办法为什么浪费：给每个人圈一大段连续房间

最直觉的存法是：给每条序列开一块「连续的大数组」，而且大小要按「最长可能的长度」来开。

为什么非要连续、非要按最长开？因为老式的注意力实现喜欢「一口气把这条序列的所有 K 读进来」，连续摆放才好一把抓；而它事先不知道这条序列最终会多长，只能按最坏情况预留。这就像一家酒店的蠢规矩：

每来一位客人，前台一律按他可能要的最多间数，把连续的一长排房间整段包给他一个人——中途别人一间都插不进来，哪怕他根本住不满。

实测中，这种「连续 + 预留最长」的存法，真正用上的显存常常只有 20%~40%，剩下大半都在替「可能用得上但其实没用上」的空间站岗。

我们需要一种不必连续、用多少占多少、还能随时长大的存法。计算机科学里早有现成的答案——操作系统的分页。

3灵感来自操作系统：分页（paging）

这一节不需要你懂操作系统，看一个比喻就够了。

回到酒店。聪明的酒店不会一上来就给客人圈走一大段连续房间，而是：

  1. 把所有房间编上号：0 号房、1 号房、2 号房……（这些是真实存在的物理房间）。
  2. 客人来了，要几间就发几把钥匙（一把钥匙开一间房），而且这些房间不必相邻——这次给你 207 房，下次接着给你 512 房，中间隔着别人在住也无所谓。
  3. 前台拿一本登记簿：记下「张三：第 1 段住 207，第 2 段住 512……」。想找张三第几段住哪，查登记簿就行。

这样一来：

  • 没有「整段预留」的浪费：要几间就发几间，用多少占多少。
  • 不怕夹缝：哪个房间空着都能用，反正不要求相邻。
  • 能随时长大：客人临时要续住？再发一把空房钥匙、登记簿上加一行即可。

把这套搬到 KV Cache 上，就是 PagedAttention。术语对照表先放这，下一节逐个讲：

酒店比喻PagedAttention 术语
编号的物理房间物理块（physical block / page）
一间房能住几人块大小（block size）
客人住的「第几段」逻辑页（logical block）
前台的登记簿块表（block table）
一位客人一条序列 / 一个请求

4三个主角：物理块、逻辑页、块表

4.1 物理块（physical block）：散装的存储格子

把整块显存切成许多固定大小的小格子，叫物理块，从 0 开始编号：块 0、块 1、块 2、……一共 block_num 个。每个物理块能装 block_size 个 token 的 KV（比如 block_size = 128，就是一块装 128 个词）。

关键：这些块散装、乱序、谁空给谁，不要求一条序列用到的块在内存里挨着。

4.2 逻辑页（logical block）：序列「自己以为」的样子

站在一条序列的角度，它的 token 是从 0 开始连续编号的：token 0、token 1、token 2、……我们把连续的每 block_size 个 token 打包成一个逻辑页：

「逻辑页」是序列自己以为的连续排版；它并不知道（也不需要知道）自己实际被塞进了内存里哪几个散装物理块。

4.3 块表（block table）：把逻辑页翻译成物理块

块表就是那本登记簿：它为每条序列记一行，内容是「我的逻辑页 0 在物理块几、逻辑页 1 在物理块几……」。它是一张二维整数表 block_table，形状 [B, table_len]：

  • B：序列条数（batch size），一条序列一行；
  • table_len：每条序列最多有多少个逻辑页（登记簿每行留几个格子）；
  • block_table[b, j] 的值 = 「第 b 条序列的逻辑页 j」实际存在物理块几。

还有一个约定（很重要）：block_table[b, j] == -1 表示「这一格没用到」（这条序列没那么长，逻辑页 j 不存在）。-1 就是登记簿上「此段空着」的标记。

5完整小例子：从「第 6 个 token」到「内存里第几行」

为了能拿纸笔验算，这一节把数字调小：设 block_size = 4（一个物理块装 4 个 token）。（真实里通常是 128，原理一模一样，只是数大。）

5.1 场景设定

  • 内存里一共有 6 个物理块：块 0 ~ 块 5。
  • 来了一条序列（batch 里第 b 条），它有 10 个 token（token 0 ~ 9）。
  • 它需要几个逻辑页？ceil(10 / 4) = 3 个：逻辑页 0、1、2。
  • 系统随便挑了 3 个空物理块发给它（故意打乱，证明不需要连续）：逻辑页 0 → 物理块 3，逻辑页 1 → 物理块 0，逻辑页 2 → 物理块 5。

5.3 把这套算术写成一句话

给定逻辑 token 号 g：
    物理块 = block_table[b,  g // block_size]
    行号   =                 g %  block_size
    要的 KV = KV[物理块, 行号]

记住这三行，你就掌握了 PagedAttention 的全部精髓。 剩下的都是工程细节。

6真实张量布局

把上面的小例子换成真实的数字和张量形状。

6.1 KV 张量：[block_num, block_size, H, D]

分页 KV 张量的形状是四维的：

  • 第 0 维 block_num：物理块总数（散装格子有几个）；
  • 第 1 维 block_size：一个块装几个 token（例：128），就是 5.2 里的「块内行号」那一维；
  • 第 2 维 H：KV 头数，这里固定为 1；
  • 第 3 维 D：每个 K/V 向量的长度，这里固定为 512。

所以「物理块 phys、行号 row」对应的那条 KV 向量，就是切片 KV[phys, row, 0, :]——一条长度 512 的向量。

6.2 块表：[B, table_len] 的 int32 表，-1 占位

7取数：把三步算术落到张量上

有了第 6 节的四维张量布局，第 5 节那三步算术就能直接落地。给定一个逻辑 token 号 g：用 g // block_size 算出它在第几个逻辑页，用 g % block_size 算出页内第几行，再查块表把逻辑页翻译成物理块号，最后读切片 KV[物理块号, 行号, 0, :]——这条长度 D = 512 的向量，就是要取的那条 KV。

关键在于：每个 token 各查各的块表、各取各的一行，彼此完全独立，所以取数这一步天然适合大规模并行——许多 token 同时各取各的，互不干扰。

8几个容易卡住的细节

8.1 为什么用 // 和 %，而不是别的？

因为「连续编号切成固定大小的段」这件事，天生就是除法和取余：

  • g // block_size：g 号 token 前面有多少个完整的块，就是它在第几页；
  • g % block_size：除完剩下的零头，就是它在本页第几行。

这跟「第 6 个鸡蛋放进每盒装 4 个的蛋盒：在第 6//4=1 盒、盒里第 6%4=2 格」是同一个算术。

8.2 -1 占位是干嘛的？会不会被误读？

块表是个规整的矩形张量 [B, table_len]，但每条序列长短不一，短序列用不满一行。空着的格子填 -1 当「此处无块」的旗标。

取数前会先判断这个 token 是否真实存在（在不在有效窗口/范围内）。不存在的，直接把那条 KV 填 0，根本不会拿 -1 去查 KV，所以不会越界、不会读到垃圾数据。-1 只是「别碰这一格」的提示牌。

8.3 最后一个块没装满怎么办？

像 5.1 里块 5 只装了 t8、t9，后两行是空的。没关系：

  • 这是唯一残留的浪费，而且每条序列最多浪费不到一个块（block_size - 1 个 token 的位置）；
  • block_size 越小，这点尾巴浪费越小，但块表越长、查表越频繁——这是一个工程上的权衡，实践中常取 128 这种中等值。

对比「圈一大段连续房间」动辄浪费几百上千个 token 的空间，这点尾巴可以忽略。

8.4 分页会让计算变慢吗？

几乎不会。多出来的成本只是「查一次块表 + 一次除法取余」，相比后面真正的矩阵乘法（Q·Kᵀ、P·V，几十万次乘加）微不足道。换来的是显存利用率从 ~30% 提到接近 100%——一台机器能同时服务的用户数翻几倍。这买卖非常划算，所以现代推理引擎（如 vLLM）普遍采用它。

9两套分页缓存 + 设备端取数

前面讲的是通用 PagedAttention。在「稀疏 + 滑窗」这类注意力里，分页机制还有两个值得一提的延伸。

9.1 两套独立的分页缓存

有的注意力算子同时做两种注意力，于是各自管一套分页 KV、各自一张块表：

两条取数路径除了「读哪套缓存、查哪张块表」不同，其余完全一样——分页这套机制可以独立地套在多股 KV 上，互不影响。

9.2 块表解析放在「设备端」，不在主机端预先拆包

有些实现会在 CPU（主机端）上先把分页 KV 拆回连续大数组，再喂给加速器——那等于又把「圈一大段连续房间」的浪费请回来了。

更好的做法是坚持在设备上现查现取：块表和分页 KV 原样送进设备，由设备上的每个并行通道自己算出 (物理块, 行号)、自己搬运那一行。这样全程没有「拆包成连续大数组」这一步，分页省下来的显存是实打实省下来的。

10一页速查 + 总结

速查：PagedAttention 三步取数

已知：要第 b 条序列的逻辑 token g 的 KV，块大小 block_size
─────────────────────────────────────────────────────────
  逻辑页号 = g // block_size            # 在第几个块
  块内行号 = g %  block_size            # 块里第几行
  物理块号 = block_table[b, 逻辑页号]    # 查登记簿翻译成真实位置
                                        #   == -1 → 此 token 不存在，填 0
  要的 KV  = KV[物理块号, 块内行号, 0, :]  # 一条 [D=512] 向量

一句话总结

PagedAttention = 把 KV Cache 像操作系统管内存那样「分页」管理。

把显存切成固定大小的散装物理块，序列要多少发多少、不必相邻，用一张「逻辑页 → 物理块」的块表记账。取数时只需 g // block_size、g % block_size、查块表 三步算术，就能把序列眼中连续的 token，找到它在散装内存里的真身。

代价：每次取数多一点除法和查表（可忽略）；回报：显存利用率从 ~30% 拉满到接近 100%，浪费几乎归零，同一台机器能服务的用户数成倍增长。
