昇腾流水线 · 任务分核 + 软件流水

四级深度3流水 原理详解

一个注意力任务要走 QK → softmax → PV → 输出 四步，两步在 Cube、两步在 Vector —— 怎么让相邻任务错开叠着跑，让四个阶段同时运转、Cube 与 Vector 都不闲。

本文假设读者已熟悉昇腾的基本盘：一个核里有 1 个 Cube 单元 + 2 个 Vector 单元（1 : 2）：Cube 专做矩阵乘；两个 Vector 一起做 softmax / 归一化这类逐元素运算（一个任务的行对半分、各算一半）。Cube 与 Vector 是核内不同的执行单元、能同时开工。下文把这两个 Vector 合起来当「Vector 一侧」看。

要讲清楚的是：把一大批注意力任务分给各个核之后，每个核内部怎么用「软件流水」把这些任务错开叠着跑。这条流水有四个阶段（因此叫「四级」）、任何时刻有三个任务同时在飞（因此叫「深度 3」）。只需会分奇偶即可跟下来。

0先说结论：四个阶段、三个任务在飞

把一整批注意力任务分给各个核之后，每个核要一个接一个地处理分给自己的那一批任务。一个任务不是一步算完，而是要依次走四步，前一步的结果喂给后一步：

① QK 打分──▶ ② softmax──▶ ③ PV 汇总──▶ ④ 输出整理

关键点有两个：①③ 在 Cube 上做、②④ 在 Vector 上做（Cube、Vector 是核内两种执行单元）；而「真正同时」只发生在 Cube 和 Vector 之间 —— 同一个单元一次只能干一件事。如果老老实实「一个任务走完四步，再开下一个」，那任何时刻都只有 Cube、Vector 一方在动、另一方闲着，白白浪费一半算力。

办法：让相邻任务错开叠着跑。让 Vector 整体比 Cube 慢一拍，于是同一时刻 Cube 在忙这个任务、Vector 在忙上一个任务 —— 四个阶段像流水线上的四个工位一样同时运转，Cube 与 Vector 都不闲。因为一个任务从进到出要跨好几轮，任何时刻都有三个不同的任务同时「在飞」（一个刚进、一个在中间、一个快出）—— 这就是「深度 3」。

「四级」= 阶段数（4 个工位），「深度 3」= 同时在飞的任务数（3 个）。下面从「一个任务的四步」讲起，一步步搭出这条流水，最后解释这两个数为什么不相等。

1一个任务的四步：QK → softmax → PV → 输出

先看单个任务要干什么。一个任务 = 给某个查询位置算注意力，它要依次走这四步，前一步的半成品交给后一步：

步阶段在哪个单元干什么
①Q · QK 打分Cube查询向量和历史各词的键做矩阵乘，得到每个历史词的「相关度分数」。
②S · softmaxVector把一行分数归一成一组和为 1 的权重（减最大值、取 exp、除以总和）。
③P · PV 汇总Cube用这组权重去加权求和历史各词的值向量，得到「加权和」。
④O · 输出整理Vector把加权和除以总权重归一化、转成输出精度、写回结果。

📦 半成品放哪？Cube 和 Vector 同一时刻处理的是不同任务，一个单元算出的半成品得先放进 GM 里几块「中转区」（分数、权重、加权和各一块），另一个单元下一步再来取。后面第 7 节会讲这些中转区为什么各开「两份」。

2为什么要错开：同一单元串行、Cube 与 Vector 才并行

先看不错开会怎样。假设死板地「一个任务从头走到尾，再开下一个」：

要消掉这些空等，得给闲着的单元找活干。而它唯一能同时干的，是别的任务的活 —— 因为同一个任务的相邻两步有先后依赖，不同任务之间却互不依赖。于是思路就来了：

当 Cube 在给任务 j 做 Q 时，让 Vector 同时给另一个任务做它的 S；当 Cube 转去做 P 时，让 Vector 去做又一个任务的 O。每一刻凑成一对「Cube 干一件 ‖ Vector 干另一件」，而这一对分属不同任务、互不依赖，才能真正同时跑。要凑成这种对子，就得让相邻任务在时间上错开。

🔑 一句话记住：并行只发生在 Cube 与 Vector「之间」，且必须是不同任务。同一个单元内部（比如 Cube 的 Q 和 P）永远是先后做的，不是同时。

3让 Vector 慢一拍 —— 四级流水主图

把「错开」落实成一句话：让 Vector 整体比 Cube 慢一拍，处理的永远是 Cube 上一轮已经备好的半成品。这样一排，四个阶段就同时转起来了。先定义「一轮」：

「一轮」= 处理循环往前迭代一次。注意关键：一轮里推进的不是一个任务的一步，而是四个工位各推进一个不同任务。下面这张表，一列 = 一轮（往右是时间），格子里填「该轮这个工位正在处理第几号任务」：

先记住这条错位规律：第 r 轮里，Q 做任务 r、S 和 P 做任务 r−1、O 做任务 r−2。Vector 那两行（S、O）永远比 Cube 慢，处理的都是前面轮次早已备好的数据，所以从不停下等 —— 吞吐拉满。

4两种读法：竖读一轮 / 斜读一个任务；前后半拍

同一张表有两种读法，各回答一个问题。

4.1 竖读一列 = 这一轮同时干的四件事

竖着读第 4 轮那一列：Q=任务4、S=任务3、P=任务3、O=任务2。四个工位同一轮都在忙，服务三个不同任务。再把这一轮切成前后两个半拍，每半拍恰好是一对「Cube ‖ Vector」的真并行：

4.2 斜读追一个任务 = 它自己的 Q→S→P→O

换一种读法：在表里跟着同一个任务号走。盯住任务 3：它在第 3 轮做 Q → 第 4 轮做 S → 还是第 4 轮做 P → 第 5 轮做 O，一路沿对角线往右下角走。这也点破一个容易迷惑的地方：任务 3 的 S 和 P 都落在第 4 轮（S 前半拍、P 后半拍），它俩是同一个任务前后脚的两步（S 先算好、P 再用它的结果），不是同时。

🧭 两种读法别混：竖读一列回答「这一轮四个工位在同时忙什么」（不同任务）；斜读一条对角线回答「某一个任务是怎么一步步走完的」（同一任务）。竖看到的「同时」是真并行；斜看到的「相邻」是先后依赖。

5「四级」和「深度 3」是两个不同的数

普通流水线里，阶段数 = 同时在飞的任务数（有几个工位就有几个任务各占一个）。可这里偏偏是四级、深度 3 —— 少了 1。原因就在上一节那个细节：

一个任务的 S 和 P 被挤进了同一轮（S 前半拍、P 后半拍）。于是一个任务从 Q 到 O，本该跨 4 轮，实际只跨 3 轮就走完 —— 四个阶段只占了三轮的宽度。

名词数指的是
四级4阶段（工位）数：Q / S / P / O 四个工位，一轮里全都在忙。
深度 33同时在飞的任务数：任一轮里出现 3 个不同任务号（一个在做 Q、一个在做 S/P、一个在做 O）。

两者本该相等，这里因为 S、P 并到一轮而恰好差 1 —— 所以「四级」和「深度 3」不矛盾，是两个不同的量。

6预热 2 轮、排空 2 轮

流水线要「满负荷」，前提是四个工位都填上了任务。可最开始没那么多任务在飞，末尾也会有任务陆续走完 —— 于是流水两头各有一小段没填满：

  • 预热（开头）：头几轮里，后面的工位还没等到数据。第 1 轮只有 Q 有活（任务 1），S/P/O 都还没料；第 2 轮才轮到 S/P…… 要2 轮才把前几个任务陆续喂满四个工位。
  • 排空（结尾）：最后一个任务做完 Q 之后，还得再跑2 轮，把它和它前一个任务剩下的 S/P/O 做完。

7中转缓冲为什么只开 2 份、不开 3 份

回到第 1 节留的尾巴：Cube 和 Vector 交接半成品，靠 GM 里几块中转区（分数、权重、加权和各一块）。因为 Vector 慢一拍，当「新任务往某块中转区写」时，「上一个任务对它的读」可能还没读完 —— 直接覆盖就出错。办法是每块中转区都开两份，按任务号的奇偶轮流用（这叫双缓冲 / 乒乓）：任务 t 用第 t%2 份。

可流水明明有 3 个任务在飞，为什么每块中转区只开 2 份就够、不用 3 份？拿其中一块 —— 存 QK 分数的那块 —— 实际走一遍最清楚。它由 Cube 在 Q 阶段写、由 Vector 在 S 阶段读；开 2 个槽，任务 t 用第 t%2 个槽：

为什么 2 份够、3 份多余，三句话说清：

  • 竖看任意一轮（第 1 轮起）：总是一个槽在写、另一个槽在读，两件事永远落在不同槽 → 不打架。
  • 谁和谁共用一个槽：是任务 t 与 t+2（都用 t%2），它俩相差 2 轮、错得开；夹在中间的 t+1 用的是另一个槽。所以任何时刻最多 2 个任务碰这块中转区，2 份足矣。
  • 反证只开 1 份：第 1 轮就得「一边写任务 1、一边读任务 0」到同一个槽 → 撞车。所以至少要 2 份。

💡 「深度 3」和「缓冲 2 份」不矛盾：深度 3 是整条流水同时在飞的任务数；而每一块中转区只被相邻的两个任务（写者与读者）碰到 —— 第三个在飞的任务此刻在忙别的中转区。所以每块各 2 份就够。

8四层重叠：从任务到内存，怎么层层嵌套

本文讲的四级深度3流水，只是这个算子里最外面一层的重叠。同一个「让搬 / 算并行、别干等」的思路，其实在四个层级上各套了一遍 —— 从最外的「整批任务在两个计算单元间错开」，一路套进最里的「每一跳内存搬运都藏进矩阵乘背后」。先看它们怎么嵌在一起：

看懂嵌套，再逐层看每一层各让哪两件事并行、各消掉了谁的等待（上下两条并排 = 同时进行）：

#机制哪一层 / 哪一跳让哪两件事并行消掉的等待
①四级深度3流水（本文）任务级 · 跨 Cube / VectorCube 矩阵乘 ‖ Vector 逐元素一个计算单元干活时、另一个空等
②KV 三槽环GM→L1 · Cube 内搬 K/V（GM→L1）‖ Cube 矩阵乘Cube 等 K/V 从 GM 搬进来
③L0AB 双缓冲L1→L0 · 矩阵乘内上料（L1→L0）‖ mmamma 等操作数从 L1 上到 L0
④L0C 结果双缓冲L0C→外 · 矩阵乘内结果搬出（L0C→中转区）‖ 下一次 mma下一次 mma 等上次结果搬走

从 ① 到 ④，粒度越来越细、位置越来越靠里：① 管两个算力单元不闲；②③④ 管 Cube 从 GM 到 L0C 每一跳都不停下等内存。四层各管一段、层层嵌套，整条链子才从头到尾不卡。

8.1 这三跳是「串行依赖」，还是「能各自考虑」？

②③④ 处在同一条数据通路的三跳上（GM→L1→L0→算→L0C），很自然会问：数据不是得先进 L1、才能进 L0、才能算完进 L0C 吗，这三件事还能分开看？答案要分两个角度：

  • 追一份数据看 —— 是不可拆的一条链：某一块 K/V 必然 GM→L1（②）→ L1→L0（③）→ mma 算 → 结果进 L0C（④ 搬走），先后不可颠倒。
  • 看同一瞬间 —— 它们各踩不同的数据块、同时在跑（和第 3 章那张任务表同一个道理，只是下沉进一次矩阵乘内部）：

所以「能不能分开考虑」有两层意思：

  • 能局部设计：每一跳之间夹着一个缓冲（L1 的 3 槽环、L0 的 2 份、L0C 的 2 份）。缓冲的作用就是解耦相邻两级 —— 上游往里写、下游从里读，只要不空不满，两级各跑各的、不必锁步。所以「某一跳缓冲开几份够不够」是个局部问题，可以单独想。
  • 不能完全独立：相邻两级共用一块缓冲、靠一对握手信号连着，是一条握手链、不是三个孤岛。比如 ② 要覆盖一个 L1 槽前，得等 ③ 发来「我已把它搬去 L0 了」。所以 ②「能不能装下一块」直接取决于 ③「有没有把上一块取走」。

⏱ 还有一条总规律：整条链的速度 = 最慢那一跳（瓶颈），其余快的都藏进它的影子里。缓冲能吸收短时快慢抖动，却吸收不了长期的速率不匹配 —— 某跳持续更慢，它下游的缓冲迟早排空、把下游饿停，瓶颈就往下游传。

一句话：纵向追一份数据是不可拆的一条链；横向看同一瞬间是不同数据在各跳上并行。各机制的缓冲深度能局部定，但接缝处被握手信号锁在一起、且同受最慢一跳节制 —— 局部可分、整体成链。

9一页速查 + 总结

三句话总结

  1. 四个阶段、两种单元：一个任务走 Q（QK 打分）→ S（softmax）→ P（PV 汇总）→ O（输出整理），其中 Q、P 在 Cube，S、O 在 Vector。真并行只发生在 Cube 与 Vector 之间、且必须是不同任务。
  2. 让 Vector 慢一拍：第 r 轮里 Q 做任务 r、S 和 P 做任务 r−1、O 做任务 r−2。四个工位同一轮都忙、各服务不同任务 —— 四级流水同时运转，Cube 与 Vector 都不闲。
  3. 四级 ≠ 深度 3：阶段数是 4，但一个任务的 S、P 挤在同一轮，只跨 3 轮就走完，所以任何时刻只有 3 个任务在飞（深度 3）。交接用的中转区各开 2 份乒乓即可（每块只被相邻两任务碰），代价是两头各预热 / 排空 2 轮。
问题答案
四个阶段是什么？Q＝QK 打分、S＝softmax、P＝PV 汇总、O＝输出整理；前一步喂给后一步。
谁在哪个单元？Q、P 在 Cube（矩阵乘）；S、O 在 Vector（逐元素 / 归一化）。
怎么错开？让 Vector 慢一拍：第 r 轮 Q 做任务 r、S/P 做 r−1、O 做 r−2。
「真同时」发生在哪？只在 Cube 与 Vector 之间，且两件事必属不同任务。同一个单元一次只干一件。
四级 vs 深度 3？四级＝阶段数（4）；深度 3＝同时在飞的任务数（3）。差 1，是因为 S、P 挤进同一轮、一个任务只跨 3 轮。
中转区为什么 2 份？每块只被相邻两任务（写者、读者）碰，任务 t 与 t+2 共用一槽、相差 2 轮错得开；开 2 份乒乓即可。
两头的代价？预热 2 轮把任务喂满、排空 2 轮把尾巴做完（各 = 深度 − 1）；任务越多越可忽略。

让 Vector 慢 Cube 一拍，四个工位就同时转了起来 —— Cube 与 Vector 都不闲、吞吐拉满。这就是四级深度3流水的全部用意。
